# Cooking Platform · Claude Code 工作手册

## 一、项目定位

面向公网上线的商业级烹饪内容分享平台，对标字节/腾讯产品研发规范。
核心气质：松弛、真实、有温度、可复现、有场景感、不焦虑。
目标用户：22~28 岁独居/合租年轻人。

**项目还有一个目的**：用户的 Go 后端面试准备。代码质量、设计决策深度、能否讲得清楚，
与功能完成同等重要。每步必须产出可在面试中复述的故事线。

---

## 二、技术栈（不可更改）

- **后端**：Go + Gin（cmd / internal / pkg 分层）
- **ORM**：GORM（读写分离用 DBResolver）
- **MySQL** 8.0（主从复制，GTID）
- **Redis**：Sentinel 模式（缓存 / Session / 点赞状态 / 限流）
- **MQ**：RabbitMQ（持久化 + ACK + 死信），通过 EventBus 接口抽象。
  MVP 用 Go Channel，生产切 RabbitMQ。
- **OSS**：阿里云（STS 预签名 + 客户端直传，Go 服务器不过图片流量）
- **短信 / 内容安全**：阿里云
- **部署**：Docker Compose（双 Go 实例 + Nginx + MySQL 主从 + Redis Sentinel + RabbitMQ）
- **监控**：Prometheus + Grafana
- **CI/CD**：GitHub Actions
- **数据库迁移**：golang-migrate

## 七、可复用资产规范（强制套用，不许另起炉灶）

新增/修改代码前，必须先判断能否复用现有抽象。模板归口：

| 类型 | 模板来源 | 套用规则 |
|---|---|---|
| Cache | `internal/cache/user_cache.go` | Key 命名 + TTL + 错误返回必须一致 |
| Consumer | `internal/consumer/like_consumer.go` | 批量消费 + 优雅停机 + drained 日志必须套用 |
| Repository | `internal/repository/user_repository.go` | 接口先行（FooRepository interface），ErrXxxNotFound 封装 |
| Service | `internal/service/user_service.go` | ctx 透传 + 错误处理 + 日志格式 |
| Handler | `internal/handler/user.go` | bind → validate → service → response 四步 |
| 外部依赖 | `pkg/sms/{mock,aliyun}.go` | 接口 + Mock + Real 三件套，dev/prod 配置切换 |

**常量与 Key 集中管理**：

- 所有 MQ Topic：`internal/event/types.go`（追加式）
- 所有 Redis Key 模板：引入新 Key 时同步追加到 PRD-v2.2 §6.3 Key 命名全表
- 命名：`{module}:{purpose}:{key}`，TTL 必须显式声明

**Migration**：严格按 `000NNN_<verb>_<resource>_table.up.sql` / `.down.sql`，时间顺序递增，不跳号、不复用。

**验证脚本**：

- 公共片段在 `scripts/verify_common.sh`
- 每步特有断言 `scripts/verify_step_N.sh`，开头 `source ./scripts/verify_common.sh`
- 禁止重写已沉淀的通用片段

## 八、规则挑战机制

本文档的所有"硬性规定"在 95% 场景下应当遵守。但本项目处于持续演进中，
PRD 撰写时未必预见所有情况。当你（Claude Code）判断某条规则在当前场景下
确实不合理时，**允许挑战**，遵循以下流程：

1. **不要默默偏离**——发现规则不合理时先停下，明确告诉用户："规则 X 在
   当前场景下我建议偏离，原因是 ABC，建议改为 Y"。
2. **等用户确认**——用户同意后才能偏离。用户坚持原规则时，按原规则执行
   并把你的顾虑记入收尾时的"偏离点登记"或"待跟进事项"。
3. **必须留痕**——任何被批准的偏离，本步收尾时写进进度追踪文件的
   「本步与 PRD 的偏离点」章节，格式：
   `PRD 章节 / 规则编号 → 实现差异 → 选择此方案的原因`
4. **下次同类场景复用**——若同一偏离重复出现 2 次以上，主动建议用户更新
   本 CLAUDE.md 或 PRD，让规则跟上现实。

**什么算"合理偏离"**：

- 模板套用会导致明显的过度设计（如简单的工具函数被迫塞进 Repository 接口）
- 错误码段位用尽（理论上每段 999 个够用，但万一）
- 上游依赖变化使原架构失效（如阿里云 API 改版）
- 新发现的性能 / 安全问题需要打破原约定

**什么不算合理偏离**：

- "这样写更省事"——不是理由
- "用户没明确要求"——不是理由，PRD 是默认要求
- "我觉得这样更优雅"——审美差异不构成偏离

## 九、每步开工契约（两步）

### 第 1 步：100 字说明 + 读文件

- 100 字以内说明本步要做什么、为什么
- 列出本步要读的文件清单（精确到路径），用 Read 工具逐一读取
- 单步读取文件 ≤ 15 个，超出分批
- 路径不确定时自己用 Glob / Grep 定位，不要问用户

### 第 2 步：假设清单 + 批量化生成

基于已读代码输出假设清单：

- 会调用的函数签名（参数、返回值）
- 会访问的结构体字段（名字、类型）
- 会使用的配置项路径（如 `cfg.Audit.Provider`）
- 本步使用的错误码段位
- 本步引入的新 Redis Key（含 TTL）
- 本步引入的新 MQ Topic

用户确认后按子模块分组生成：

1. **数据层**：migration + model + repository（含 interface）
2. **业务层**：service + cache（如有）
3. **接入层**：handler + DTO + 路由装配
4. **收尾**：main.go 装配 + verify_step_N.sh + Makefile 命令追加

每组完毕等用户确认再进下一组。用户可随时喊停、要求改动单文件。

## 十、已完成文件的修改边界

**原则**：所有已完成文件都允许修改，无需用户豁免。但必须遵守两条规则：

1. **修改前明确告知**：在改动前一句话说明"我准备修改 X 文件的 Y 处，原因是 Z"，
   让用户有机会喊停。
2. **修改后留痕**：本步收尾时在代码变更清单中标记"修改"而非"新增"，
   并在该文件条目下注明改了什么、为什么改。

**仍然禁止**：

- 删除已完成文件（除非用户明确要求）
- 重命名已完成文件的导出符号（函数、结构体、接口名）—— 会污染 git blame，
  必要时用 alias 过渡
- 修改已发布的 API 路由路径 / 错误码定义 / 数据库表字段名（破坏向后兼容）

下列扩展性变更属于"日常追加"，连"修改前告知"都不需要：

- `cmd/server/main.go`：添加新模块的依赖装配与路由注册
- `configs/config.yaml` + `config.prod.yaml`：新增配置段
- `pkg/config/config.go`：新增配置结构体
- `pkg/errcode/errcode.go`：在文件末追加新错误码
- `Makefile`：追加新命令
- `go.mod / go.sum`：`go get` 新依赖

## 十一、收尾交付物（v4 版，四样）

用户发出收尾指令（"收尾"/"这步完成了"/"总结交付"/"结束本步"）时，**依次产出**：

### 1. 更新 `docs/progress/0x_项目进度追踪.md`

精简版，目的是给下次 Claude Code 启动续上上下文。包含：

- 当前在第几步、步骤名
- 本步已完成的子模块清单（数据层 ✅ / 业务层 ✅ / 接入层 🔄 / 收尾 ⬜）
- 下一个要做的子模块（如本步未完成）
- 全部步骤列表（带 tag 名）
- **「本步与 PRD 的偏离点」** 章节：
  - 格式：`PRD 章节 / 规则编号 → 实现差异 → 选择此方案的原因`
  - 无偏离写"本步严格按 PRD 实现，无偏离"
  - 这是 Step 12 / 17 生成新版 PRD 的唯一依据

### 2. `docs/storylines/0x_故事线-第N步-模块名.md`

**Part A · 本步故事线（800-1500 字）**

按时间线展开，目的是积累面试可复述材料：

1. **起点**：进入本步时的项目状态（前一步遗留什么、本步要解决什么）
2. **设计思考**：2-3 个关键技术决策的取舍过程，关联 PRD ADR 编号
3. **实现节奏**：开发过程的关键节点（哪几个文件先写、Consumer 怎么测）
4. **踩坑记录**：真实发生的问题（不是预想的）
5. **收获**：理解上的提升 + 后续步骤的伏笔

**Part B · 关联知识点清单（只列名字不展开）**

分六类（无内容写"（无）"）：

- 基础概念（语言 / 框架层面）
- 设计模式与架构思想
- Go 语言特性与并发模型
- 工程实践（测试 / 部署 / 可观测性等）
- 数据安全与合规
- 可能被延伸追问的关联领域

数量以本步实际涉及为准，不凑数。

### 3. `docs/changes/0x_代码变更清单-第N步-模块名.md`

按层组织本步新增 / 修改的文件，**不放具体代码**，格式：

```markdown
## 数据层

- `migrations/000NNN_xxx.up.sql`（新增）
  · 表结构要点、索引设计意图（80 字内）

- `internal/repository/xxx_repository.go`（新增）
  · 接口：`XxxRepository`
  · 方法：`Create` / `FindByID` / `List`
  · 设计要点：80 字内说明为什么这么设计

## 业务层
...
```

承担"边写边讲解"的职能——Claude Code 把每个设计要点写进这里的"设计要点"里，
不在终端输出。用户在 VS Code 看代码 + 看这份清单 = 理解本步设计。

### 4. commit message 草稿

直接可复制使用，含本步偏离点摘要。格式：

```
feat(step-N): <模块名> 实现完成

主要变更：
- 数据层：...
- 业务层：...
- 接入层：...

偏离点：
- <PRD 章节> → <实际实现> → <原因>（无偏离写"严格按 PRD 实现"）

验证：scripts/verify_step_N.sh 全部通过
```

### 收尾后的提醒（不生成文件，对话中说明）

- 提醒用户 `grep -r "AccessKey\|secret\|password" docs/` 自检敏感信息
- 提醒用户 `git add docs/ && git commit -m "<commit message 草稿>"`
- 提醒用户 `git checkout main && git merge --no-ff feature/step-N-<模块名>`
- 提醒用户 `git tag step-N-done && git push origin main --tags`

## 十二、学习硬性规定

本项目兼顾面试准备，下列规则不可省略：

1. **设计要点必须沉淀进代码变更清单**，每个新文件 80 字内说明"为什么这么设计"。
   不要在终端输出讲解（节省 token）。
2. **故事线 Part A 必须按时间线叙事**，不要总结型干描述。
   "我先尝试 X 发现 Y 改用 Z" 这种密度才有面试价值。
3. **每个技术决策必须能关联到 PRD ADR 编号**。找不到对应 ADR 时停下来问用户。
4. **遇到 PRD 没覆盖的设计点**，先输出 2-3 个方案对比，让用户选，不要默默实现。

## 十三、其他规则

- 代码必须可直接运行，禁止 `// TODO: implement` 占位
- 敏感配置（.env、阿里云 AccessKey 真实值）走环境变量，禁止 commit
- 写代码前先用 Read 工具看现有文件，不要凭记忆
- 修改文件时优先用 Edit 工具的精确替换，避免全文重写
- 一次回复内不要同时改超过 5 个文件，分批确认
- 中文回答，技术术语保留英文

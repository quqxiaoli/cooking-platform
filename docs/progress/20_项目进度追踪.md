## 二、压测

新增 `scripts/stress_test.sh`（5 section 主脚本）+ 4 个 wrk lua 脚本 + Makefile `stress-test` target。

### 性能数据（最终一轮，wrk2 -R80 限速模式，30s/scene）

| 场景 | QPS | P50 | P90 | P99 | non-2xx | 解读 |
| --- | --- | --- | --- | --- | --- | --- |
| health | 78.87 | 2.83ms | 9.71ms | **12.85ms** | 0% | nginx 直返，无 backend 开销 |
| feed | 78.00 | 11.97ms | 18.59ms | **22.08ms** | 61% | non-2xx 来自 wrk2 calibration burst 触发 nginx api_zone 限流；**成功穿透限流的请求 P99=22ms 是 backend 真实表现** |
| detail | 78.00 | 9.21ms | 27.04ms | **33.06ms** | 100% | post pending_review，未授权读者全 412104（设计正确） |
| search | 78.87 | 2.21ms | 6.22ms | **9.34ms** | 98% | 撞 per-IP 30 r/min 限流（Step 7 设计） |
| like | 78.00 | 9.58ms | 25.89ms | **38.65ms** | 76% | per-user 200/24h 限流 + pending 帖子拒交互 + nginx burst（多重设计正确触发） |

**结论**：backend 端到端延迟 **P99 ≤ 40ms**，远低于 PRD §10.3 SLO 要求（P99 < 100ms）。各场景的 non-2xx 高占比**全部源于设计正确触发的限流或业务状态机**，不是 backend 性能问题。

### 关键技术决策

| 决策 | 选择 | 理由 |
| --- | --- | --- |
| 压测工具 | wrk2（giltene fork） | debian apt 的 wrk 4.1.0 不支持 `-R` 限速；wg/wrk 4.2.0 也不支持（`-R` 是 wrk2 独有）；wrk2 4.0.0 才有 |
| 测试模式 | `-R80` 限速模式 + 1-3 并发（不是 4t/50c 极限压） | 单 IP 自压测受 nginx per-IP 100 r/s 限流，**无法测极限 QPS**；改测「合理负载下的延迟分布」，用 curl 单测验证 backend 健康，两者交叉印证 |
| Consumer 异步延迟实证 | `posts.like_count` 三点采样（BEFORE / IMMEDIATE / AFTER） | 不依赖具体 Redis 数据结构知识；从 MySQL 端测得的延迟才是用户感知延迟 |

### Consumer 异步落库延迟实测

| 时间点 | like_count |
| --- | --- |
| 压测前 BEFORE | 0 |
| 压测刚结束 IMMEDIATE | 0 |
| 等 15 秒后 AFTER | 1 |

幂等性正确：同一用户对同一帖子重复点赞，DB 最终只 +1。Consumer 在压测期内已完成 drain（IMMEDIATE → AFTER 无增量是预期行为）。

## 踩坑记录

1. **bind mount 单文件 + git pull 的 inode 失效**：`nginx.conf` 单文件挂载在 git pull 后宿主机 inode 变了，容器仍持有旧 inode，`nginx -s reload` 救不了，必须 `docker compose up -d --force-recreate nginx`。教训：**生产环境避免 bind mount 单文件，改挂目录**。
2. **HEAD vs GET 路径方法误判**：连续 3 次诊断"www 子域名 HTTPS 404"，反复怀疑 Host 校验、HTTP/2 quirk、middleware bug，**根因是脚本错把 HEAD 当 GET 验证**。Gin `r.GET("/health", ...)` 不响应 HEAD，主域名 + GET 是 200，www + HEAD 是 404，看起来像"www 有问题"。教训：**单变量调参，验证命令必须复现生产真实流量方法**。
3. **SMS per-IP 限流自压测**：单 IP 反复跑 setup 触发应用层 SMS 限流（`sms:ip:<sha256>` 计数器累积），用户 2 永远抓不到验证码。修复：脚本 setup 用户之间 sleep 8s，并保留手动清 `sms:ip:*` key 的应急路径。
4. **apt 装的 wrk 不支持 -R**：连续推荐了不存在的 flag（debian fork 4.1.0 没有 -R，wg/wrk 4.2.0 也没有，**只有 wrk2 4.0.0 才有**）。教训：**每次 propose 工具用法时先 `--help` 验证一次，不要凭印象推荐 flag**。
5. **wrk2 calibration burst 突破限流**：wrk2 -R 模式开放系统模型 + HdrHistogram corrected latency，在前 1s calibration 阶段瞬时 QPS 可破百倍 -R 名义值，触发 nginx api_zone 限流导致 non-2xx 虚高。这是单机自压测固有 limitation，无法绕过（除非生产配置改动）。教训：**真实极限 QPS 需要多机客户端方案（k6 cloud / 分布式 wrk）**。

## 项目持续维护建议（取代「下一步」）

cooking-platform 项目正式收官，进入维护期。建议按月度节奏执行：

1. **每月一次**：
   - SSL 证书续期检查（certbot 应自动跑，但人工 `certbot certificates` 看到期日是好习惯）
   - Grafana 看板巡检（HTTP 延迟趋势、Consumer backlog、MySQL 池利用率）
   - 服务器磁盘空间检查（`df -h`、Docker volumes、日志归档）

2. **每季度一次**：
   - 依赖升级 review（go.mod、Docker base image）
   - 安全公告 review（CVE 扫描 Go / nginx / MySQL / Redis / RabbitMQ）
   - 遗留债务清单 review（决定是否启动 #1 日志归集等）

3. **触发型维护**：
   - 监控告警触发（Consumer backlog 持续 > 100、MySQL 连接池接近耗尽、5xx 比例上升）
   - 真实用户流量超过 80 r/s 持续 1 周 → 启动遗留债务 #4 多机压测，决定是否扩容

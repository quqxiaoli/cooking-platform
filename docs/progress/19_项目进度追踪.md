## 踩坑回顾（精炼版，完整复盘见故事线 Part A）

### 1. 「Viper 嵌套 env 覆盖」的工程级稳健解

- 单 `AutomaticEnv()` 在嵌套 key + yaml 同名值场景下不可靠（mapstructure 合并路径的边界 bug）
- 正解：`BindEnv("database.dsn", "APP_DATABASE_DSN")` 注册键路径让 `v.Get` 走 env 优先级，**Unmarshal 后再用 `v.GetString` 强制回填到 struct**，彻底绕开 mapstructure 对嵌套 key 的不可靠合并
- 教训：上窗口选择「python 盲注 + 同时改 yaml dsn 为空」违反单一变更原则，引入了"零输出 exit 1"次生症状；本窗口正解在干净基线、最小变更上落地，commit `c4c3d2c` 一次过

### 2. 「零输出 exit 1 幽灵」的真身 = 观测路径盲区

- prod 配置 `log.console:false` + `filename:/var/log/cooking/app.log`，zap 全部写文件、stdout 一字不出
- 上窗口的"零输出"现象**从来不是 app 没启动**，是 debug 工具链与 prod 日志归集策略错位（盯 docker logs / stdout，但日志在容器内 `/var/log/cooking/app.log`）
- 教训：**可观测性设计要包括"调试时怎么看日志"**——prod 安静哲学和 debug 习惯天然冲突，必须明示工具链路径。这是本步真正的工程财富，比"6 迁移跑通"重要得多

### 3. GTID + super_read_only 的精巧边界

- slave 上 `super_read_only=ON` 拦截所有外部写入，但 replication SQL 线程作为"系统线程"被例外放行
- master 上 `CREATE USER`/`CREATE TABLE` 等 DDL 经 GTID 自动 apply 到 slave
- 这让"master 单点建用户/迁移、slave 自动同步"成为成立的工程范式

### 4. 一次性容器模式 vs daemon 模式在调试中的价值差

- `docker compose up -d` 的 app 在 `restart:always` 下崩了会陷入 Restarting 循环、日志被快速重启冲刷
- `compose run --rm --no-deps` 在受控一次性容器里跑同样的环境映射，错误现场被冻结、可慢慢看，这是本步顺利通过 Risk 2 的方法论基础

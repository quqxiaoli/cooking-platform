本步无 migration、无 model、无 cache：`ft_title` 索引复用 Step 4 的 migration 000002，搜索复用 `model.Post`，搜索结果不缓存。

**关键决策**：

1. **SearchRepository 独立，不挂 PostRepository**：PostRepository 的读全部是 created_at keyset 游标（已文档化的契约）；FULLTEXT 相关度排序无法 keyset 分页，强行合并会污染 post_repository 的"全游标"不变量。独立 repository 让每个文件的分页故事内部自洽。
2. **offset 分页（PRD §5.4 的明确例外）**：FULLTEXT relevance score 每次查询重算、有 ties、随数据漂移，不具备 keyset 游标所需的稳定单调列；`(relevance,id)` 元组游标会导致跨页跳行/重复。搜索是"前 1-3 页"访问模式，offset 深翻页代价在此场景可忽略。`cursor` 参数对外仍是不透明字符串（装 offset 十进制），API 形态与 §7.1 一致，留迁移空间。
3. **AuthorAssembler 提取 + post_service 同步重构**：search 结果也是"带作者快照的 post 列表"，与 feed 装配逻辑同构。选择把逻辑提取为唯一事实源、并同步重构 post_service 复用它，而非让 search 复制 30 行 —— 彻底 DRY，不留两份装配逻辑漂移的技术债。
4. **R-1 走方案 B（service 显式赋值）而非 06 报告原推荐的方案 A**：fetch 到 post_service.go 后发现代码库已有先例 —— post 的 Create 就是显式赋值 CreatedAt/UpdatedAt 并附注释论证。为全代码库一致选方案 B，user 与 post 都走"service 层显式赋值"。
5. **BOOLEAN MODE 操作符净化**：用户关键词里的 `+ - > < ( ) ~ * " @` 会被 MySQL 当操作符解析。MVP 保守选择 strip 这些字符按字面匹配，牺牲 power-user 布尔语法换 99% 普通搜索的可预测安全行为。
6. **搜索限流硬编码在路由（30 次/60s/IP）**：fetch 后发现 post/like 限流参数全是 main.go 硬编码，`config.yaml` 的 `ratelimit:` 段只放 SMS 三维限流。为与现有代码一致，搜索限流也硬编码，不入 config（修正了开工假设清单第七条）。

**验证方式**：`make verify-step7` —— 8 大类断言全部通过（R-1 回归 created_at 非零值 / 关键词命中 + 反例排除 / scene_tag 服务端过滤 / offset 游标翻页 / 错误用例 450101+450102+400001 / AC-6 未登录可搜 / AC-7 超长关键词截断不报错）。

**遗留问题**：
- **搜索范围仅 title**：二级标签全文检索数据 P1 才有，FULLTEXT 索引也只建了 title（见偏离点 #2）。
- **关键词高亮未实现**：AC-3 高亮由前端处理，后端只返回原始 title（见偏离点 #5）。

## 第 7 步踩坑记录

1. **GitHub MCP 桌面端 ≠ 网页端 Connector**：Claude.ai 网页端 Project Connectors 的 GitHub 集成对 Claude Desktop 无效。桌面端必须改 `claude_desktop_config.json` 配本地 MCP server。配置要点：保留已有 `preferences` 段、PAT 走细粒度只读（仅 Contents+Metadata）、完全退出重启而非关窗口。

2. **`verify_common.sh` 名存实亡**：Project Instruction 的"脚本思想"规范描述了一个公共脚本抽象，但 Step 1-6 实际从未创建它。fetch `scripts/` 目录才发现只有 `backup/migrate/seed/verify_step5.sh`。教训：v3 工作流下 fetch 目录列表先于假设文件存在，是必要的探查动作。

3. **分组生成时 Group ② 报 `undefined` 是预期现象**：search_service.go 引用了 Group ③ 才创建的 `dto.SearchQuery` / `errcode.ErrSearchKeywordEmpty`，编译器在 Group ③ 贴完前必然报错。分组生成的依赖方向（业务层依赖接入层 DTO）决定了中间态不可编译，全组贴完才 build。

4. **errcode 段位三方不一致**：实际代码 `410xxx/412xxx` ≠ Project Instruction 的 `41YZZZ/42YZZZ` 方案 ≠ v2.2 §7.3 的"YY=模块号"规划。这是 Step 3-4 就埋下的偏离，本步如实登记并从 450xxx 起改用"每模块独占整段"，但根治要等 Step 12。

5. **`config` import 未使用陷阱**：search_service.go 初稿 import 了 `pkg/config`，但限流参数最终决定不入 service（走路由硬编码），`config` 变成未使用 import 会编译失败。生成代码后必须核对 import 块与实际引用一致。

**关键决策**：

1. **likes 表无 deleted_at**：物理删除即可；软删除会破坏 uk_user_post 唯一约束（取消后再点会主键冲突），除非把 deleted_at 也加进唯一索引（变 uk_user_post_deleted），那会让索引体积/维护成本/查询计划全部劣化
2. **likes 表无 UpdatedAt**：行不可变（要么存在=已点，要么不存在=未点），任何字段不会变更，UpdatedAt 是无效字段
3. **Redis SET + 独立 INCR 计数器双轨**：SET 用于 SISMEMBER 幂等校验 + 未来 SCARD 反查；独立 String 计数避免 SET 被驱逐时 SCARD 性能塌陷；牺牲毫秒级强一致换简洁，最终一致由 LikeConsumer + 周期 reconcile 保证
4. **Like 路由 RESTful（POST/DELETE/GET 同 URL）**：而非 POST /like + POST /unlike；like 关系是一种资源，HTTP 动词表达 CRUD 即可
5. **Like POST 限流 200/24h，Unlike 不限流**：滥用模型不对称（spam 点赞会污染 vanity metrics 触发假通知；mass-unlike 没有等价滥用），限制 unlike 反而拖慢"我反悔了"的合法流
6. **DECR-clamp 用 Lua 脚本，AddLike 不用**：DECR 下溢会让前端看到负数（UX bug，前端往往不处理）；AddLike 多记一次只是后台慢慢对齐，无 UX 副作用，不值得 Lua 复杂度
7. **不允许给自己点赞 = 不实现限制**：与主流平台一致；自我点赞造成的 vanity metric 失真极小，限制反而引入边界 case
8. **错误码完全复用**：帖子不存在用 412104，限流用 429001，未登录用 401001——本步不新增 13xxxx 段，避免错误码爆炸
9. **LikeConsumer 同时订阅 like + unlike topic**：双向操作落到同一 batch loop，"先 L 再 U"和"先 U 再 L"都能在同 tick 内净化为零，避免两个 consumer 在 like_count 上互相覆盖
10. **批量 INSERT IGNORE + RowsAffected 反推 delta**：天然抗重复投递（uk_user_post 唯一索引），不需要额外幂等表；scaleDeltas 比例分配在 RowsAffected ≠ 输入长度时仍能保持 sum 守恒
11. **PV/Like UPDATE 只用增量 SQL 不用绝对值**：多实例场景必须，绝对值 SQL 在 Step 15 双实例 LB 时会出现 lost-update 灾难
12. **CountConsumer 三 topic 一个消费者**：跨 topic 同 user 的 delta 可在一条 UPDATE 完成（post_count + total_likes 一起更新），减少 users 表（auth 热路径表）的行锁压力
13. **GREATEST(0, CAST(total_likes AS SIGNED) + ?)**：UNSIGNED 列减法 underflow 在严格模式下会抛错；GREATEST 钳位是最少代码量的兜底方案，配合 CountConsumer 周期性 reconcile 自愈
14. **3 个 Consumer 都用"订阅 goroutine + 批量 goroutine"双层模式**：订阅 goroutine 只负责 unmarshal 和入队，批量 goroutine 是单消费者拥有完整批状态——避免 batch buffer 上的并发竞争
15. **优雅停机：drain 用 context.Background()**：父 ctx 已取消，但 MySQL 写还要完成，否则用户的"我刚点的赞"在重启边界丢失。ConsumerManager.wg.Wait() 兜住总停机时间
16. **main.go consumerMgr.StartAll() 时机调整到 7.6 末尾**：第 4 步占位空启动是无 Consumer 的，本步是其正式启用时机；阶段 6 退化为只创建空 Manager（仅为 shutdown 阶段持有变量引用）
17. **scaleDeltas 用 insertion-sort 而非 sort 包**：批最多 50 条 → 数组 ≤50 元素，insertion-sort 在小数据集上比 sort.Slice 反射调度更快；纯库内部 helper 不引入新依赖

**遗留问题**：
- **Channel 模式下 Publish 失败（bus 已关闭）只 log 不重试**：Step 13 切 RabbitMQ 时引入 transactional outbox 或 RabbitMQ confirm 解决

## 第 5 步踩坑记录

1. **`response.OK` vs `response.Success`**：handler 包统一用 `response.Success(c, data)`，第 4 步 feed/post handler 都是这个；Claude 第一版生成 like.go 时误写为 `response.OK`，编译报 undefined。教训：跨包调用前必须 `grep` 确认公开方法名，不能记忆推断。

2. **ConsumerManager StartAll 时机**：第 2 步 main.go 在阶段 6 直接 StartAll 一个空 manager（占位），第 5 步必须把 StartAll 推到所有 Consumer Register 之后（阶段 7.6 末尾），否则启动日志会出现 `consumer manager started {"consumers":0}` 但帖子点赞后事件被 drop。

3. **MockSender 验证码抓取**：本地开发时一定要用 `go run ./cmd/server 2>&1 | tee logs/dev.log` 启动，否则验证脚本 grep 不到 `[SMS MOCK]` 行。这个习惯从第 5 步起固化在 `scripts/verify_step5.sh` 里。

4. **CountConsumer 收事件入 channel 设计**：订阅 goroutine 入队用 `select case eventCh <- ...: case <-ctx.Done():` 让 channel 满时阻塞传播 backpressure 到 ChannelBus，而非 default 丢弃，避免 like 事件被静默吞掉。

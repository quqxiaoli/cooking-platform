### 第 2 步：EventBus Channel 实现（2026-04-24）

**产出文件**：
- `internal/event/types.go` — Topic 常量（6 个）+ 事件信封 Event 结构体 + 6 种 Payload 结构体（LikeEvent / UnlikeEvent / PVEvent / AuditEvent / CountEvent / PostEvent）
- `internal/event/bus.go` — EventPublisher / EventSubscriber / EventBus 接口 + Handler 类型定义
- `internal/event/channel.go` — ChannelBus 进程内实现 + MustMarshalPayload 辅助函数
- `internal/event/channel_test.go` — 4 个单元测试（发布订阅、多订阅者广播、Close 后 Publish 返回 error、ctx cancel 排空）
- `internal/consumer/consumer.go` — EventConsumer 接口
- `internal/consumer/manager.go` — ConsumerManager 结构体 + Register / StartAll / Shutdown
- `cmd/server/main.go`（更新）— 新增步骤 5 EventBus 初始化、步骤 6 ConsumerManager 启动；优雅关闭顺序改为 HTTP Server → ConsumerManager → EventBus → Redis → MySQL

**关键实现决策**：
- Topic 用字符串常量而非 iota int，便于 RabbitMQ routing key 对齐 + 日志可读
- Event.Payload 用 `json.RawMessage`：序列化零反射开销；Consumer 按需 Unmarshal 到具体结构体；切 RabbitMQ 时消息格式天然兼容
- Handler 签名 `func(ctx, payload []byte) error` 而非 `func(Event) error`：不强迫 Consumer 感知 Event 信封，切 RabbitMQ 后 Handler 零修改
- ChannelBus 每个订阅者独立 channel（不是共享 channel）：多订阅者广播互不影响，慢 Consumer 不饿死快 Consumer
- bufSize=1024：约 200KB/topic 内存，可吸收约 1 秒的 1000 events/s 突发
- Publish 非阻塞（select+default）：Publish 在用户请求路径上，channel 满时丢弃 + warn 日志比阻塞业务线程更安全
- Close 只设标志位、不关闭 channel：goroutine 生命周期交给 ConsumerManager；避免 close(ch) 后 Publish panic
- ConsumerManager.Shutdown 先 cancel ctx 再 wg.Wait：顺序反了会死锁
- Subscribe 在 ctx cancel 后调用 drain()：用 context.Background() 处理剩余消息，避免 handler 内部检查 ctx.Err() 导致提前退出

**已验证**：

```bash
go test ./internal/event/... -v
# --- PASS: TestChannelBus_PublishSubscribe
# --- PASS: TestChannelBus_MultipleSubscribers
# --- PASS: TestChannelBus_CloseStopsPublish
# --- PASS: TestChannelBus_DrainOnCancel

make run
# 启动日志顺序：mysql connected → redis connected → event bus initialized(provider=channel)
#              → consumer manager started(consumers=0) → http server listening

# Ctrl+C 后关闭日志顺序：
# shutdown signal received → http server stopped → consumer manager shutting down...
# → consumer manager stopped → event bus closed → redis close ok → mysql close ok
# → server exited cleanly
```

**遗留问题**：
- `make run` 收到 SIGINT 后 Go 进程返回非零退出码，make 显示 `Error 1`（非实际错误）。第 6 步全链路验证时可在 Makefile 的 run 目标前加 `-` 前缀忽略
- Channel 模式下消息不持久化，进程重启即丢失；Channel 模式不支持多实例消费。两项都是第 13 步切 RabbitMQ 后解决

## 实现过程中的偏离记录

> 编码时如果发现 v2.1 架构文档的设计需要调整，记在这里。
> 下次打开新窗口时 Claude 能看到这些调整，避免重复踩坑。

### 第 2 步偏离记录

**偏离 1：Handler 签名调整**
- PRD-Phase3 v2.1 § 4.4 写的是 `Subscribe(ctx, topic, handler func(Event) error)`
- 实际实现改为 `Subscribe(ctx, topic, Handler)`，其中 `Handler = func(ctx context.Context, payload []byte) error`
- 原因：①多传一个 ctx 便于 Handler 内部做超时控制和请求追踪；②payload 直接给 `[]byte`，Consumer 按需反序列化，切 RabbitMQ 后零修改（MQ 消息体本就是 bytes）

**偏离 2：EventBus 接口组合方式**
- PRD-Phase3 v2.1 § 4.4 定义 `EventBus { Publisher() EventPublisher; Subscriber() EventSubscriber; Close() error }`（方法返回接口）
- 实际实现改为 `EventBus` 直接组合嵌入 `EventPublisher` + `EventSubscriber` + `Close()`（接口嵌入）
- 原因：业务代码可以直接 `bus.Publish(...)` 而不是 `bus.Publisher().Publish(...)`，API 更简洁；切 RabbitMQ 时实现代码只需提供 Publish/Subscribe/Close 三个方法即可，不需要额外的 Publisher() / Subscriber() getter

**偏离 3：ConsumerManager 归属 internal/consumer 而非 internal/event**
- PRD-Phase3 v2.1 § 3.4 示例代码把 ConsumerManager 放在 event 包
- 实际拆为 `internal/consumer/manager.go`
- 原因：职责分离——event 包只管消息传递，consumer 包管 Consumer 生命周期。后续第 5 步起 LikeConsumer / PVConsumer 等业务 Consumer 也放在 consumer 包下，组织更清晰

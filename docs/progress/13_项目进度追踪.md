**关键决策**：

1. **命名队列格式 `cooking.<topic>`**：EventSubscriber 接口固定为 `(ctx, topic, handler)`，无 consumer_group 参数，改接口会破坏已有契约。决定用 `cooking.<topic>` 代替 conventions 规定的 `cooking.<topic>.<consumer_group>` —— 单应用场景下等价，多实例竞争同一队列天然负载均衡。（见偏离点）

2. **DLX 重试策略：nack→直接 DLX**：经典队列无 `x-delivery-count`，无法在 RabbitMQ 层做重试计数。handler error → `Nack(false, false)` → 直接进 DLX，避免 poison message 无限重排。transient error 应由 handler 内部逻辑自行处理（AuditConsumer 已是 fail-closed 设计）。

3. **重连架构：Subscribe 内部循环**：重连逻辑内置于 `Subscribe` 的 retry loop，`ensureConnLocked` 保证多个并发 Subscribe goroutine 只有一个真正 redial。Publish 侧用 `reopenPubChLocked` 懒重连。无需改动 ConsumerManager。

4. **Config-First 严格遵守**：`ReconnectMaxRetries` / `ReconnectInitialDelay` 全部进 `MQConfig`，`NewRabbitMQBus` 改为接受整个 `config.MQConfig`（而非 url + timeout 两个零散参数）。

**本步与 PRD 的偏离点**：

| PRD 章节 / 规则 | 实现差异 | 选择此方案的原因 |
|---|---|---|
| conventions §1.4 队列命名 `cooking.<topic>.<consumer_group>` | 实际使用 `cooking.<topic>`（无 consumer_group 后缀） | `EventSubscriber` 接口不含 consumer_group 参数，修改接口会破坏所有现有 Consumer；单应用场景下两者语义等价 |
| conventions §1.4 "countBatchSize 次 Nack 后进 DLX" | handler error 直接 `Nack(requeue=false)` 进 DLX，无重试 | 经典队列无 `x-delivery-count`，重试计数需引入 quorum queue 或 Redis 辅助计数，复杂度不成比例；handler-level transient error 应在 handler 内部重试 |

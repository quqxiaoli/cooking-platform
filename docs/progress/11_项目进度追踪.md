**关键决策**：

1. **dev 空 key 向后兼容**：`EncryptPhone("", "")` 返回明文，`HashPhone(phone, "")` 等于旧 `SHA256(phone)`，Step 3-10 期间注册的用户无需迁移即可正常登录。
2. **GCM auth-tag 作为状态探针**：迁移脚本通过 `DecryptPhone` 失败判定"明文行"，不需要额外 `is_migrated` 状态列，不改 schema。电话号码（全数字，≤15 字符）base64 解码后不足 12 字节 nonce，GCM 必然报错，零误判。
3. **GetMyProfile 解密失败降级**：`plain=""` → `maskPhone("")` 返回空串而非 500，避免密钥轮换未重跑迁移时全体用户看到 500。
4. **行级原子更新，不用大事务**：每行独立 UPDATE，迁移中途崩溃可安全重跑。
5. **本步无新 migration SQL**：`phone_encrypted VARCHAR(200)` 和 `phone_hash VARCHAR(64)` 在 Step 3 时已按 PRD v2.2 §5 预留，schema 无需变更。

**验证方式**：

```bash
go test ./pkg/crypto/... -v -count=1   # 10 个单元测试（本地已通过）
go build ./...                          # 全项目编译（本地已通过）
make docker-up && make migrate-up
go run ./cmd/server 2>&1 | tee -i logs/dev.log
make verify-step11                      # 另一终端，8 段端到端验证
```

verify_step11.sh 8 段覆盖：
- §1  pkg/crypto 单元测试全部通过
- §2  migrate-phone --dry-run 在 dev（key=""）输出 nothing to migrate
- §3  send-code + login 成功
- §4  GET /users/me → phone_masked 符合 3位****4位 格式
- §5  MySQL phone_hash == SHA256(phone)（dev pepper=""）
- §6  MySQL phone_encrypted == 明文（dev key=""）
- §7  logout 后原 token 返回 401003（黑名单机制不受影响）
- §8  有用户时 migrate-phone --dry-run 仍幂等

**本步与 PRD 的偏离点**：

| PRD 章节 | 实现差异 | 选择此方案的原因 |
|---|---|---|
| PRD v2.2 §6.3.4 ADR-022：pepper 与 phone_hash 一起做 `SHA256(phone \|\| pepper)` | 实现为 `SHA256(phone + pepper)`（字符串拼接，非二进制拼接）；语义等价，Go `crypto/sha256` 接收 `[]byte(phone + pepper)` | Go 标准库没有流式 SHA256 拼接 API，字符串拼接结果完全等价；pepper 作为密钥不含 NUL 字节，无长度扩展攻击风险 |
| PRD v2.2 §6.3.4 Step 11 迁移步骤 3（phone_hash 字段保持不变的表述） | 迁移脚本同时更新 phone_encrypted 和 phone_hash（当 pepper 变化时）；schema 确实不变，但行值可变 | PRD 原文"字段保持不变"指 schema（无 ALTER TABLE），迁移期数据行仍须更新 hash 值才能让 pepper 生效 |

**关键决策**：

1. **config.docker.yaml + CONFIG_PATH 而非 env var 覆盖 SlavesDSN**：Viper 对数组型字段的 env var 覆盖行为不直观（需 JSON 或逗号格式），难以在 docker-compose.yml 里干净表达。用显式配置文件加环境变量路径切换，三套配置（dev/docker/prod）分层清晰，面试中更容易解释。

2. **移除 app1/app2 对 mysql-slave-init 的 depends_on**：首次运行后再次 `docker compose up -d` 时，Compose 重建 mysql-slave-init 容器，第二次运行遭遇 `super_read_only=ON` 导致 exit 1。App 只需要 MySQL master/slave 容器健康即可启动；replication 是基础设施关注点，不是应用启动前提。

3. **`/health/ready` 别名路由**：conventions §2 明确要求此路径，现有 `/readiness` 保留向后兼容。两个路由映射到同一 handler，语义完全一致。

4. **被动 health check（passive）代替主动 health check（active）**：Nginx 开源版不支持 active health check，需 nginx-plus 或 `ngx_http_upstream_check_module`。`proxy_next_upstream error timeout http_503` 是开源版标准做法。（登记为偏离点）

**验证结果**：


**关键决策**：

1. **super_read_only 不进 slave.cnf**：MySQL Docker entrypoint 使用相同 conf.d 文件启动临时 mysqld 初始化数据目录。super_read_only 在此阶段阻止 root 用户创建，导致容器永久无法连接。改为在 init-slave.sh 最后 `SET GLOBAL super_read_only=ON`，分离"启动时初始化"与"容器运行时保护"两个关注点。同理，`command: ["mysqld", "--super-read-only=1"]` 方案也被排除——Docker entrypoint 会将该参数同时传给临时 server。（见偏离点 1）

2. **mysqldump 全量快照而非 GTID-skip 技巧**：最初尝试设置 `SET GLOBAL gtid_purged` 跳过 master 已清除的日志（transactions 1-39），但这同时跳过了建表 DDL。slave 数据目录为空，当 master 发来 DML（INSERT INTO schema_migrations）时报 error 1146（表不存在）。正确做法：`mysqldump --single-transaction --set-gtid-purged=ON` 产生一致性快照，slave 从完整状态起步，只需追赶增量。（见偏离点 2）

3. **DBResolver 仅在 SlavesDSN 非空时注册**：`SlavesDSN: []` 时跳过 `db.Use(resolver)`，单机模式下行为不变，现有所有 Consumer 和 Repository 代码零感知。连接池参数同时通过 `sqlDB.SetMax*` 和 `resolver.SetMax*` 配置，master 和 slave 使用相同的池设置。

| PRD 章节 / 规则 | 实现差异 | 选择此方案的原因 |
|---|---|---|
| conventions §2 Step 14 "slave read_only=ON enforced via slave.cnf" | read_only / super_read_only 不在 slave.cnf，由 init-slave.sh `SET GLOBAL` 设置（容器生命周期内有效，重启后需重跑 init 容器） | MySQL Docker entrypoint 使用相同 conf.d 启动临时 server 初始化数据目录；conf.d 中的 super_read_only 阻止 root 用户创建，导致容器永久不可用；`--command` 方案同理失效 |

```json
  "nickname": "厨友8000",
  ...
  "created_at": -62135596800000   ← Go time.Time 零值序列化
}
```

**根因**：

`UserService.Login` 在首次登录自动注册流程中，构造 `&model.User{}` 然后调用 `repo.Create(ctx, user)`。MySQL 通过 `DEFAULT CURRENT_TIMESTAMP(3)` 在 INSERT 时填入了 `created_at` 真实值，但 GORM 不会自动把 DB 默认值回填到 Go 内存中的结构体；后续 `toPublicResp(user)` 序列化时拿到的还是零值时间。

`GET /users/me` 走的是 `repo.FindByID()` SELECT 流程，从 DB 读出真实值，所以正确。

**影响**：
- 仅影响 login 响应一次性返回的 user 对象
- 客户端立即可以通过 `/users/me` 拿到正确值
- 不影响数据正确性（DB 内 created_at 是对的）

**修复方案（留 Step 7 开头处理）**：

二选一：

**方案 A**（推荐）：在 `internal/model/user.go` 的 `CreatedAt` 字段加 GORM 标签：

```go
CreatedAt time.Time `gorm:"column:created_at;autoCreateTime:milli;..."`
```

GORM 在 Create 前自动用 `time.Now()` 填充内存对象，DB 默认值变成幂等兜底。

**方案 B**：在 `UserService.Login` 自动注册分支显式赋值：

```go
user := &model.User{
    PhoneHash:      phoneHash,
    PhoneEncrypted: phone,
    Nickname:       defaultNickname(phone),
    CreatedAt:      time.Now(),   // 显式赋值，避免依赖 DB 默认值
}
```

**为什么不在第 6 步修**：第 6 步硬规则不改 .go；该 bug 不阻塞任何核心链路；Step 7（搜索模块）正好涉及 user_repository 改动，修复一行成本最低。

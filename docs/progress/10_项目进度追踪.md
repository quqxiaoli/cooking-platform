**关键决策**：

1. **一跳架构（PRD §9.2 [v2.2] 权威）**：PostService.Create 发 AuditEvent(AuditStatus=0) 到 TopicAudit；AuditConsumer 订阅 TopicAudit 后调 Aliyun API 写结果。最初误读 §1 旧框架设计了两跳（Consumer订TopicPost→API→发AuditEvent），被用户纠正后收敛。
2. **AuditStatus=0 复用为提交信号**：AuditEvent.AuditStatus=0（model.AuditStatusPending）作为"请审核此帖"的提交标记；AuditConsumer 检查 AuditStatus==0 才调 API，非 0 则跳过（为未来人工审核 admin 接口留路）。
3. **fail-closed 降级**：Green API 失败时帖子保持 is_visible=0 / audit_status=0（pending），不强行翻转。日志打 errcode=470101 供 Prometheus 告警。
4. **audit_log 先写，UpdateAuditStatus 后写**：即使 UpdateAuditStatus 失败，合规日志已落库；audit_log 写失败则继续尝试 UpdateAuditStatus（帖子状态优先级更高）。
5. **SDK 选型**：SMS 用 dysmsapi，Green 用 alibaba-cloud-sdk-go/services/green（ImageSyncScan，非 ImageScan，实际编译报错后修正）。
6. **is_visible=0 是新帖默认值（PRD §9.2 Step 10 起）**：打破了 Step 4-9 的 dev mode 临时策略；MockAuditor 毫秒级处理，feed + GetDetail 行为实际无感知。
7. **post_count 计发帖时刻**：PostEvent 发出后 CountConsumer 立即 +1（含审核中帖子），与 PRD §1 行 86-87 语义一致，不改 CountConsumer。
8. **MySQL 8.0 TEXT 列不允许 DEFAULT 值**：`raw_response TEXT NOT NULL DEFAULT ''` 在严格模式下报 Error 1101，移除 DEFAULT 即可；migration 因 dirty state（版本 6）需先 `migrate force 5` 再 `migrate up` 重跑。

|---|---|---|

**关键决策**：
1. **JWT 算法 HS256 + alg confusion 防御**：parse 时硬校验 `*jwt.SigningMethodHMAC`，挡掉 alg:none 攻击
2. **Claims 结构 uid (int64) + jti (UUIDv4)**：uid 替代标准 sub 字段，省 strconv 转换；jti 客户端生成，零网络往返
3. **JWT 黑名单 TTL 动态计算**：登出时取 access token 剩余 ttl，过期后自然清理 Redis
4. **手机号 + 验证码登录"找不到自动注册"**：去除独立注册接口，UX 简洁
5. **Refresh 不旋转旧 token**：MVP 不做 refresh token rotation，第 11 步加密迁移时再评估
6. **Auth 中间件只注入 user_id 不预加载 User**：节省每请求一次 DB 查询，热路径友好
7. **限流"fail open"**：Redis 故障时放行所有请求，避免单点故障让 API 全挂
8. **SMS 限流三维**：phone 短窗 60s + phone 日限 5 次 + IP 日限 10 次，固定窗口实现（非滑动窗口）
9. **IP 哈希后做 Key**：避免明文 IP 进 Redis（GDPR-aligned 习惯）
10. **滑动窗口 ZAdd 仅在 allowed 时执行**：拒绝的请求不消耗窗口槽位，防"窗口卡满"攻击
11. **验证码用 crypto/rand 而非 math/rand**：math/rand 可被预测，攻击者可枚举下一码
12. **手机号 phone_encrypted 第 3 步存明文**：第 11 步迁移到 AES-GCM，phone_hash 字段保持不变
13. **migrations 用 SQL 而非 GORM AutoMigrate**：可审计、可回滚、可 review
14. **Repository UpdateProfile 用 map 而非 struct**：避免 GORM struct 更新跳过零值的陷阱
15. **config Load() 加 validate**：JWT secret 弱、TTL 配错等致命错误启动时立即暴露

**验证方式**：

8 步 curl 全部通过：
- `GET /health`、`GET /readiness` → 200
- `POST /api/v1/auth/send-code` 首次 → 200，验证码打印到日志
- `POST /api/v1/auth/login` 首次 → 200，返回 access_token + refresh_token + user 快照（user.id=1，nickname="厨友8000"）
- `GET /api/v1/users/me` 带 token → 200，返回私有视图（含 phone_masked="138****8000"）
- `GET /api/v1/users/1` 不带 token → 200，返回公开视图（无 phone_masked）
- `PATCH /api/v1/users/me` → 更新 nickname 和 bio 成功
- 短窗限流（同一手机号 60s 内连发 2 次）→ 第 2 次返回 410105 / HTTP 429
- `POST /api/v1/auth/logout` → 200，随后用同 token 调 /users/me 被拒（401003 token invalid）
- 手机号格式错误 → 400001 / HTTP 400
- 优雅关闭：Ctrl+C 后 8 条日志按顺序打印，最终 "server exited cleanly"

**本步与 PRD 的偏离点**：

| # | PRD 章节 | 实现差异 | 选择此方案的原因 |
|---|---|---|---|
| 1 | PRD-Phase2 用户系统模块未列 `/auth/refresh` 和 `/auth/logout` | 实现新增了这两个接口 | JWT 双轨没有 refresh，access token 过期用户立即掉线，UX 不可接受；logout 不写 JWT 黑名单等于"登了等于没登"，安全形同虚设。两接口为 JWT 双轨方案的必备组成 |
| 2 | PRD-Phase3 §10 安全合规未明确 JWT 黑名单 TTL 策略 | 实现取"access token 剩余 ttl"作为黑名单 TTL（动态计算） | 节省 Redis 内存（过期后自然清理）；登出后立刻拒绝；过期后无需再黑名单（签名校验已自动拒绝） |
| 3 | PRD-Phase3 §6.6 滑动窗口示例代码 | SMS 限流采用三维固定窗口（phone 短窗 60s + phone 日限 + IP 日限），非滑动窗口 | 滑动窗口对三维限流复杂度过高（4-5 倍），业务收益接近 0；通用滑动窗口工具仍实现于 `middleware/ratelimit.go` 备用，第 4 步发布限流时启用 |
| 4 | PRD-Phase3 §6.3 Key 设计表 sms:ip:{ip_hash} 未明示 ip_hash 算法 | 实现采用 SHA-256(ip) 十六进制串 | 避免明文 IP 进 Redis，符合 §10 安全合规精神（数据最小化原则） |
| 5 | PRD-Phase3 §5.2 users 表字段 phone_encrypted 设计为 AES-GCM 密文 | 第 3 步先存明文手机号，第 11 步执行迁移 | 第 3 步引入 AES-GCM 会让加密密钥管理（密钥从哪儿读、密钥轮换）成为单步过大负担；分阶段交付更安全，phone_hash 字段不变保证迁移期间无需双写索引 |
| 6 | PRD-Phase3 §5.2 phone_hash 字段类型 | 实现采用 VARCHAR(64) 存十六进制字符串，而非 BINARY(32) | VARCHAR(64) 可在 MySQL CLI 中直接 `WHERE phone_hash='...'` 调试，工程便利性 > 16MB 空间损失（按 50 万用户估算） |
| 7 | PRD-Phase2 用户首次注册的默认昵称未规定 | 实现采用 `厨友` + 手机号末 4 位（如"厨友8000"） | 与 PRD-Phase1 产品定位"松弛、有温度、有场景感"匹配；用户可通过 PATCH /users/me 自行修改 |

**关键决策**：

1. **关注关系同步写 MySQL，不引入 Redis SET 缓存**：关注是低频动作（一辈子关注某人一次），无写放大，不需要 Redis SET + async consumer 那套。同步写 = 关系立即持久化可查，无最终一致窗口。幂等由 `uk_follower_following` 天然保证。**本步不新增任何 Redis Key。**
2. **唯一的异步路径 = users 双向计数，复用 CountConsumer**：一条 `FollowEvent` 影响两个用户的计数（follower 的 `following_count` +1、followee 的 `follower_count` +1），subscribe goroutine 内拆成两笔 countDelta，复用现有按用户聚合的 UPDATE 路径。
3. **`Follow` 检查顺序"最便宜/最具体优先"**：self-check（纯比较）→ target 存在（一次索引 SELECT）→ 已关注（幂等快路径，跳过限额检查与事件）→ 3000 限额 → INSERT IGNORE。**已关注时返回 `following=true` 但不查限额** —— 已存在的关系绝不能被一个本就把它算在内的限额拒绝。
4. **`Unfollow` 不允许幂等成功，必须报 `440103`**：与 `Follow` 的"已关注幂等"非对称设计，遵循 Step 8 提示词的明确语义。Unfollow 是"显式删除"动作，删不存在的资源就该报错。
5. **粉丝/关注列表用 `follows JOIN users`，规避 N+1**：列表项是 user 不是 post，`AuthorAssembler.LoadMap`/`BuildListItems` 结构上不适配；JOIN 一条 SQL 直接返回所需用户字段。注销用户用 `u.deleted_at IS NULL` 过滤直接掉队（与 PRD F-F02 "关注的人全部注销→空状态"语义一致），无需占位行。
6. **`FollowUser` 不内嵌 `model.User`** —— 真·坑点：`model.User` 带 `gorm.DeletedAt`，内嵌后 `.Table("follows AS f")` 查询会让 GORM 把 `deleted_at IS NULL` 拼到 `follows` 表上（它没这列）→ SQL 报错。扁平结构体 + 显式 `u.deleted_at IS NULL` 规避。
7. **keyset 游标走 `follows.id` 不走 `created_at`**：自增 id 单调稳定，无同毫秒并列问题；`idx_following_id` 在 InnoDB 中隐式带 PK，对 `WHERE following_id=? AND id<? ORDER BY id DESC` 天然高效。
8. **关注/取消关注 RESTful（POST/DELETE 同 URL）**：和点赞同款风格，资源是"关注关系"，HTTP 动词表 CRUD。
9. **follow/unfollow 不限流**：与 like POST 不同，关注的滥用向量弱，真正护栏是 3000 上限；加限流只拖慢正常使用。
10. **`CountFollowing` 查 `follows` 表而非 `users.following_count`**：硬上限不能用最终一致的冗余计数判断（CountConsumer 批量 20/10s 有秒级 lag）。
11. **`flush` 由 switch 改为动态 SET 子句拼接**：Step 5 两列手写 3-case switch 还能读，Step 8 四列会炸成 15 case；动态拼让"加列"成本降到 4 行 `if`，聚合-by-user / `GREATEST` 钳位 / drained 日志语义零变化。
12. **`post_count` 不加 `GREATEST` 钳位**：append-only 计数，没有任何事件发负的 post delta，不会下溢；其余三列都可被 unlike/unfollow 减，必须钳位。
13. **`UserBrief` 不复用 `PostAuthorBrief`**：字段相同（id/nickname/avatar_url），但语义不同（一个是 follow-list 行、一个是 post 作者快照）。新 DTO 避免命名错位与未来双向耦合，Step 12 DTO 收口阶段可重新评估。
14. **Unfollow 无 self-check**：Follow 在 step 1 就拦截了 self-follow，自关注边永不存在；Unfollow 直接走 Delete，`RowsAffected=0` 自然产出 `ErrFollowNotFound`，更诚实。
15. **错误码 440xxx 独占整段**：延续 Step 7 起"每模块独占 4MM xxx 块"约定（search=450, follow=440, upload=460, audit=470）。

## Cache 层

### `internal/cache/like_cache.go`（修改）

**M2 — TTL 配置化：**
- 删除 `const likeStateTTL = 7 * 24 * time.Hour`
- `LikeCache` struct 添加字段 `stateTTL time.Duration`
- `NewLikeCache(rdb, stateTTL time.Duration)` 新增第二个参数
- 全文将 `likeStateTTL` 替换为 `c.stateTTL`（影响 `AddLike`、`RemoveLike` 的 Lua EXPIRE 参数）

**M3 — AddLike 原子化（Lua 脚本）：**
- 删除旧的 pipeline 实现（SADD + Expire + INCR + Expire 四步，INCR 无条件执行）
- 新增 `const addLikeScript`（Lua 脚本，27 行）：
  - `SADD KEYS[1] ARGV[1]` → 若返回 0（已是成员），直接 GET 当前 count 返回，**不执行 INCR**
  - 若返回 1（新成员），INCR KEYS[2]，EXPIRE 两个 key，返回新 count
- `AddLike` 实现改为 `c.rdb.Eval(ctx, addLikeScript, ...)`
- **设计要点**：与 `RemoveLike` 的 Lua clamp-DECR 对称。旧实现存在的 off-by-one 场景：并发两次 like 都通过 SISMEMBER 检查 → 各自 INCR → count+2 实际只+1；Lua 脚本消除此竞态。RabbitMQ at-least-once 上线后重复投递概率上升，此修复变得必要而非锦上添花

### `internal/cache/feed_cache.go`（修改）

- 删除导出常量 `FeedCacheTTL`、`PVDedupTTL`（它们是 M2 的一部分，被 post_service 外部引用）
- `FeedCache` struct 添加字段 `feedCacheTTL time.Duration`、`pvDedupTTL time.Duration`
- `NewFeedCache(rdb, feedCacheTTL, pvDedupTTL time.Duration)` 新增两个参数
- `MarkPVSeen` 中 `PVDedupTTL` → `c.pvDedupTTL`
- **设计要点**：`SetFeed` 已接受 `ttl time.Duration` 参数，因此 `FeedCacheTTL` 只在调用方 post_service 使用；删除导出常量后 post_service 改用 `s.cacheCfg.FeedCacheTTL`（见下）

## Event 层

### `internal/event/rabbitmq.go`（新增，H1）

实现 `EventBus` 接口（`EventPublisher` + `EventSubscriber` + `Close()`）。

| 组件 | 设计选择 | 理由 |
|------|---------|------|
| Exchange | `cooking.events`，type=topic，durable=true | topic 类型允许通配符订阅（`event.*`）；durable 在 broker 重启后保留 |
| Queue | server 生成名，ephemeral，exclusive，auto-delete | 每次 Subscribe 独立队列，镜像 ChannelBus 的 per-subscriber channel 模型；消费者不需要任何改动 |
| Routing key | == `evt.Topic`（如 `event.like`）| Topic 字符串常量已适配 AMQP routing key，无需转换 |
| Delivery | `autoAck=true`，`DeliveryMode=Transient` | MVP 阶段 at-most-once，与 ChannelBus 语义一致；Step 13 生产加固改为 manual ACK + Persistent |
| 并发安全 | `pubCh` 共享 + `sync.Mutex`；Subscribe 各自独立 Channel | amqp.Channel 非线程安全，仅 Publish 需要锁 |
| Close | `atomic.Bool` 保证幂等；先关 pubCh 再关 conn | conn.Close() 触发所有 Subscribe 的 msgs channel 关闭，驱动它们 return nil |
| Drain on ctx.Done | 内嵌 for-select-default 循环 | 与 ChannelBus.drain() 逻辑一致；避免重启时丢失已在 Go channel 缓冲区的消息 |

## 补充修复：middleware 解耦（体检漏项）

### `internal/middleware/auth.go`（修改）

- 删除 `import "cooking-platform/internal/service"`（消除 middleware→service 跨包具体类型依赖）
- 新增 `context` import
- 新增 `TokenVerifier` 接口：

  ```go
  type TokenVerifier interface {
      VerifyAccessToken(ctx context.Context, token string) (uid int64, jti string, err error)
  }
  ```

- `Auth(userSvc *service.UserService)` → `Auth(v TokenVerifier)`
- 函数体内 `userSvc.VerifyAccessToken` → `v.VerifyAccessToken`
- **设计要点**：`*service.UserService` 已有该方法签名，满足接口，调用方零改动。接口定义在 middleware 包内（而非 service 包）遵循"接口归消费方"原则，避免反向 import

### `cmd/server/main.go`（修改）

- `main()` 中在 `setupRouter` 调用前新增 `healthHandler := handler.NewHealthHandler(db, rdb)`
- `setupRouter` 签名变更：
  - 移除 `db *gorm.DB`（只为创建 HealthHandler，现已外移）
  - `userSvc *service.UserService` → `tokenVerifier middleware.TokenVerifier`（接口化）
  - 新增 `healthHandler *handler.HealthHandler`
- 函数内 `handler.NewHealthHandler(db, rdb)` 调用删除（handler 已由参数传入）
- 所有 `middleware.Auth(userSvc)` → `middleware.Auth(tokenVerifier)`（共 7 处）
- **设计要点**：`setupRouter` 现在只依赖 `middleware.TokenVerifier`（接口）+ `*handler.*Handler`（handler 是 leaf node，具体类型可接受）+ `*goredis.Client`（rate limiting 合理持有）。`*gorm.DB` 从 router 签名彻底消失

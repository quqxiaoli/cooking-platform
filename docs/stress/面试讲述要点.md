# cooking-platform 全面压测 · 面试讲述要点

> **用途**：给小离做"压测过程"主题面试时的脚手架。
> 配合 `docs/stress/全面压测方案.md` 实际跑完一遍后使用。
> 每个 Phase 都按 **「测什么 / 面试官常问 / 重点观察 / 对应架构」** 四块写。
>
> 用法：跑完每个 Phase，**回到本文件**对照"重点观察"那行，把当时看到的真实数字填进
> 「我的实测」括号里。面试时你讲的就不是教科书，而是你亲手摸过的现场。

---

## §0 30 秒电梯演讲（开场素材）

> "我在一个单机集群双 Go 实例 + MySQL 主从 + Redis + RabbitMQ 的部署上做了一轮
> **从冒烟到崩溃点 7 个 Phase 的全面压测**。目标不是验收，是**真实摸边界**。
> 关键产出是三个：
> ① 单机 `MAX_SAFE_RPS = ___`，瓶颈组件是 `___`；
> ② KPI 实测：读写分离 slave 占比 `___%`、Redis 命中率 `___%`、
>    LikeConsumer drain 时间 `___s`、主从延迟峰值 `___s`；
> ③ 一份 v2 优化路线图，第一优先级是 `___`。"

**面试官 follow-up 一般会落在三个方向**，提前准备好弹药：
1. **方法论**——"你怎么定的 Phase 顺序"（答 §1-7）
2. **架构问题**——"压测验证了哪些架构决策"（答 §对应架构 段）
3. **教训**——"踩了什么坑 / 重做会怎么改"（答 §8 总结）

## §0' 全面压测和 Step 20 验收压测有什么区别？（必问）

| 维度 | Step 20（验收） | 全面压测（探边） |
| --- | --- | --- |
| 目标 | 证明 PRD KPI 达标 | 找真实瓶颈与崩溃点 |
| RPS | 固定 80 r/s（留 20% margin） | 阶梯 100 → 3500 跑到崩 |
| 限流 | **全部保留** | **临时关闭** |
| 安全网 | 保留 | 允许调连接池 / 关 SMS per-IP |
| 通过标准 | KPI 通过即停 | 错误率 > 1% 或 P99 > 1s 才停 |

> "Step 20 是发布门禁，回答的是'能不能上'；全面压测是工程师好奇心，回答的是'真的极限是多少、瓶颈在哪、能撑多少倍突发'。两份报告分别归档到 `docs/stress/step20_*.png` 和 `docs/stress/results/phase_*.md`，决策时机和读者都不一样。"

## §Phase 0  环境准备

### 这一步在做什么
临时把生产的安全网拆掉一部分（nginx 限流 + SMS per-IP），不然单机出口 IP 的所有请求都会被限到 30 r/s，**根本测不到 backend**。同时把客户端机器的资源上限（ulimit / max_connections）调高。

### 面试官常问
1. **"你都关了哪些限流？为什么这么做？"**
   → "关了 nginx `api_zone` 的 30 r/s + `sms_zone` 的 1 r/min + 应用层 SMS per-IP。理由：单机自压所有请求共用一个 IP，限流是给万级真实用户准备的，不是给压测客户端准备的。但**每一条都准备了一行回滚命令**，压完按反序恢复。"
2. **"如果生产用户在用，你怎么办？"**
   → "选半夜 2-5 点低峰窗口；或者起一个独立 staging 环境跑。这次我是项目刚上线还没用户的窗口期。"
3. **"调高 max_connections 会不会影响 MySQL 稳定性？"**
   → "默认 151 是给小流量的，2000 是临时为压测放开。压完用 `SET GLOBAL` 改回来；持久化的改动会走 my.cnf + 重启容器，但这次是 runtime 改动，重启自动归零。"

### 重点观察
- [ ] `docker exec cooking-nginx-prod nginx -T | grep limit_req_zone` 显示 `rate=100000r/s`
- [ ] Grafana 4 块看板都活（HTTP 延迟、Consumer backlog、MySQL 池、Redis 延迟）

### 对应架构
- **CLAUDE.md §9.2**：bind-mount 单文件改了必须 `--force-recreate`，`reload` 不够。这条坑我亲历过。
- **遗留债务 #3**：v2 准备把 nginx.conf 改成目录挂载（`/etc/nginx/conf.d/`）就不用再吃这条苦。

## §Phase 1  T1 冒烟（5 min）

### 这一步在做什么
**10 RPS × 60s × 9 场景**，目标是验证「测试链路本身没坏」。错误率必须 = 0，否则后面什么数字都不可信。

### 面试官常问
1. **"为什么冒烟用 10 RPS 这么低？"**
   → "冒烟测的不是性能，是流水线连通性。先排除掉'lua 写错了'、'token 过期了'、'pool 文件路径不对'这类低级错误。"
2. **"如果冒烟时某个场景错误率 > 0 怎么办？"**
   → "立刻停。先看 nginx access log 看 status，再看 `cooking-app1` 的 `app.log`。绝不能带着错误进 Phase 2。"

### 重点观察
- [ ] 9 个场景错误率 = 0
- [ ] QPS 实际值 ≈ 10（差太多说明 wrk 配置不对）
- 我的实测：9 个场景错误率 = ___，最低 QPS = ___

### 对应架构
- 这一步隐含验证了 **三层架构 → handler → service → repository** 的链路完整性，包含点赞场景走 EventBus → RabbitMQ → LikeConsumer 的异步路径。

## §Phase 2  T2 基线负载（10 min × 3 档）

### 这一步在做什么
**50 / 100 / 200 RPS** 三档稳态负载，**每档跑 5 分钟**。这是 PRD KPI 验证档位 —— **G2 读写分离、G3 缓存命中率、G6 主从延迟** 在这一档采。

### 面试官常问
1. **"读写分离怎么实证'读真的走了 slave'？"**
   → "GORM DBResolver Random Policy 配的。压测前后各采一次 master/slave 的 `Com_select` 累积值，做差比，slave 占比目标 ≥ 80%。我实测 `___%`。"
2. **"缓存命中率怎么算？"**
   → "Redis 的 `keyspace_hits / (keyspace_hits + keyspace_misses)`，Feed 和 Detail 场景目标 ≥ 90%。第一档 50 RPS 跑完 Redis 已经预热满，第二档 100 RPS 起命中率才有意义。"
3. **"主从延迟用什么指标？"**
   → "`SHOW REPLICA STATUS\G` 里 `Seconds_Behind_Source`。压测全程目标 ≤ 2s。如果飙到 30s 说明 IO 线程吃不消，可能要拆库或加 slave。"

### 重点观察
- 终端 A：`docker stats` 看 app1/app2 CPU
- 终端 B：每分钟跑一次 `SHOW REPLICA STATUS\G | grep Seconds_Behind_Source`
- 终端 C：`docker exec cooking-redis-prod redis-cli INFO stats | grep keyspace_`
- 我的实测：
  - slave 读占比（200 档） = ___ %
  - Redis 命中率（200 档） = ___ %
  - 主从延迟峰值 = ___ s

### 对应架构
- **GORM + DBResolver Random Policy**（CLAUDE.md §3）：单一配置项实现读写分离
- **旁路缓存（Cache-Aside）**（CLAUDE.md §4.2）：service 先查 Redis 未命中再查 MySQL 再回写，**不是穿透式**
- **MySQL GTID 主从**：主从延迟数字直接是 IO/SQL 线程效率的实证

## §Phase 3  T3 阶梯压力（45 min，全场重头戏）

### 这一步在做什么
**100 → 3500 RPS 八档递增**，每档 60s。找出「错误率 > 1% 或 P99 > 1000ms」第一次出现的 RPS，**那就是 `MAX_SAFE_RPS`**。

### 面试官常问（这是最长的对话窗口，准备充分点）
1. **"你的 `MAX_SAFE_RPS` 是多少？瓶颈在哪？"**
   → "MAX_SAFE_RPS = `___`，瓶颈是 `___`（app CPU / MySQL 连接池 / Go heap / 出口带宽 / 客户端自身）。我是在 `docker stats` 和 Grafana MySQL 池利用率两块同时盯出来的。"
2. **"如果瓶颈是客户端自己（CPU 撞顶），你怎么办？"**
   → "立刻转多机分布式。本方案 §4.4 列了三个选项：k6 cloud / 自建 wrk 集群 / 单 IP 拉大 -c。单机的物理极限不能算作系统极限。"
3. **"为什么档位选 100/300/500/800/1200/1800/2500/3500 而不是等距？"**
   → "对数式递增。系统在低 RPS 区是线性的，高 RPS 区是指数级劣化，对数档位让我在'还没崩'和'刚崩'之间有足够分辨率。"
4. **"app CPU 撞顶你会怎么扩？"**
   → "横向加 app3/app4 实例，挂在同一个 nginx upstream 后面。垂直扩 GOMAXPROCS 没意义，2 核机器 = 2 个 P，已经吃满。"

### 重点观察
- **3 个终端常开**：
  - A：`tail -F /var/log/nginx/access.log` + awk 算 P99
  - B：`docker stats cooking-app{1,2}-prod cooking-mysql-prod cooking-redis-prod cooking-rabbitmq-prod`
  - C：`watch -n 2 docker exec cooking-rabbitmq-prod rabbitmqctl list_queues name messages consumers`
- **断点判定**（任一即停）：
  - nginx P99 > 1000ms
  - 错误率 > 1%
  - MySQL `Threads_connected` / `max_connections` > 90%
  - Go heap > 80% container limit
- 我的实测：
  - L4 800 RPS：P99 = ___ ms，错误率 = ___ %
  - L5 1200 RPS：P99 = ___ ms，错误率 = ___ %
  - 第一次命中断点的档 = L___，MAX_SAFE_RPS = ___
  - 瓶颈组件 = ___

### 对应架构
- **单体优先 + 双 Go 实例**（CLAUDE.md §4.1）：扩容路径清晰
- **MySQL 连接池**（GORM 默认 + pool.go 配置）：天花板在这一步现形
- **错误码段位分配**（CLAUDE.md §4.4）：观察 5xx 时 errcode 段位告诉你哪个模块在崩（503xxx = HTTP 5xx，41xxxx = 用户模块业务错）

## §Phase 4  T4 峰值（10 min）

### 这一步在做什么
基线 100 RPS 持续跑，**60s 后突然拉到 `MAX_SAFE_RPS × 1.5` 持续 30s**，然后回到 100 看多久 P99 回正常。测的是**突发流量耐受 + 自愈速度**。

### 面试官常问
1. **"为什么 ×1.5 而不是 ×3？"**
   → "×1.5 是真实社交平台的'热点事件'流量比例（如某个网红帖被推上首页）。×3 是 Phase 7 崩溃点的测试。两个测的不是一回事。"
2. **"系统怎么扛突发？你做了什么？"**
   → "三层缓冲：① nginx burst 给 20 个请求缓冲；② Redis 旁路缓存让 80%+ 请求不到 MySQL；③ 异步事件（点赞/PV）把写库压力推到 Consumer，**不在 spike 当下消化**。压力被时间维度摊平了。"
3. **"spike 结束后多久 P99 回正常？"**
   → "我实测 `___s` 内回到基线 P99 ± 20%。理想是 < 30s。"

### 重点观察
- spike 期间：错误率峰值 / P99 峰值
- spike 结束后：P99 回归基线用时
- LikeConsumer backlog 在 spike 期间峰值，spike 后 drain 时长
- 我的实测：
  - spike 峰值 P99 = ___ ms，错误率 = ___ %
  - 回归用时 = ___ s
  - Consumer backlog 峰值 = ___

### 对应架构
- **EventBus + RabbitMQ 持久化 + ACK + DLX**（CLAUDE.md §3）：异步通道扛 spike 的核心
- **dev/prod EventBus 双实现**（Channel vs RabbitMQ）：dev 不需要 broker 也能跑

## §Phase 5  T5 长稳（30 min）

### 这一步在做什么
**`MAX_SAFE_RPS × 0.6`** 持续 30 分钟，每 2 分钟采一次资源。检测**内存泄漏 / 句柄泄漏 / 慢累积**。

### 面试官常问
1. **"为什么 0.6 而不是 1.0？"**
   → "1.0 是单点峰值，0.6 才是日均稳态。长稳关心的不是极限 QPS，是 30 min 后内存有没有线性涨、句柄数有没有泄。"
2. **"Go 内存怎么看？"**
   → "看两层：① 容器层 RSS（`docker stats`），是实际占用；② 应用层 heap_inuse（`/metrics` 暴露的 Go runtime 指标）。两个都看是因为 RSS 包含未归还的 madvise 内存，heap 是 GC 视角的活对象。"
3. **"如果有线性增长，你怎么定位？"**
   → "pprof。Go 程序开 `import _ \"net/http/pprof\"`，curl `localhost:6060/debug/pprof/heap` 拿 profile，`go tool pprof -alloc_objects` 看泄漏对象。"

### 重点观察
- 每 2 分钟采样：
  - app1/app2 RSS、heap_inuse
  - MySQL `Threads_connected`、`Innodb_buffer_pool_pages_dirty`
  - Redis `used_memory_human`、`connected_clients`
  - RabbitMQ queue depth
- **通过条件**：30 min 末与 5 min 末差异 < 20% = 稳态
- 我的实测：
  - app1 RSS 5/30 min = ___ / ___ MB（增长 ___ %）
  - heap_inuse 5/30 min = ___ / ___ MB
  - 结论：稳态 / 单调增长 / 振荡

### 对应架构
- **GORM 连接池上限 / Redis MaxIdleConns**：长稳是验证这些上限设置合理性的唯一方式

## §Phase 6  T6 混合场景（10 min，k6 接管）

### 这一步在做什么
**按真实流量配比并发跑 8 个场景**（feed 40% / detail 25% / search 10% / like 12% / 其他 13%），总 RPS = `MAX_SAFE_RPS × 0.8`。**wrk 一次只能压一个端点，k6 才能模拟真实流量画像**。

### 面试官常问
1. **"为什么用 k6 不用 wrk？"**
   → "wrk 每个进程压一个端点，多端点要起多进程然后合并结果，繁琐。k6 的 `constant-arrival-rate` executor 一个配置就能同时跑 8 个场景，每个独立配 rate，互不抢连接。"
2. **"混合压测的意义是什么？"**
   → "单端点压测会高估容量。真实流量里 feed 占 40%、search 走全文索引慢、like 走异步，这些会争 MySQL 连接池和 nginx worker。混合场景才知道**端点间的相互拖累**。"
3. **"流量配比是怎么定的？"**
   → "现在是产品经验估算（feed 是首屏所以 40%，search 是低频但慢所以单独看），上线后会用 nginx access log 做真实采样替换。"

### 重点观察
- k6 的 `thresholds` 输出：哪些场景红了
- 每场景 P99（k6 summary 自动给）
- MySQL 慢查询日志：哪个端点的 query 进了慢查询
- 我的实测：
  - 最红的场景 = ___（P99 = ___ ms）
  - 端点间是否相互拖累 = 是 / 否
  - 慢查询数量 = ___

### 对应架构
- **nginx upstream + 双 Go 实例**：混合场景下两实例 CPU 是否均衡
- **MySQL 索引**（idx_scene_visible_created / idx_author_visible_created）：混合负载下索引是否被正确选中

## §Phase 7  T7 崩溃点（选做，30 min）

### 这一步在做什么
**`MAX_SAFE_RPS × 3` 跑 5 分钟**，把系统压崩。看**谁先死、能不能自愈**。

### 面试官常问
1. **"你测崩溃点的目的是什么？"**
   → "知道**最先崩的是谁**比知道极限 QPS 更重要。如果是 app OOM，说明 GOMEMLIMIT 没设；如果是 MySQL 连接打满，说明连接池上限太宽；如果是 RabbitMQ 队列爆，说明 Consumer 数量不够。最先崩的就是 v2 第一个要加固的点。"
2. **"崩了之后能自愈吗？"**
   → "停掉 wrk 后我观察 5 分钟。我实测 `___` 自愈（健康检查 + Docker restart policy 拉起），`___` 需要人工 `docker restart` 重启。"
3. **"为什么标'选做'？"**
   → "崩溃点测试本身有破坏性 —— 可能产生 OOM 后未提交事务的脏数据，可能让 Consumer 出现死信。所以这一步在 Phase 6 之后单独的窗口跑，**跑前先备份 MySQL**。"

### 重点观察
- 哪个容器最先 unhealthy（看 `docker ps`）
- 是否触发 OOM kill（`dmesg | grep oom`）
- app log 最后 200 行（panic / runtime error）
- 自愈与否
- 我的实测：
  - 最先死的组件 = ___
  - OOM 触发 = 是 / 否
  - 自愈 = 是 / 否，耗时 ___ s

### 对应架构
- **Docker restart policy**（CLAUDE.md 隐含约定）：自愈的最后兜底
- **健康检查接口** `/health` `/readiness`：让 nginx 知道何时该把流量从死实例摘掉

## §8 总结的金句模板（结尾陈词）

跑完 7 个 Phase 之后，面试官大概率问 **"复盘一下整个压测最大的 takeaway 是什么"**。准备 3 句不同维度的：

### 技术维度
> "压测最大的收获是把'架构上写的话'变成了'测量到的数字'。比如读写分离 KPI 写的是 ≥ 80%，但实测要在 Phase 2 100 RPS 以上才稳定打到 90%+。低流量时 GORM Random Policy 反而会因为查询少而方差大，**配置和实测之间永远有一层验证缺口**。"

### 方法论维度
> "我学到的是**冒烟 → 基线 → 阶梯 → 峰值 → 长稳 → 混合 → 崩溃**这个顺序背后的逻辑：每一步都依赖上一步的产物。没有 Phase 3 的 MAX_SAFE_RPS，Phase 4 的 ×1.5 和 Phase 5 的 ×0.6 都无意义；没有 Phase 6 的混合，Phase 7 的崩溃点只能测出一个端点的极限。"

### 工程文化维度
> "压测让我对**安全网**的理解变深了。nginx 限流、SMS per-IP、MySQL max_connections 这些东西平时看着碍事，但 Phase 0 把它们关掉之后我才真正意识到——它们是从开发期到生产期的**心理保险**。压完必须按反序回滚，这是工程素养。"

### 教训维度（如被问到 "踩了什么坑"）
> "最大的坑是 **SMS per-IP 限流没关就跑 seed**，结果 100 用户只成功注册了 2 个。后来在 seed 脚本前面加了一个 probe，发 2 条 send-code 间隔 1s，第 2 条不成功就提前 fail —— 把这个隐式前置变成显式断言。"

### 改进维度（如被问到 "再做一遍会怎么改"）
> 三件事，按优先级：
> 1. **上 Loki**（遗留债务 #1）：压测时日志归集太痛苦，要 `docker exec` 进每个容器 grep。
> 2. **改 nginx 挂载方式**（遗留债务 #3）：从单文件 bind-mount 改成目录挂载，免去每次改配置都得 `--force-recreate`。
> 3. **多机客户端**（遗留债务 #4）：单机自压撞到客户端瓶颈，测不出 backend 极限。

## §9 反复练习的 5 个问题清单

面试前对着镜子能流畅讲完这 5 题，基本就能 hold 住任何压测主题的问答：

- [ ] 你的 MAX_SAFE_RPS 是多少？怎么得出来的？
- [ ] 你怎么实证读写分离真的生效了？
- [ ] 异步事件（点赞）压测时怎么观察 drain 完成？
- [ ] 你测崩溃点是为了什么？最先崩的是什么？为什么是它？
- [ ] 如果让你再做一遍，会改三件什么事？

## §2 回滚执行顺序

### Step A — 恢复 nginx + app 配置（本地）

```bash
# 在本地 cooking-platform 仓库
cd ~/Desktop/cooking-platform

# 找到压测变更的 commit
git log --oneline --grep='全面压测临时变更' | head -5

# 假设变更只在一个 commit 里（commit_hash 替换成实际）：
git revert <commit_hash>
# 或者手动改回这些文件：
#   - configs/config.prod.yaml 第 73 行 mock → aliyun
#   - configs/config.prod.yaml 第 125 行 100 → 5
#   - configs/config.prod.yaml 第 126 行 100000 → 10
#   - deploy/nginx/nginx.conf 行 49 100000r/s → 30r/s
#   - deploy/nginx/nginx.conf 行 50 100000r/s → 1r/m
#   - deploy/nginx/nginx.conf location / 的 limit_conn conn_zone 10000 → 20
#   - scripts/stress_test.sh §2 删掉 setup post force-visible 的 if 块（grep [全面压测临时变更]）

git push origin main
```

### Step B — 服务器拉新配置 + 重启

```bash
ssh root@47.238.29.251
cd /opt/cooking-platform

git pull origin main

# force-recreate 三个服务，限流恢复 + sms provider 切回 aliyun
docker compose -f docker-compose.prod.yml --env-file .env.prod \
  up -d --force-recreate nginx app1 app2

# 验证 nginx 限流已恢复（必须看到 30r/s 和 1r/m）
docker exec cooking-nginx-prod nginx -T 2>/dev/null | grep limit_req_zone

# 验证 sms.provider=aliyun（看 app 启动日志里的 provider 字样）
docker exec cooking-app1-prod head -50 /var/log/cooking/app.log | grep -i sms
```

### Step C — 恢复 MySQL max_connections（如调过）

```bash
RPW=$(grep '^MYSQL_ROOT_PASSWORD=' /opt/cooking-platform/.env.prod | cut -d= -f2-)

# 临时调整动态值（重启容器后自动回归 my.cnf 里的默认）
docker exec cooking-mysql-prod \
  mysql -uroot -p"$RPW" -e "SET GLOBAL max_connections=151;"

# 验证
docker exec cooking-mysql-prod \
  mysql -uroot -p"$RPW" -e "SHOW VARIABLES LIKE 'max_connections';"
```

### Step D — 清掉 prod DB 测试数据

```bash
# 走 seed 脚本自带的 cleanup（最干净）
ssh root@47.238.29.251
cd /opt/cooking-platform
source .env.prod
bash scripts/stress/seed_stress_data.sh --cleanup

# 验证：should be 0
RPW=$(grep '^MYSQL_ROOT_PASSWORD=' .env.prod | cut -d= -f2-)
docker exec cooking-mysql-prod mysql -uroot -p"$RPW" -N -B \
  -e "SELECT COUNT(*) FROM cooking_platform.users WHERE phone LIKE '139%';"
docker exec cooking-mysql-prod mysql -uroot -p"$RPW" -N -B \
  -e "SELECT COUNT(*) FROM cooking_platform.posts WHERE title LIKE 'stress %';"
```

### Step E — 清 Redis 残留

```bash
RPW=$(grep '^REDIS_PASSWORD=' /opt/cooking-platform/.env.prod | cut -d= -f2-)

# SMS 限流计数器
docker exec cooking-redis-prod redis-cli -a "$RPW" --no-auth-warning \
  --scan --pattern 'sms:ip:*' | \
  xargs -L 100 docker exec -i cooking-redis-prod redis-cli -a "$RPW" --no-auth-warning DEL

docker exec cooking-redis-prod redis-cli -a "$RPW" --no-auth-warning \
  --scan --pattern 'sms:limit:*' | \
  xargs -L 100 docker exec -i cooking-redis-prod redis-cli -a "$RPW" --no-auth-warning DEL

# 验证 0 个
docker exec cooking-redis-prod redis-cli -a "$RPW" --no-auth-warning \
  --scan --pattern 'sms:*' | wc -l
```

### Step F — 清客户端临时文件

```bash

rm -f /tmp/stress_pool_users.tsv /tmp/stress_pool_posts.tsv /tmp/stress_pool.json
```

### Step G — 最终验证：跑 verify-step20

## §3 紧急回滚（Phase 7 崩溃 / 服务不可用）

如果 Phase 7 把服务打挂、容器不健康，直接跳到这里执行全套回滚：

```bash
ssh root@47.238.29.251
cd /opt/cooking-platform

# 1. 立刻 kill 所有压测进程（在压测客户端机器）
pkill -9 wrk
pkill -9 k6

# 2. 重启所有 prod 服务
docker compose -f docker-compose.prod.yml --env-file .env.prod restart

# 3. 等 30s healthy 后看状态
sleep 30
docker compose -f docker-compose.prod.yml --env-file .env.prod ps

# 4. 如果 MySQL 不健康（OOM kill 等）
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d mysql mysql-slave
sleep 15
RPW=$(grep '^MYSQL_ROOT_PASSWORD=' .env.prod | cut -d= -f2-)
docker exec cooking-mysql-slave-prod mysql -uroot -p"$RPW" -e "SHOW REPLICA STATUS\G" | grep Running

# 5. 然后再走 §2 Step A 起的常规回滚
```

# cooking-platform 全面压测方案

> **版本**：v1.0
> **创建日期**：2026-05-30
> **执行阶段**：开发期（dev / 预生产）压测，**允许临时绕开 nginx 限流与生产安全网**
> **目标环境**：单机自压（生产服务器 47.238.29.251） + 多机分布式压测（k6 cloud / 外部 wrk 节点，可选）
> **责任人**：小离

---

## §0 文档目的与适用范围

本方案是 cooking-platform 在「**开发期 / 预生产阶段**」执行的全面压测计划，目标是**探明系统真实极限**而非验收。与 Step 20 公网验收的差异：

| 维度 | Step 20（生产验收） | 本方案（开发期压测） |
| --- | --- | --- |
| 目标 | 证明系统在 PRD KPI 下稳定 | 找出真实瓶颈与崩溃点 |
| 流量上限 | 80 RPS（留 nginx 限流 20% margin） | 不设上限（临时关闭限流） |
| 安全网 | 全部保留 | 允许临时关闭限流 / 调大连接数 / 关闭 SMS per-IP |
| 接受标准 | KPI 达标即停 | 跑到错误率 > 1% 或 P99 > 1s 为止 |
| 频率 | 每个 release 跑一次 | 集中一周内压完，留作基线 |

> ⚠️ **重要前提**：本方案默认在「**没有真实用户**」的窗口执行。如果已有生产流量，必须先选择维护窗或在独立预生产环境跑。本方案给出的「关闭限流」步骤可逆，每条都配套了回滚命令。

## §1 压测目标（必须量化）

| 目标 | 指标 | 通过标准 |
| --- | --- | --- |
| G1. 找出单机集群真实极限 RPS | wrk2 -R 逐档递增 | 错误率 ≥ 1% 或 P99 ≥ 1000ms 的那一档 |
| G2. 验证读写分离 KPI | slave 读流量占比 | ≥ 85%（PRD 目标 80%） |
| G3. 验证旁路缓存命中率 | Redis hit/(hit+miss) | feed/detail 场景 ≥ 90% |
| G4. 验证异步通道吞吐 | LikeConsumer drain 速率 | 压测结束后 30s 内 Redis 累积值清零 |
| G5. 找出 MQ 队列堆积阈值 | RabbitMQ queue depth | 持续上涨说明 consumer 是瓶颈 |
| G6. 验证主从复制延迟 | `Seconds_Behind_Source` | 压测期间 ≤ 2s |
| G7. 找出 MySQL 连接池天花板 | `Threads_connected` | 接近 `max_connections` 时观察拒绝行为 |
| G8. 长稳运行内存稳定性 | Go heap / RSS | 30 分钟 soak 无单调增长 |

## §2 压测类型矩阵

七种压测类型，按风险递增排序：

| # | 类型 | 时长 | 目的 | 必做 |
| --- | --- | --- | --- | --- |
| T1 | **冒烟（Smoke）** | 60s | 流水线打通，确认链路无错 | ✅ |
| T2 | **基线负载（Load）** | 5 min | 在「正常预期峰值」下测稳态指标 | ✅ |
| T3 | **压力（Stress）** | 阶梯递增 30s × 8 档 | 找真实极限 RPS | ✅ |
| T4 | **峰值（Spike）** | 10s 突进 + 30s 持续 + 观察恢复 | 测突发流量耐受 | ✅ |
| T5 | **长稳（Soak）** | 30 min | 测内存泄漏 / 句柄泄漏 / 慢累积型问题 | ✅ |
| T6 | **混合场景（Mixed）** | 5 min | 按真实流量配比并发跑多端点 | ✅ |
| T7 | **崩溃点（Breakpoint）** | 跑到崩 | 摸最底线，记录系统如何降级 / 死 | 选做（最后跑） |

## §4 测试环境准备（关键章节）

### 4.1 临时关闭 nginx 限流（本方案前提）

在 `deploy/nginx/nginx.conf` 顶部创建一个开发期开关：

```nginx
# 临时关闭限流（仅开发期压测使用）
# 回滚：把 rate 改回 30r/s + 1r/m
limit_req_zone $binary_remote_addr zone=api_zone:10m rate=100000r/s;
limit_req_zone $binary_remote_addr zone=sms_zone:10m rate=100000r/s;
```

执行：

```bash
# 1. 备份现网配置
cp deploy/nginx/nginx.conf deploy/nginx/nginx.conf.prod.bak

# 2. 改 rate 值（手动 edit 或 sed）
sed -i.tmp 's/rate=30r\/s/rate=100000r\/s/' deploy/nginx/nginx.conf
sed -i.tmp 's/rate=1r\/m/rate=100000r\/s/' deploy/nginx/nginx.conf
rm deploy/nginx/nginx.conf.tmp

# 3. force-recreate nginx（CLAUDE.md §9.2：bind-mount 单文件，reload 不够）
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d --force-recreate nginx

# 4. 验证
docker exec cooking-nginx-prod nginx -T 2>/dev/null | grep limit_req_zone

# === 回滚 ===
cp deploy/nginx/nginx.conf.prod.bak deploy/nginx/nginx.conf
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d --force-recreate nginx
```

### 4.2 临时关闭应用层 SMS per-IP 限流

`pkg/sms/` 或 `internal/service/auth_service.go` 中找到 per-IP 限流，配置项关掉。具体位置压测前 grep 一次：

```bash
grep -rn 'sms:ip:' internal/ pkg/sms/
```

若是配置项控制（如 `auth.sms_per_ip_per_min: 0` 表示禁用），改 `config.prod.yaml` 后 force-recreate app1+app2。
若是硬编码，跳过 S5/S6 的真实并发，改用 mock token。

### 4.3 临时调高资源限制

```bash
# 1. 调高 nginx worker_connections
# nginx.conf: worker_connections 1024 → 16384

# 2. 调高 MySQL max_connections（默认 151 太低）
docker exec cooking-mysql-prod mysql -uroot -p"$MYSQL_ROOT_PW" -e \
  "SET GLOBAL max_connections=2000;"

# 3. 调高 OS file descriptor 上限（在压测客户端机器）
ulimit -n 65536

# 4. 调高 Go app GOMAXPROCS（默认 = CPU 数，已是 2，不动）
```

### 4.4 客户端机器准备（wrk2 / k6）

**单机自压（默认）**：服务器本机跑 wrk2，受本机 CPU + 出口带宽限制。

**多机分布式（探真实极限）**：
- 选项 A：k6 cloud（付费，最简单，跨地理分布）
- 选项 B：自建 3-5 台云主机跑同一 wrk 脚本，结果手动合并
- 选项 C：单 IP 模拟多客户端时，加 `--latency` 同时改 `WRK_CONNS` 拉高，但本质受限

```bash
# 装原版 wrk 4.2.0+（支持 -R rate-limit）
rm -rf /tmp/wrk-src
git clone https://github.com/wg/wrk.git /tmp/wrk-src
cd /tmp/wrk-src && make && cp wrk /usr/local/bin/wrk
hash -r
wrk --version 2>&1 | head -1  # 必须 ≥ 4.2.0
```

### 4.5 监控就位检查清单

压测开始前必须确认下面 4 块 Grafana 看板都活：

- [ ] HTTP Request Latency（P50/P95/P99 by endpoint）
- [ ] Consumer Backlog（Like / PV / Audit）
- [ ] MySQL Connection Pool & Slow Query
- [ ] Redis Command Latency & Memory

SSH 隧道方式见 CLAUDE.md §7.4。

## §5 测试数据准备

**用户池**：注册 100 个用户作为「点赞 / 关注 / 发帖」的真实身份池，避免压测中所有请求都来自同一 token（会被去重 / 受 service 层幂等机制干扰）。

**帖子池**：发布 50 篇帖子，覆盖 8 个 scene_tag，每篇含 5 个热点关键词中的至少 1 个，让 search 场景能命中。

```bash
# scripts/stress/seed_stress_data.sh（待写）
# 输入：USER_COUNT=100  POST_COUNT=50
# 输出：/tmp/stress_pool.json
#   {
#     "users": [{ "phone": "...", "token": "...", "id": ... }, ...],
#     "posts": [{ "id": ..., "title": "..." }, ...]
#   }
```

wrk lua 脚本启动时读 `/tmp/stress_pool.json`，每个虚拟用户连接随机选一个 token + 一个 post_id，避免热点偏置。

## §6 分阶段执行步骤

### Phase 0 ─ 环境准备（30 min）

- [ ] §4.1 关闭 nginx 限流
- [ ] §4.2 关闭 SMS per-IP（若需要压注册流程）
- [ ] §4.3 调高 max_connections / ulimit
- [ ] §4.4 装 wrk2 4.2.0+
- [ ] §4.5 Grafana 隧道开启，4 块看板都活
- [ ] §5 seed 数据池 100 用户 + 50 帖

### Phase 1 ─ T1 冒烟（5 min）

```bash
# 单线程低 RPS 跑 9 个场景每个 60s，确认链路无错
WRK_RATE=10 WRK_DURATION=60s WRK_THREADS=1 WRK_CONNS=10 \
  bash scripts/stress_test.sh
```

**通过条件**：每个场景 错误率 = 0，QPS ≈ 10。

**失败处理**：错误率 > 0 → 回查 app 日志 → 修问题 → 重跑。

### Phase 2 ─ T2 基线负载（10 min × 3 轮）

3 个 RPS 档位：50 / 100 / 200。每档跑 5 分钟 + 1 分钟冷却。

```bash
for rate in 50 100 200; do
  WRK_RATE=$rate WRK_DURATION=300s WRK_CONNS=$((rate*2)) \
    bash scripts/stress_test.sh
  sleep 60
done
```

**通过条件**：P99 < 200ms，错误率 < 0.1%。

**采集数据**：
- 每场景 QPS / P50 / P95 / P99
- MySQL slave 读流量占比（验证 KPI G2）
- Redis 命中率（验证 KPI G3）

### Phase 3 ─ T3 阶梯压力（45 min）

逐档递增找真实极限：

| 档 | RPS | 时长 | 期望 |
| --- | --- | --- | --- |
| L1 | 100 | 60s | 错误率 0 |
| L2 | 300 | 60s | 错误率 0 |
| L3 | 500 | 60s | 错误率 < 0.1% |
| L4 | 800 | 60s | 错误率 < 0.5% |
| L5 | 1200 | 60s | 错误率 < 1% |
| L6 | 1800 | 60s | 错误率 < 2%（开始拉警报） |
| L7 | 2500 | 60s | 错误率 < 5% |
| L8 | 3500 | 60s | 跑到崩为止 |

```bash
for rate in 100 300 500 800 1200 1800 2500 3500; do
  echo "===== STRESS rate=$rate ====="
  WRK_RATE=$rate WRK_DURATION=60s WRK_CONNS=$((rate*2)) \
    bash scripts/stress_test.sh > /tmp/stress_L_$rate.log 2>&1
  sleep 30
done
```

**记录的「断点」**：第一次出现下列之一就是上限：
- P99 > 1000ms
- 错误率 > 1%
- MySQL `Threads_connected` > 90% max_connections
- Go app heap > 80% 容器 mem limit

把断点 RPS 标记为 **`MAX_SAFE_RPS`** 写入结果文档。

### Phase 4 ─ T4 峰值（10 min）

模拟突发：基线 100 RPS → 突然拉到 `MAX_SAFE_RPS × 1.5` 持续 30s → 回到 100 RPS 观察 5 min 恢复。

```bash
# 终端 1：基线
WRK_RATE=100 WRK_DURATION=600s \
  wrk -t2 -c20 -d600s -R100 https://mellowck.com/health &

# 终端 2：60s 后注入 spike
sleep 60
SPIKE=$((MAX_SAFE_RPS * 3 / 2))
WRK_RATE=$SPIKE WRK_DURATION=30s \
  wrk -t8 -c200 -d30s -R$SPIKE https://mellowck.com/api/v1/feed
```

**观察点**：
- 峰值期间错误率是否暴涨？
- spike 结束后多久 P99 回归正常？（理想 < 30s）
- 是否触发熔断 / 退化逻辑？

### Phase 5 ─ T5 长稳（30 min）

`MAX_SAFE_RPS × 0.6` 持续 30 分钟，观察资源单调性。

```bash
SOAK_RATE=$((MAX_SAFE_RPS * 6 / 10))
WRK_RATE=$SOAK_RATE WRK_DURATION=1800s \
  bash scripts/stress_test.sh
```

**采样**：每 2 分钟记录一次
- Go 容器 RSS / heap
- MySQL `Threads_connected`、`Innodb_buffer_pool_pages_dirty`
- Redis `used_memory_human`、`connected_clients`
- RabbitMQ 队列长度

**通过条件**：所有指标在 30 分钟末与 5 分钟末的差异 < 20%（即「稳态」而非单调增）。

### Phase 6 ─ T6 混合场景（10 min）

按真实流量配比并发跑（用 k6 或多个 wrk 进程）。RPS = `MAX_SAFE_RPS × 0.8`。

```
S2 feed       : 40% × 0.8 × MAX = ?
S3 detail     : 25% × ...
S4 search     : 10% × ...
S5 like       : 12% × ...
其他          : 13% × ...
```

推荐用 k6（脚本 `scripts/stress/k6_mixed.js` 待写）：

```javascript
import { check, group } from 'k6'
import http from 'k6/http'

export const options = {
  scenarios: {
    feed:   { executor: 'constant-arrival-rate', rate: 320, duration: '10m', preAllocatedVUs: 200, ... },
    detail: { executor: 'constant-arrival-rate', rate: 200, duration: '10m', ... },
    // ...
  },
}
```

### Phase 7 ─ T7 崩溃点（选做，30 min）

**注意**：跑这步前先备份 MySQL 数据，因为可能产生「破坏性」状态（OOM / 数据库连接打满）。

直接拉到 `MAX_SAFE_RPS × 3` 跑 5 分钟。期望：

- 看到容器 OOM kill 或 panic 或拒绝服务
- 观察服务是否能在 wrk 停止后**自愈**
- 记录哪个组件最先死（app / mysql / redis / rabbitmq）

跑完立刻：
- [ ] 看 4 块 Grafana 看板，截图归档
- [ ] 看 app 日志最后 200 行
- [ ] 重启所有 unhealthy 容器
- [ ] 跑一次 Phase 1 冒烟确认自愈

## §7 监控指标与告警阈值（压测期间）

每个 Phase 都要同时盯下面这些指标。建议开 3 个终端：

### 终端 A：实时延迟
```bash
# 每秒拉一次 nginx access log P99
docker exec cooking-nginx-prod sh -c 'tail -F /var/log/nginx/access.log' \
  | awk '{print $NF}' | ...
```

### 终端 B：实时容器资源
```bash
docker stats cooking-app1-prod cooking-app2-prod cooking-mysql-prod cooking-redis-prod cooking-rabbitmq-prod
```

### 终端 C：实时 MQ 队列长度
```bash
watch -n 2 'docker exec cooking-rabbitmq-prod rabbitmqctl list_queues name messages consumers'
```

### 关键阈值表

| 指标 | 警戒值 | 终止压测条件 |
| --- | --- | --- |
| nginx P99 | > 500ms | > 2000ms |
| 错误率 | > 1% | > 5% |
| MySQL Threads_connected | > 80% max | > 95% max |
| MySQL Seconds_Behind_Source | > 5s | > 30s |
| Redis used_memory | > 70% maxmemory | > 90% maxmemory |
| RabbitMQ queue depth | 持续上涨 5 min | > 100k |
| Go heap | > 70% limit | OOM kill |
| app1/app2 CPU | > 85% | > 95% 持续 60s |

任一指标命中「终止」列就立即 `kill wrk`，记录现场。

## §9 回滚与收尾

压测全部跑完后，**必须**按反序回滚所有临时变更：

```bash
# 1. 恢复 nginx 限流
cp deploy/nginx/nginx.conf.prod.bak deploy/nginx/nginx.conf
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d --force-recreate nginx

# 2. 恢复 SMS per-IP 限流
# 改 config.prod.yaml 回原值 → force-recreate app1+app2

# 3. 恢复 MySQL max_connections（如果 my.cnf 没改，重启 mysql 容器自动回归）
docker exec cooking-mysql-prod mysql -uroot -p"$MYSQL_ROOT_PW" -e \
  "SET GLOBAL max_connections=151;"

# 4. 清理 seed 数据
docker exec cooking-mysql-prod mysql -uroot -p"$MYSQL_ROOT_PW" cooking_platform -e \
  "DELETE FROM posts WHERE title LIKE 'stress %';
   DELETE FROM users WHERE phone LIKE '138%' AND created_at > '2026-05-30';"

# 5. 清理 Redis 残留
docker exec cooking-redis-prod redis-cli -a "$REDIS_PW" --no-auth-warning \
  --scan --pattern 'sms:ip:*' | xargs -L 100 docker exec -i cooking-redis-prod \
  redis-cli -a "$REDIS_PW" --no-auth-warning DEL

# 6. 跑一次 verify-step20 确认生产恢复
make verify-step20

# 7. 提交压测结果报告
git add docs/stress/
git commit -m "docs(stress): 全面压测结果归档（Phase 1~7）"
```

## §11 风险与已知约束

| 风险 | 应对 |
| --- | --- |

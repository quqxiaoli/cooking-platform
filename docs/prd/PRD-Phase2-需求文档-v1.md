## 1. 文档说明

### 1.1 优先级定义

|优先级|含义|上线时机|
|---|---|---|
|P0|上线必须，缺失则产品不可用|Day 1（MVP 上线日）|
|P1|首月内补齐，影响核心体验|Month 1|
|P2|三个月内完成，提升产品深度|Month 3|
|永久排除|产品决策，不在任何版本中实现|永不|

### 1.2 验收标准（AC）格式说明

每条 AC 格式为：`AC-N [条件] → [预期结果]`，所有 AC 均需通过方可视为功能完成。

### 1.3 产品气质一致性检查（每次需求评审前必读）

所有功能决策须通过以下六个词的检验：

```
松弛    → 这个设计让用户感到轻松吗？
真实    → 这个设计鼓励真实内容而非表演吗？
有温度  → 这个设计让人感到被看见、被理解吗？
可复现  → 这个设计帮助用户真正做出那道菜吗？
有场景感 → 这个设计强化了场景认同感吗？
不焦虑  → 这个设计不会带来「不够好」的焦虑感吗？
```

## 3. 模块一：用户系统

### F-U01 · 用户注册（P0）

**用户故事**

> 作为一个第一次来到平台的年轻人，我想用手机号快速注册一个账号，不希望填写太多信息，注册完成后能立刻开始浏览内容。

**功能描述**

- 注册方式：手机号 + 短信验证码（MVP 唯一方式）
- 注册成功后自动生成默认昵称（`厨房探索者_XXXX`，XXXX 为随机 4 位数字）和默认头像（系统内置 6 款随机分配）
- 注册即视为同意《用户协议》和《隐私政策》（需在界面上显示链接）

**验收标准（AC）**

```
AC-1  输入有效手机号，60 秒内收到 6 位纯数字验证码短信
AC-2  验证码输入正确，1 秒内完成注册，自动登录并跳转首页
AC-3  同一手机号 / 同一 IP 每天最多发送 5 次验证码，超限后返回「今日发送次数已达上限，请明天再试」
AC-4  注册已有手机号，返回「该号码已注册」并提供跳转登录入口
AC-5  注册成功后颁发 JWT AccessToken（有效期 7 天）和 RefreshToken（有效期 30 天）
AC-6  验证码 10 分钟内有效，超时需重新获取
```

**边界说明**

- MVP 不做微信 / Apple / Google 第三方登录（二期补充）
- 不做邮箱注册
- 不做密码设置（纯验证码登录体系）

---

### F-U02 · 用户登录（P0）

**用户故事**

> 作为已注册用户，我想随时用手机号登录，也想在不同设备上保持登录状态，不用每次都重新输入验证码。

**功能描述**

- 登录方式：手机号 + 短信验证码（与注册共用同一套验证码服务）
- Session 管理：JWT（无状态验证）+ Redis Session（强制下线 / 封号能力）双轨并行
- 同一账号最多同时在 3 台设备登录，超出时踢出最早登录的设备
- Token 过期前 24 小时静默刷新，用户无感知

**验收标准（AC）**

```
AC-1  有效验证码登录，1 秒内返回 JWT Token 和用户基本信息（id / 昵称 / 头像）
AC-2  Token 过期前 24 小时内，客户端调用 /auth/refresh 自动换取新 Token
AC-3  Redis Session 中标记封号状态，该用户下次任意请求立即返回 401 并提示封号原因
AC-4  超过 3 台设备时，第 4 台登录成功，最早登录的设备下次请求返回 401「已在其他设备登录」
AC-5  退出登录时，服务端立即在 Redis 中使对应 Session 失效，Token 加入黑名单
```

**技术决策说明**

> 选择「短信验证码」而非「账号密码」：密码管理是认知负担，对 25 岁主力用户群体，验证码登录体验更流畅，也消除「忘记密码」的支持成本。后期可视需求在「设置」中补充密码功能。

### F-U03 · 个人主页（P0）

**用户故事**

> 作为创作者，我希望有一个属于自己的主页，展示我的所有内容，让关注我的人能看到我的烹饪记录。 作为消费者，我希望点进作者主页时能看到 TA 的所有内容和基本信息。

**功能描述**

个人主页展示内容：

|内容|说明|
|---|---|
|头像 + 昵称|顶部醒目展示|
|个人签名|最多 50 字，可选|
|统计数据|发帖数 / 获赞总数 / 关注数 / 粉丝数|
|内容列表|该用户发布的所有内容，按发布时间倒序，瀑布流展示|
|操作按钮|查看他人主页时显示「关注 / 已关注」按钮；查看自己主页时显示「编辑资料」|

**验收标准（AC）**

```
AC-1  个人主页 URL 格式：/u/{user_id}，支持直接分享访问
AC-2  未登录用户可查看他人主页和内容列表，关注按钮点击后跳转登录引导
AC-3  内容列表分页加载，每页 20 条，下拉触底自动加载下一页
AC-4  内容列表为空时，显示「还没有发过内容，去记录第一道菜吧」
AC-5  自己的主页右上角显示「编辑资料」入口
```

### F-U04 · 编辑个人资料（P0）

**功能描述**

可编辑字段：

|字段|规则|
|---|---|
|头像|支持 JPG / PNG / WebP，最大 5MB，服务端裁剪为 400×400px 正方形存储|
|昵称|2 ~ 20 字符，不允许纯空格，过敏感词过滤|
|个人签名|最多 50 字，可留空|

**验收标准（AC）**

```
AC-1  头像上传成功后，全站该用户头像在 5 秒内全部更新（CDN 缓存刷新）
AC-2  昵称包含敏感词时，明确提示「昵称包含不允许使用的词语」，不保存
AC-3  昵称与他人完全相同时，允许保存（不强制唯一昵称）
AC-4  所有字段更新后即时生效，无需重新登录
```

## 4. 模块二：内容发布

### F-C01 · 发布烹饪内容（P0）

**用户故事**

> 作为创作者，我想发布一篇记录今天烹饪过程的图文内容，可以上传每个步骤的图片，写上简单说明，选一个场景标签，然后发布出去。即使只有 1 张图和几行字也应该能发出来。

**内容字段定义**

|字段|必填|上限 / 规则|
|---|---|---|
|标题|是|2 ~ 50 字|
|场景标签|是|从 8 个预设场景中选 1 个，详见第 9 节|
|步骤列表|是|至少 1 步，最多 30 步；每步含文字说明（必填，最多 500 字）+ 图片（可选，每步最多 3 张）|
|成品图|否|最多 3 张；在 Feed 流卡片中作为封面展示；无成品图则取第一步第一张图|
|食材清单|否|每行格式：「食材名 + 用量」，最多 30 行|
|口味 / 食材二级标签|否|多选，从预设列表选择（见第 9 节）|
|烹饪时长|否|单选：15 分钟内 / 30 分钟内 / 1 小时内 / 1 小时以上|

**图片处理规则**

- 支持格式：JPG / PNG / WebP / HEIC
- 单张最大：20MB（上传前客户端提示，超限不允许上传）
- 服务端处理：自动压缩至宽度 1080px（等比缩放），转为 WebP 格式存储，保留原图备用
- 存储路径：阿里云 OSS / 腾讯云 COS（按部署环境决定）

**发布流程**

```
用户填写内容
  → 客户端校验（标题非空 / 至少 1 步 / 已选场景标签）
  → 图片上传（并发上传，显示进度条）
  → 提交发布请求
  → 服务端敏感词过滤（标题 + 步骤文字）
  → 写入 MySQL
  → 清除首页 Feed 缓存（Redis）
  → 跳转内容详情页
```

**验收标准（AC）**

```
AC-1  最简发布：仅填标题 + 选场景标签 + 写 1 步文字，即可发布成功
AC-2  图片上传支持多图同时选择，进度条实时显示每张图上传状态
AC-3  发布过程中网络中断，草稿自动保存至本地（LocalStorage / 客户端缓存），
      用户重新打开发布页提示「继续上次未完成的内容」
AC-4  发布成功后 1 秒内跳转至该内容详情页
AC-5  每用户每天最多发布 20 篇，超出后提示「今日发布次数已达上限」
AC-6  标题或步骤文字含敏感词，返回「内容包含不允许的词语，请修改后重新发布」并指出具体字段
AC-7  图片上传失败时，单张图片支持单独重试，不影响其他已上传图片
```

**边界说明**

- MVP 不支持视频上传，仅图文
- 不支持定时发布（二期）
- 发布成功后可无限次编辑（F-C02）
- 草稿功能仅本地缓存，不同步至服务端（二期考虑云端草稿）

### F-C02 · 编辑 / 删除内容（P1）

**用户故事**

> 发布后发现步骤写错了，或者想补一张成品图，我希望能直接修改而不用删掉重发。

**功能描述**

**编辑：**

- 作者可编辑所有字段（标题 / 步骤 / 图片 / 标签 / 食材 / 时长）
- 编辑后「最后编辑时间」字段更新，发布时间不变，Feed 流排序位置不变
- 已上传图片支持单独删除和补充上传

**删除：**

- 软删除：数据标记 `deleted_at`，前台立即不可见，后台保留 30 天后物理删除
- 删除后该内容的点赞数、浏览量数据同步标记失效

**验收标准（AC）**

```
AC-1  只有内容作者本人可以编辑 / 删除自己的内容，非作者操作返回 403
AC-2  删除后内容立即从所有 Feed 流、搜索结果、作者主页中消失（< 5 秒）
AC-3  编辑后内容在详情页立即更新，Feed 流缓存在下次刷新时更新（最长 5 分钟）
AC-4  删除操作需二次确认弹窗（「确认删除？删除后不可恢复」）
```

## 5. 模块三：内容浏览

### F-B01 · 首页 Feed 流（P0）

**用户故事**

> 我打开首页，希望看到各种场景的烹饪内容，能快速扫一眼标题和成品图，决定是否点进去。

**Feed 卡片展示内容**

|元素|说明|
|---|---|
|封面图|成品图第一张；无则取步骤第一张图；均无则显示系统占位图|
|标题|最多显示 2 行，超出截断|
|场景标签|彩色小标签，视觉区分 8 个场景|
|作者信息|头像（小）+ 昵称|
|点赞数|显示简化数字（1.2k / 3.4w）|
|浏览量|同上|

**排序与筛选策略**

- 默认排序：按发布时间倒序（最新优先）
- 场景筛选：顶部固定 Tab，包含「全部」+ 8 个场景标签，点击切换
- 不做算法推荐（产品决策，有意为之）

**验收标准（AC）**

```
AC-1  首页首屏（前 20 条）加载时间 < 2 秒（P95，4G 网络）
AC-2  场景 Tab 切换后 300ms 内完成刷新
AC-3  下拉刷新获取最新内容，上拉触底加载下一页（每次 20 条）
AC-4  未登录用户可正常浏览全部内容
AC-5  Feed 数据缓存于 Redis，TTL 5 分钟（新内容发布时主动清除缓存）
AC-6  全部内容不足 1 页时，不显示「加载更多」，显示「已经到底啦」
```

**技术决策说明**

> 刻意不做算法推荐，是「松弛感」定位的工程实现——没有内容会因「不被算法青睐」而消失。时间排序对新创作者（T3）极友好，也是与小红书 / 抖音最显著的产品气质差异之一。当 DAU 超过 10 万时，可考虑在「全部」之外增加一个「精选」Tab 引入人工运营排序，但场景 Tab 永远保持时间排序。

### F-B02 · 内容详情页（P0）

**用户故事**

> 我点进一篇内容，想完整看到所有步骤和图片，清楚知道需要哪些食材、大概做多久，在真正做饭时可以一步一步对照。

**详情页展示内容**

|区域|内容|
|---|---|
|头部|标题 + 场景标签 + 发布时间（「X 天前」相对时间）|
|成品图|最多 3 张，横向滑动查看，支持点击放大|
|基本信息|烹饪时长（如有）+ 口味 / 食材标签|
|食材清单|卡片式展示，食材名 + 用量，可整体复制（P1）|
|步骤列表|序号 + 文字说明 + 步骤图（点击放大），步骤间视觉分割线|
|作者信息卡|头像 + 昵称 + 签名 + 关注按钮 + 「TA 的其他内容」入口（最多 3 条）|
|互动区|点赞按钮 + 点赞数 + 浏览量|

**验收标准（AC）**

```
AC-1  详情页加载时间 < 1.5 秒（P95）
AC-2  访问详情页触发浏览量 +1（防重复：同一用户/IP 同一内容 1 小时内只计 1 次）
AC-3  步骤图和成品图点击后全屏放大，支持双指缩放，左右滑动切换
AC-4  页面 URL 可直接分享，未登录用户可完整查看所有内容
AC-5  作者信息卡中的「TA 的其他内容」展示该作者最新 3 篇（排除当前这篇）
AC-6  内容不存在或已删除时，返回 404 页面（显示「内容不存在或已被删除」）
```

## 6. 模块四：互动（点赞 / 浏览量）

### F-I01 · 点赞 / 取消点赞（P0）

**用户故事**

> 我看完一篇觉得很赞的内容，想点赞表示喜欢。如果手误点了，可以取消。点赞后创作者能看到数据变化。

**功能描述**

- 登录用户可对内容点赞 / 取消点赞，同一用户同一内容只能点赞一次
- 点赞状态持久化，用户登出再登入后仍显示已点赞
- 点赞数显示在 Feed 卡片和详情页

**核心技术实现**

```
写入流程（点赞）：
  1. 用户请求点赞 → 校验登录态
  2. 检查 Redis Set（key: likes:{post_id}）中是否已存在 user_id
  3. 不存在（未点赞）→ SADD likes:{post_id} {user_id} + INCR like_count:{post_id}
  4. 存在（已点赞）→ 返回幂等响应，不重复计数
  5. 异步 Worker 每 5 分钟将 Redis 数据批量同步至 MySQL

写入流程（取消点赞）：
  1. SREM likes:{post_id} {user_id} + DECR like_count:{post_id}
  2. 同步逻辑一致

降级策略：
  Redis 不可用时，直接写 MySQL，保证功能可用，性能降级可接受
```

**验收标准（AC）**

```
AC-1  点赞操作端到端响应时间 < 200ms（Redis 写入，不等待 MySQL 同步）
AC-2  点赞后按钮状态即时变化（实心 → 空心 / 颜色变化），数字 +1 动效
AC-3  取消点赞后数字 -1，不低于 0
AC-4  未登录用户点击点赞，弹出登录引导浮层
AC-5  Redis 故障时，降级为直接写 MySQL，功能不中断
AC-6  并发场景下重复点赞请求，幂等处理，点赞数不重复累加
```

### F-I02 · 浏览量统计（P0）

**功能描述**

- 每次访问内容详情页触发浏览量 +1
- 防刷策略：同一用户（已登录，以 user_id 去重）或同一 IP（未登录，以 MD5(ip) 去重）同一内容 1 小时内只计 1 次

**核心技术实现**

```
Redis Key 设计：
  去重 Key: pv:dedup:{post_id}:{user_id 或 md5(ip)}   TTL: 3600s
  计数 Key: pv:count:{post_id}

访问流程：
  1. 用户访问详情页
  2. 检查去重 Key 是否存在
  3. 不存在 → INCR pv:count:{post_id} + SET 去重 Key TTL 3600s
  4. 存在 → 不计数，直接返回
  5. 异步 Worker 每 10 分钟批量同步至 MySQL
```

**验收标准（AC）**

```
AC-1  浏览量统计不影响详情页加载速度（异步写入，不阻塞响应）
AC-2  同一用户在 1 小时内多次访问同一内容，浏览量只 +1
AC-3  浏览量在详情页和 Feed 卡片中均正确展示（允许 10 分钟以内的数据延迟）
AC-4  浏览量最小显示为 1（即作者自己访问也计入）
```

## 7. 模块五：搜索

### F-S01 · 关键词搜索（P0）

**用户故事**

> 我想搜「出租屋红烧肉」，找到相关内容，搜索结果要快，最好能按场景继续筛选。

**功能描述**

- 搜索范围：内容标题（权重高）+ 场景标签（权重中）+ 口味 / 食材二级标签（权重低）
- 搜索结果支持按场景标签二次筛选
- 搜索结果排序：MySQL FULLTEXT 相关度得分优先，同等相关度下按点赞数倒序
- 支持搜索历史（本地缓存，最近 10 条，P1）

**技术决策说明**

> MVP 使用 MySQL 全文索引（FULLTEXT INDEX）而非 Elasticsearch，理由如下：
> 
> - 10 万 MAU 以内，MySQL FULLTEXT 搜索延迟可控制在 200ms 以内
> - 避免引入 ES 的运维复杂度（ES 单节点最低需要 2GB 内存，显著增加服务器成本）
> - 迁移触发条件明确：当 MAU 超过 50 万 **且** P95 搜索延迟超过 500ms 时，启动 ES 迁移评估
> 
> 索引设计：在 `posts` 表的 `title` 字段建 FULLTEXT 索引，使用 `MATCH ... AGAINST ... IN BOOLEAN MODE`。

**验收标准（AC）**

```
AC-1  搜索响应时间 < 500ms（P95，10 万内容量级）
AC-2  空搜索词 / 仅空格，不执行搜索，提示「请输入搜索内容」
AC-3  搜索结果中关键词高亮显示（标题中匹配词加粗或颜色标注）
AC-4  无搜索结果时，显示「还没有相关内容，要不要来第一个发？」+ 发布引导
AC-5  搜索结果支持按场景标签筛选，切换 Tab 不重新触发搜索请求（前端过滤）
AC-6  未登录用户可正常使用搜索功能
AC-7  搜索词长度限制：1 ~ 50 字符，超出截断
```

## 8. 模块六：关注

### F-F01 · 关注 / 取消关注（P1）

**用户故事**

> 我很喜欢某个作者的出租屋料理风格，想关注 TA，以后能方便看到 TA 的更新。

**功能描述**

- 用户 A 关注用户 B：A 的「关注数」+1，B 的「粉丝数」+1
- 支持互相关注，无需对方同意
- 关注 / 粉丝列表在个人主页展示（头像 + 昵称，可点击跳转对方主页）
- 每人最多关注 3,000 人（防僵尸关注）

**验收标准（AC）**

```
AC-1  关注 / 取消关注操作响应时间 < 300ms
AC-2  未登录用户点击关注，弹出登录引导
AC-3  不能关注自己，接口层拦截并返回 400
AC-4  已关注再次点击「关注」按钮，变为「已关注」状态，再次点击取消关注（需二次确认）
AC-5  超过 3,000 关注上限时，提示「关注人数已达上限」，不允许继续关注
```

**边界说明**

- MVP 不做「关注的人的内容专属 Feed」（F-F02，二期）
- 不做关注通知推送（三期）
- 不做私信 / 聊天功能（永久不在规划内）

**用户故事**

## 10. 全局边界说明

### 10.1 永久排除功能

|功能|排除理由|
|---|---|
|评论功能|产品决策，永久排除。评论会显著增加内容审核成本、放大社区负能量，与「不焦虑」的产品气质相悖。平台的正反馈闭环通过「点赞」实现，无需评论。|
|私信 / 聊天|超出 MVP 范围，且有较高安全合规风险，三期以后单独评估|
|视频内容发布|MVP 只做图文，视频需要额外的转码服务和存储成本，二期评估|
|直播|不在产品定位范围内|
|付费内容|二期评估|

### 10.2 内容安全边界

- 所有用户生成的文字内容（标题 / 步骤文字 / 昵称 / 签名）在写入前经过敏感词过滤
- 图片内容通过接入第三方图片审核 API（阿里云内容安全 / 腾讯云天御）做异步审核
- 违规内容：前台隐藏，后台标记，人工复审队列处理
- 账号封禁：Redis Session 立即失效，JWT 加入黑名单

### 10.3 性能目标（SLO 初稿）

|指标|目标值|
|---|---|
|首页 Feed 流首屏加载|< 2s（P95，4G 网络）|
|内容详情页加载|< 1.5s（P95）|
|搜索响应时间|< 500ms（P95）|
|点赞操作响应|< 200ms（P95）|
|图片上传（单张 5MB）|< 5s（P95，4G 网络）|
|系统可用性|≥ 99.5%（月度）|

## 11. 数据埋点初稿

> 以下埋点为 MVP 阶段最小必要集，用于后续产品迭代决策。格式：`事件名 | 触发时机 | 必要属性`

### 用户行为类

```
user_register         | 注册成功            | user_id, platform(iOS/Android/Web)
user_login            | 登录成功            | user_id, platform
post_view             | 访问内容详情页       | post_id, user_id(可为空), scene_tag, referrer(feed/search/profile)
post_like             | 点赞               | post_id, user_id, scene_tag
post_unlike           | 取消点赞           | post_id, user_id
post_publish          | 发布内容成功        | post_id, user_id, scene_tag, step_count, has_cover_image, has_ingredient_list
search_query          | 执行搜索           | keyword, result_count, user_id(可为空)
search_result_click   | 点击搜索结果        | post_id, keyword, rank_position
scene_tab_click       | 点击场景 Tab       | scene_tag, user_id(可为空)
profile_view          | 访问他人主页        | target_user_id, visitor_user_id(可为空)
follow                | 关注用户           | target_user_id, user_id
unfollow              | 取消关注           | target_user_id, user_id
```

### 核心业务指标（通过埋点计算）

|指标|计算方式|关注频率|
|---|---|---|
|DAU / MAU|去重 user_id 日 / 月计数|每日|
|发帖率|发帖用户数 / DAU|每日|
|点赞率|点赞事件数 / post_view 事件数|每日|
|搜索使用率|有搜索行为的用户数 / DAU|每日|
|场景分布|各 scene_tag 的 post_view 占比|每周|
|内容生命周期|内容发布 N 天后仍获得的浏览量|每月|

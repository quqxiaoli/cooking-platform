## 3. 模块一：用户系统

### F-U01 · 用户注册（P0）

**用户故事**

> 作为一个第一次来到平台的年轻人，我想用手机号快速注册一个账号，不希望填写太多信息，注册完成后能立刻开始浏览内容。

**功能描述**

- 注册方式：手机号 + 短信验证码（MVP 唯一方式）
- **[v1.1 from Step 3]** 注册成功后自动生成默认昵称 `厨友` + 手机号末 4 位（例如 `厨友8000`），以及系统内置的默认头像（Step 9 接入图片上传后切换为 OSS 6 款图随机分配；MVP 阶段头像字段为空字符串，前端用占位图渲染）
- 注册即视为同意《用户协议》和《隐私政策》（需在界面上显示链接）

**[v1.1 from Step 3] 关于默认昵称文案变更**

v1.0 设计文案为 `厨房探索者_XXXX`，实施时调整为 `厨友 + 末 4 位`：

- 更短（4 字 vs 6 字），视觉占位友好
- 更亲切、口语化，与 PRD-Phase1 §2.4 关键词「松弛 / 真实 / 有温度」匹配
- 用户可通过 PATCH /users/me 自行修改昵称，默认值仅作占位

**验收标准（AC）**

```
AC-1  输入有效手机号，60 秒内收到 6 位纯数字验证码短信
AC-2  验证码输入正确，1 秒内完成注册，自动登录并跳转首页
AC-3  同一手机号每天最多发送 5 次验证码，超限后返回「今日发送次数已达上限，请明天再试」(错误码 410106)
AC-4  同一 IP 每天最多发送 10 次验证码，超限后返回 410107
      [v1.1 from Step 3] AC-3 / AC-4 分两条独立 AC，因实现为两个独立 Redis Key 与错误码
AC-5  注册已有手机号，返回「该号码已注册」并提供跳转登录入口
      [v1.1 from Step 3 备注] 实现层 MVP 改为"验证码登录即注册"——已注册号码用 login 接口自动通过；无独立 register 接口
AC-6  注册成功后颁发 JWT AccessToken（有效期 2 小时）和 RefreshToken（有效期 7 天）
      [v1.1 from Step 3] v1.0 写的 7d/30d 实际部署值是 2h/7d（更安全的短命 access）
AC-7  验证码 5 分钟内有效，超时需重新获取
      [v1.1 from Step 3] v1.0 写的 10min，实际部署值是 5min（更严格，验证码暴力破解风险更低）
```

**边界说明**

- MVP 不做微信 / Apple / Google 第三方登录（二期补充）
- 不做邮箱注册
- 不做密码设置（纯验证码登录体系）

### F-U02 · 用户登录 + Token 刷新 + 登出（P0）

> **[v1.1 from Step 3] 标题变更**：v1.0 原标题为「用户登录」，实施层确认 refresh + logout 是 JWT 双轨方案的**必备组成**（缺一则双轨架构不闭环），合并入同一功能项。

**用户故事**

> 作为已注册用户，我想随时用手机号登录，也想在不同设备上保持登录状态，不用每次都重新输入验证码。

**功能描述**

- **登录**：手机号 + 短信验证码（与注册共用同一套验证码服务）
- **[v1.1 from Step 3] Token 刷新**：客户端在 access_token 过期前调用 `/api/v1/auth/refresh`，用 refresh_token 换取新的 access_token + refresh_token 对（refresh token 不轮换签名，按自然过期）
- **[v1.1 from Step 3] 登出**：调用 `/api/v1/auth/logout`，服务端将当前 access_token 的 JTI 写入 Redis 黑名单（TTL = 该 token 剩余 ttl，动态计算），后续所有用此 token 的请求立即返回 401003
- Session 管理：JWT（无状态验证）+ Redis 黑名单（强制下线 / 封号能力）双轨并行
- 同一账号最多同时在 3 台设备登录，超出时踢出最早登录的设备（**P1 实装**，MVP 不限设备数）

**验收标准（AC）**

```
AC-1  有效验证码登录，1 秒内返回 JWT Token 和用户基本信息（id / 昵称 / 头像）
AC-2  Access Token 过期前 10 分钟内，客户端调用 /auth/refresh 自动换取新 Token
      [v1.1 from Step 3] 新增接口 AC
AC-3  Logout 后，旧 access_token 立即失效，下次受保护接口请求返回 401003
      [v1.1 from Step 3] 新增接口 AC
AC-4  Redis 中标记封号状态（user:ban:{user_id}），该用户下次任意请求立即返回 410109 并提示封号原因
AC-5  退出登录时，服务端立即在 Redis 中使 JWT 失效（写黑名单），客户端本地清理 refresh token
```

**[v1.1 from Step 3] 接口列表（实装）**

| Method | Path | 鉴权 | 说明 |
|---|---|---|---|
| POST | /api/v1/auth/send-code | ❌ | 发送短信验证码（三维限流） |
| POST | /api/v1/auth/login | ❌ | 验证码登录即注册，返回 token pair |
| POST | /api/v1/auth/refresh | ❌（凭 refresh token） | 刷新 access token |
| POST | /api/v1/auth/logout | ✅ | 黑名单当前 access token |

**技术决策说明**

> 选择「短信验证码」而非「账号密码」：密码管理是认知负担，对 25 岁主力用户群体，验证码登录体验更流畅，也消除「忘记密码」的支持成本。后期可视需求在「设置」中补充密码功能。

> **[v1.1 from Step 3] 关于 refresh + logout 接口必要性**：JWT 无状态验证下，access token 一旦签发就无法主动撤销，除非引入黑名单。无 refresh → access token 短命就让用户频繁重登（UX 差），长命就增大泄露风险（安全差）。无 logout → 用户主动退出后旧 token 仍可继续使用，等价于"登了等于没登"。两接口为 JWT 双轨方案的必备组成，不能省略。

### F-U03 · 个人主页（P0）

**用户故事**

> 作为创作者，我希望有一个属于自己的主页，展示我的所有内容，让关注我的人能看到我的烹饪记录。
> 作为消费者，我希望点进作者主页时能看到 TA 的所有内容和基本信息。

**功能描述**

个人主页展示内容：

| 内容 | 说明 |
|---|---|
| 头像 + 昵称 | 顶部醒目展示 |
| 个人签名 | 最多 200 字，可选（v1.0 写 50 字，实施层按 PRD-Phase3 §5.2 users.bio VARCHAR(200) 对齐为 200 字） |
| 统计数据 | 发帖数 / 获赞总数 / 关注数 / 粉丝数（CountConsumer 异步同步，秒级一致） |
| 内容列表 | 该用户发布的所有内容，按发布时间倒序，瀑布流展示 |
| 操作按钮 | 查看他人主页时显示「关注 / 已关注」按钮；查看自己主页时显示「编辑资料」 |

**[v1.1 from Step 4] 前端路由 与 后端 API 分工**

v1.0 写的「个人主页 URL `/u/{user_id}`」是**前端路由**；实现层后端 API 是：

| 用途 | URL | 类型 |
|---|---|---|
| 前端用户访问入口 | `/u/{user_id}` | 前端 SPA 路由 |
| 后端：用户基本信息 | `GET /api/v1/users/:id` | API |
| 后端：用户的帖子列表（游标分页） | `GET /api/v1/users/:id/posts?cursor=&size=` | API |
| 后端：用户的关注列表（Step 8） | `GET /api/v1/users/:id/following?cursor=&size=` | API |
| 后端：用户的粉丝列表（Step 8） | `GET /api/v1/users/:id/followers?cursor=&size=` | API |

前端 `/u/{id}` 页面内部并发 fetch 上述 API 后拼接渲染。

**验收标准（AC）**

```
AC-1  个人主页前端 URL 格式：/u/{user_id}，支持直接分享访问
AC-2  未登录用户可查看他人主页和内容列表，关注按钮点击后跳转登录引导
AC-3  内容列表分页加载，每页 20 条，下拉触底自动加载下一页
AC-4  内容列表为空时，显示「还没有发过内容，去记录第一道菜吧」
AC-5  自己的主页右上角显示「编辑资料」入口
AC-6  [v1.1 from Step 4] 后端 GET /api/v1/users/:id/posts 与 follow 系列路由
      (/users/:id/followers, /users/:id/following) URL 风格统一，便于前端代码复用
```

### F-U04 · 编辑个人资料（P0）

**功能描述**

可编辑字段：

| 字段 | 规则 |
|---|---|
| 头像 | 支持 JPG / PNG / WebP，最大 5MB，服务端裁剪为 400×400px 正方形存储（Step 9 OSS 直传后实装） |
| 昵称 | 2 ~ 20 字符，不允许纯空格，过敏感词过滤 |
| 个人签名 | 最多 200 字，可留空 |

**接口规范**

`PATCH /api/v1/users/me` —— PATCH 语义，仅传需要修改的字段；不传的字段保持原值。

**验收标准（AC）**

```
AC-1  头像上传成功后，全站该用户头像在 5 秒内全部更新（CDN 缓存刷新；Step 9 实装）
AC-2  昵称包含敏感词时，明确提示「昵称包含不允许使用的词语」，不保存
AC-3  昵称与他人完全相同时，允许保存（不强制唯一昵称）
AC-4  所有字段更新后即时生效，无需重新登录（cache.DeleteUserInfo 主动失效）
```

## 4. 模块二：内容发布

### F-C01 · 发布烹饪内容（P0）

**用户故事**

> 作为创作者，我想发布一篇记录今天烹饪过程的图文内容，可以上传每个步骤的图片，写上简单说明，选一个场景标签，然后发布出去。即使只有 1 张图和几行字也应该能发出来。

**[v1.1 from Step 4] 内容字段定义**

| 字段 | 必填 | 上限 / 规则 | MVP 状态 |
|---|---|---|---|
| 标题 | 是 | 2 ~ 50 字（实施层放宽至 1 ~ 100 字以兼容更多场景） | ✅ Step 4 实装 |
| 场景标签 | 是 | 从 8 个预设场景中选 1 个，详见第 9 节 | ✅ Step 4 实装 |
| 正文（content） | 是 | 最多 5000 字 | ✅ Step 4 实装（文字版） |
| 步骤列表 | 否 | 至少 1 步，最多 30 步；每步含文字说明（必填，最多 500 字）+ 图片（可选，每步最多 3 张） | ⏸️ **推迟至 Step 9 图片上传**（拆 post_steps 子表） |
| 成品图 | 否 | 最多 3 张 | ⏸️ Step 9 |
| 食材清单 | 否 | 每行格式：「食材名 + 用量」，最多 30 行 | ⏸️ P1 |
| 口味 / 食材二级标签 | 否 | 多选，从预设列表选择 | ⏸️ P1 |
| 烹饪时长（cook_duration） | 否 | 单选：15 分钟内 / 30 分钟内 / 1 小时内 / 1 小时以上 | **[v1.1 from Step 4]** DB 列已建（默认 0），DTO 与 API **推迟至 P1**接入；Step 4 CreatePostReq 不收此字段 |

**[v1.1 from Step 4] MVP 文字版策略**

v1.0 设计的内容字段是"结构化步骤图文"，Step 4 实施为"扁平 content TEXT"的简化版：

- **理由**：MVP 文字版需要快速跑通生产-分发链路；step 列表 + 图片需要 post_steps 子表 + OSS 直传两层抽象，一次性引入风险大
- **迁移路径**：Step 9 引入图片上传时新增 `migration 000004_create_post_steps.up.sql`，将旧 content 字段保留为 legacy 兼容字段；或将 content 整体迁入 post_steps.text_only_legacy 后软废弃 content 列

**图片处理规则**

（v1.0 内容沿用，Step 9 实装）

**[v1.1 from Step 4] 发布流程（实际）**

```
用户填写内容
  → 客户端校验（标题非空 / 已选场景标签）
  → [Step 9 起] 图片上传（并发上传 + 进度条）
  → 提交发布请求
  → 服务端基础校验（标题长度 / scene_tag 范围）
  → [Step 10 起] 提交内容审核 API
  → 写入 MySQL (audit_status=0)
       │
       ├─ [Step 4-9 dev mode] is_visible=1 直接可见
       └─ [Step 10 起] is_visible=0 等审核结果
  → 发出 PostEvent（CountConsumer 维护 users.post_count）
  → INCR feed:ver（Feed 缓存失效）
  → 跳转内容详情页
```

**验收标准（AC）**

```
AC-1  最简发布：仅填标题 + 选场景标签 + 写正文，即可发布成功
AC-2  发布过程中网络中断，草稿自动保存至本地（LocalStorage / 客户端缓存）
AC-3  发布成功后 1 秒内跳转至该内容详情页
AC-4  每用户每天最多发布 20 篇，超出后提示「今日发布次数已达上限」（用 limit:pub:{user_id} 通用滑动窗口限流）
AC-5  [Step 10 起] 标题或正文含敏感词，返回「内容包含不允许的词语」(40004) 并指出具体字段
AC-6  [Step 9 起] 图片上传失败时，单张支持单独重试
AC-7  [v1.1 from Step 4] Step 4-9 MVP 文字版期间，发帖直接 is_visible=1
      audit_status=0；Step 10 起进入完整审核状态机
```

**边界说明**

- MVP 不支持视频上传，仅图文
- 不支持定时发布（二期）
- 发布成功后可无限次编辑（F-C02）
- 草稿功能仅本地缓存，不同步至服务端

游标使用毫秒级 unix 时间戳（基于 created_at），不是 v1.0 提到的 ISO 字符串 — 前者更高效、避免时区歧义。

### F-B03 · 作者主页聚合（P0）

> **[v1.1 from Step 4] 后端 API 路径明确**

与 F-U03 个人主页共用同一前端页面（前端路由 `/u/{user_id}`），后端 API：

| 用途 | URL |
|---|---|
| 用户基本信息（统计数据 / 头像 / 昵称） | `GET /api/v1/users/:id` |
| 用户的帖子列表（游标分页） | `GET /api/v1/users/:id/posts?cursor=&size=` |

**为什么不用 `/feed?author_id=`**：

- 与 follow 系列路由 (`/users/:id/followers`, `/users/:id/following`) URL 风格一致
- 语义化更清晰：作者主页是 user 的子资源，而非 feed 的过滤维度
- URL 复用 `:id` 占位符已被路由组定义

### F-I01 · 点赞 / 取消点赞（P0）

**[v1.1 备注] 接口路径实际为 RESTful 三动词同 URL**：

| Method | Path | 鉴权 | 说明 |
|---|---|---|---|
| POST | /api/v1/posts/:id/like | ✅ + 限流 200/24h | 点赞（幂等：重复点赞返回 liked=true count 不变） |
| DELETE | /api/v1/posts/:id/like | ✅ | 取消点赞（幂等） |
| GET | /api/v1/posts/:id/like | ✅ | 查询当前用户对此帖的点赞状态 + 当前 count |

而非 v1.0 暗示的"POST /like + POST /unlike"双 URL — 点赞关系是一种资源，HTTP 动词表达 CRUD 即可。

**[v1.1 备注] 限流策略**：Like POST 限流 200 次/24h；Unlike 不限流（滥用模型不对称，限制 unlike 反而拖慢"我反悔了"的合法操作）。

`docs/debt/` 是 PRD v3.0 落地后我新建的一组台账文件（00–09 共 10 份），

### 2. 设计思考：15 条债务分组与执行节奏

清单一共 15 条。我把它们按"撞车阻塞度"分成三组顺序执行：

- **A 组（7 条 P0）**：不修这些 prod 起不来或起来就是错的。
  prod.yaml 缺段（启动崩）、503 越段（监控 panic）、shutdown 不完整
  （滚动重启数据丢失）、phone_key fail-open（合规炸）、Config-First 漏修
  （魔数与 prod cfg 不一致）、PV/Count 缺幂等（RabbitMQ at-least-once 漂移）、
  bus 失败语义文档（Channel→RabbitMQ 行为暴露）。
- **B 组（3 条 prod 启动门面）**：prod 起来后第一波恶意流量打过来必出事。
  没有 docker-compose.prod.yml（部署靠手敲）、Nginx 入口零防护（限流 / body
  size / metrics 公开）、admin 三面零鉴权（Grafana admin/admin / Prometheus
  暴露 / RabbitMQ 管理界面）。
- **C 组（3 条 CI / Ops）**：不影响 prod 起来，但影响"prod 起来之后还能持续
  迭代"。staticcheck @latest 飘逸、CI 不跑 verify_step17、slave init 重启全量
  重建。

执行顺序是 A → B → C 严格串行。每组做完跑 `make verify-step17 + make build +
go test ./... -race` 三连，确保上一组不破坏下一组的前提条件。

**ADR 决策关联**：CLAUDE.md §七规则 1（Config-First）、§八规则挑战机制
（A6 偏离需要登记），最关键的还是 `docs/debt/09_三视图索引.md §4.1` 这份
台账。如果没有这份台账，15 条债务很容易混进 Step 18 的开发流，又会变成
"边部署边救火"。

### 3. 实现节奏

**A 组 · 7 条 P0**

A1（prod.yaml 三段补齐）是开胃菜。`config.prod.yaml` 缺 consumer/cache/metrics
三段，prod 起来会因为 cfg validate 直接 panic。补段顺手加了三段（search / follow /
cors），把 A5 的 Config-First 改造也"顺路"做完。然后写了 `scripts/check_config_parity.sh`
+ Makefile target——三套 yaml 顶级 key 集合用 `comm` 比一遍，任意一份缺段就
非零退出。这是给 Step 18 之后再不漏配 prod 上的护栏，不是一次性工具。

A2（health.go 越段 + envelope 不一致）是最有意思的一题。原始代码用
`c.JSON(503, gin.H{"code": 503001, ...})` 自己拼了一个 503 响应——绕过了
`pkg/response/response.go` 的标准 envelope。问题是 503001 的段位
（X=5）和 errcode.go 头注释"X=4 是 client-facing, X=5 是 server-internal"
冲突。改的时候有两条路：
- **甲案**：把 health 改成走 `response.Unavailable(c, checks)`，errcode 头注释
  纠正"段位真相"，保留 503001 别名 `ErrServiceUnavailable`；
- **乙案**：把 503001 重排进 X=4。

我建议甲案，理由是 GUARD 已经禁止重排已发布错误码定义。用户确认甲案后落地。
errcode.go 头注释也借机把"X=4 / X=5 不是 HTTPStatus 的权威，AppError.HTTPStatus
才是"这条历史口径写明，避免下次有人再纠结段位。

A3（三阶段 shutdown）是 GUARD-06 的反向背刺。原始 main.go 用一个 30s
context 包所有关闭步骤，结果"HTTP 卡了 20s → ConsumerManager 只剩 10s"的
退化非常容易出。改成三阶段：Phase 1（HTTP，5s）→ Phase 2（Consumer + EventBus，
15s）→ Phase 3（OSS + Redis + MySQL，5s），父 context 总预算来自
`cfg.Server.ShutdownTimeout`（默认 30s，5s slack 吸收 goroutine 唤醒抖动）。
`ConsumerManager.Shutdown(ctx)` 改成带 ctx，超时返回 ctx.Err()，main.go
据此 Warn 后继续下一阶段而不是 fatal exit。还写了 `closeWithDeadline` helper：
把不带 ctx 的 `Close() error` 放到 goroutine 里跑，配合 select+ctx.Done 形成
软超时——关连接的 syscall 卡死时进程不至于被一个坏连接拉住。

A4（phone_key fail-closed）是合规口子。原 `EncryptPhone(keyHex="")` 直接返回
plaintext——dev 方便，但只要 prod 漏注入 env var，全表 phone_encrypted 列就是
明文。改成 `keyHex==""` 一律返 `ErrEmptyKey`，service 层捕获后翻译成
`errcode.ErrPhoneKeyMissing`（新加 480103，注释明确不返客户端、只供日志
+ 告警）。再加一道 `config.Validate`：`Mode=="release" && PhoneKey==""` 直接
拒启动——这是 GUARD-18 的实体化。dev 怎么办？config.yaml 写一个 committed
test key + `# DEV ONLY` 注释，phone_test.go 同步换上同一个 key。"环境变量
fallback 写魔法值"是常见的反模式，这个改造让 dev 路径显式 vs prod 路径显式，
没有静默回落。

A5（Config-First 五件套）按 USER-03 / SEARCH-01 / FOLLOW-01 / CORS-01 / AUDIT-01
顺序做。USER-03 和 SEARCH-01 是平凡的"常量提进 cfg.*"——加字段 + validate +
三 yaml 段。FOLLOW-01 多一步：`maxFollowing` 是 const 时直接 `cnt >= maxFollowing`
就行，提进 struct 后 `cnt`（int64）和 `s.maxFollowing`（int）类型不一致，编译
报错——加个 `int64(s.maxFollowing)` 转换就好，但这种"类型边界"是 Config-First
唯一容易踩的坑，值得注意。CORS-01 最复杂：原 `CORS()` 函数硬编码 `"*"`，要改
成接受 `config.CORSConfig`。借机做了三件事：(a) 用 origin map 做 O(1) 查；
(b) 加 `Vary: Origin` 头（CDN 缓存按 Origin 隔离）；(c) wildcard
`["*"]` 短路，单元配置零拷贝。

A6（PV/Count IDEMP）原计划走"channel 类型扩展透传 EventID"——把 `chan int64`
改成 `chan eventEnvelope` 之类的。开工前我重新评估了一下，发现可以更省：
**直接在 subscribe handler 入口做 SETNX 拦截**，channel 类型签名不动。已 dedup
的事件根本不进 batch slot，效果与"透传后再去重"等价，代码改动面小一半。
新增 `internal/cache/event_dedup_cache.go` 提供 `SeenOrMark(ctx, topic, eventID)`，
key 模板 `dedup:<topic>:<event_id>`，TTL 来自 `cfg.Cache.DedupTTL`（24h）。
Redis 故障 fail-open：rare double-count 远比 silent data loss 可接受。这是
一处刻意偏离原计划，主动告知了用户、登记进偏离点。

A7（bus 失败语义对照表）算是文档工程。在 `internal/event/bus.go` 的 Handler
类型注释里加了一张 3 列对照表（Handler 返 err / 返 nil / 进程崩溃 × Channel /
RabbitMQ），明确"返 error 不会重试，channel 不重投；同时也不要假设处理过的
事件不会再来，RabbitMQ 至少一次"。channel.go / rabbitmq.go 写交叉引用。
这条债（TD-EVT-04）的本质不是代码 bug，是"两种实现的可观察行为差异"在
Consumer 编写者那里没有显式契约——补一张表就关闭了。

**B 组 · 3 条 prod 启动门面**

B1（docker-compose.prod.yml）是个全新文件。和 dev compose 的关键差异：
(a) restart=always；(b) app1/app2/mysql/redis/rabbitmq/prometheus 全部不
publish host port，只有 nginx 80/443；(c) Grafana 单独 `127.0.0.1:3000:3000`，
admin 走 SSH tunnel；(d) 所有密码用 `${VAR:?Required}` 强制要求 `.env.prod`，
docker compose 在 parse 阶段就拦截缺值；(e) Redis 加 `--requirepass`，PRD
v3.0 §9 已经定调要密码。配套的 `.env.prod.example` 把所有 secret 列出
（MySQL / Redis / RabbitMQ / JWT / phone_key / OSS / SMS / 内容审核 /
Grafana admin），全部用 `<replace-...>` 占位，clipboard-driven 部署。
最后改 `.gitignore`：`.env` → `.env*` 然后白名单 `!.env.example` /
`!.env.prod.example`——以后再加 `.env.staging` / `.env.preview` 也不会漏。

B2（Nginx 限流 + body size）是上线第一波防御。`limit_req_zone` 双桶：
api_zone 30r/s + 50 burst nodelay（"真实用户 90% 打不到，机器人 / 死循环
打得到"），sms_zone 1r/m + 2 burst（SMS 发送有真实成本，service 层已经
按手机号 / IP 限了，Nginx 再加一层 IP 桶）。`client_max_body_size 6m`——
OSS 头像上限 5MiB（cfg.OSS.MaxImageSize），Nginx 6MiB 留 multipart
boundary 余量，超过直接 413 不进后端。然后按 `/health`（不限流）/
`= /api/v1/auth/send-code`（sms_zone）/ `/api/`（api_zone）/ `/`（catch-all）
分四个 location，注意 exact match `= /...` 优先级压过 prefix `/api/`。

B3（admin 三面鉴权）最简单的一刀：`location = /metrics { return 404; }`
压在最前面，公网入口一律 404。Prometheus 仍走 docker network 内部 scrape
app1/app2 的 :8080/metrics，不受影响。Grafana 走 SSH tunnel，前面 B1 已经处理。
RabbitMQ 管理界面 dev 暴露 15672 在 prod 完全不 publish，已经达成。

**C 组 · 3 条 CI / Ops**

C1 一行修：`staticcheck@latest` → `@2025.1.1`，加注释"升级前阅读 release
notes"。锁死版本，让"早上还绿的 PR 突然红了"不能再发生。

C2 在 pr.yml 里 staticcheck 之后插一行 `make verify-step17`。这条让"我又
忘了改 pr.yml"的事故能在 review 之前被 CI 自身发现——CI self-check。

C3 是 init-slave.sh 的幂等保护。原脚本 restart=no，但 docker compose up -d
触发容器重建时会重跑——每次都 `mysqldump master + RESET MASTER + restore +
START REPLICA`，丢掉 slave 已有 GTID 位置，并且短暂窗口 slave 处于 "truncated
state" 服务陈旧读。改的方式很轻：进脚本第一件事 `SHOW REPLICA STATUS\G`，
IO=Yes 且 SQL=Yes 就 exit 0；其他状态（empty set / 半边 Running=No）走原来
的 rebuild 流程。grep 的 exit code 用 `|| true` 兜底，因为 mysql 对 empty set
+ -e 是非零退出，但 empty 本身就是"fresh slave"信号。

### 4. 踩坑记录

**坑 1：FollowService maxFollowing 类型边界**
A5 把 const `maxFollowing = 3000` 提进 struct 之后，原代码 `if cnt >= maxFollowing`
直接报 `mismatched types int64 and int`。cnt 是 `repo.CountFollowing` 返的
int64（行数），原 const 用的是默认 int 类型——const 时 Go 自动适配，提成 struct
字段后类型就硬下来了。`int64(s.maxFollowing)` 一行修。Config-First 改造里
绝大多数字段是 string / time.Duration，只有数字字段会撞这种边界。

**坑 2：setupRouter 没有 cfg 在作用域里**
中间一度把 `middleware.CORS(cfg.CORS)` 直接塞进 setupRouter，发现 setupRouter
签名里没有 cfg。两个修法：(a) 透传 `cfg *config.Config`——但 setupRouter
不需要别的 cfg 字段，过度耦合；(b) 加 `corsCfg config.CORSConfig` 参数。
选(b)，最小侵入。这种"原本只在 main 用的 cfg 现在要传一层"的口子，反映
之前 setupRouter 设计就是 cfg-agnostic 的，现在第一次有 cfg 字段需要往下
传——把这次的口子开到"按需传具体子结构"而不是"把整个 cfg 塞下去"，方向
是对的。

**坑 3：A6 channel 类型扩展决策**
原计划是改 `chan int64` / `chan countDelta` 为带 EventID 的 envelope 类型，
batch 时去 channel 消息上读 EventID。开工时重新评估发现 subscribe 边界
（handler 内部）就是天然的去重切点——已 dedup 的不进 channel，flushLoop
逻辑完全不用动。改动面小一半，但同时也意味着"dedup 缓存 RT 进了消费热路径"。
计算了一下：SETNX 一次 round-trip ≈ 0.3ms，PV 高峰按 5k/s 算每条
0.3ms × 5000 = 1500ms/s = 1.5 core，明显可接受。决定偏离原计划，主动告知
用户并登记。

### 5. 收获

**最大收获：把一份债务台账"用掉"的体验。**

之前几步收尾都会在"本步与 PRD 的偏离点"章节记一些零散债务，但从没有过
一个**统一的视图**来回看所有债。`docs/debt/` 把 17 步累计的所有架构妥协、
设计失误、跨模块债集中到一处，再按"档位 × 触发条件 × 成本"三维排序。
排序完才看出来：原以为"先做 Step 18 部署 + 边走边修"是最快的，实际上
"先一次性清掉 P0 + prod 门面 + CI 三组"才是最快的——因为 prod 起不来 /
起来就错 / 起来后再让团队跑 CI red 都是不可逆的高代价。

第二个收获：**"刻意保留的偏离"也要登记**。A2 errcode 注释纠正 X=4/X=5 段位、
A6 channel vs subscribe-boundary、A7 audit timeout 只加字段不接调用，三处
都不是"实现差异"而是"刻意为之"——但如果不登记，下次有人重读代码会以为
是漏改。CLAUDE.md §八 的偏离点机制不只是给"违反规则"用，也给"故意按 95%
通用规则之外的方式做"用。

第三个收获：**Step 18 真正开工时，不会再有 P0 卡脖子。** 现在 prod 起不来、
phone 落明文、shutdown 丢数据、metrics 公网暴露这四类风险全部清除，
prod compose 文件 + env 模板就位，CI 自检 + slave 幂等都加好——Step 18
可以专注做服务器准备 / 域名 / TLS / 真实数据迁移，不会再被基础设施层
的"早就该修的债"打断。

## Part B · 关联知识点清单

### 基础概念（语言 / 框架层面）

- Go const → struct field 类型边界（int vs int64 自动转换消失）
- `errors.Is(err, sentinel)` 哨兵错误模式
- Gin middleware 闭包捕获 cfg
- HTTP `Vary: Origin` 头与 CDN 缓存隔离
- Docker compose `${VAR:?required}` parse-time 必填校验
- `.gitignore` 黑名单 + `!` 白名单

### 设计模式与架构思想

- Config-First 原则（参数全归 cfg，禁止包级 const）
- LIFO 关闭顺序的"三阶段"细化（HTTP → Consumer/Bus → Infra）
- Fail-closed vs Fail-open 的边界选择（phone_key 选 closed / dedup 选 open）
- 哨兵错误（package-level error variable）vs 错误码（带 HTTP 状态）
- Idempotency at consumer boundary（subscribe 边界去重 vs 业务层去重）
- Defence in depth（Nginx 限流 + service 限流 + Redis 限流三层叠加）
- Self-checking CI（verify_step17 跑在 pr.yml 自己里）

### Go 语言特性与并发模型

- `select { case <-done: ; case <-ctx.Done(): }` 软超时模式
- goroutine + sync.WaitGroup + ctx 三件套优雅停机
- `chan T` 类型签名稳定性（不必为了透传少量元数据扩展 channel 类型）

### 工程实践（测试 / 部署 / 可观测性等）

- Nginx `limit_req_zone` + `limit_conn_zone` 双桶限流
- Nginx `location =` exact match 优先级
- Nginx `client_max_body_size` 与上游 multipart 余量
- Docker compose 服务网络隔离（不 publish host port）
- Grafana SSH tunnel 访问 vs admin 公网暴露
- staticcheck 版本锁定与升级评审纪律
- MySQL `SHOW REPLICA STATUS` 幂等 boot 探测
- mysqldump `--source-data=2 --set-gtid-purged=ON` GTID-aware 备份
- shell `set -euo pipefail` + `|| true` 兜底

### 数据安全与合规

- 手机号 fail-closed：明文落库 = 合规事故
- AES-256-GCM key 长度校验（64 hex / 32 bytes）
- 错误码"never to HTTP"标记（仅供日志 / 告警，不返客户端）
- Redis 必须 requirepass（默认空密码风险）
- CORS allowed_origins 在 release 模式禁 "*"
- /metrics 公网屏蔽（goroutine 数 / GC pause / build info 信息泄露面）

### 可能被延伸追问的关联领域

- Kubernetes lifecycle.preStop hook 与三阶段 shutdown 等价对应
- mTLS / SSO 给 admin 面加鉴权（B3 当前是网络隔离，未来若开放）
- staticcheck 集成到 pre-commit hook（与 CI 双 enforcement）
- WAF / CDN 限流 vs Nginx 限流（多层联动）
- RabbitMQ idempotency 业界方案（dedup cache / 消息表 / 业务幂等键）
- 配置漂移检测（check_config_parity.sh 的延伸：值级别 diff）
- 密钥轮换（KMS / Vault 介入 phone_key 与 pepper）

# 第 9 步 故事线 · 图片上传 OSS 直传

## Part A · 本步故事线

### 1. 起点

进入第 9 步时,平台骨架已稳:用户注册登录、发帖、点赞、搜索、关注全部跑通,但**一切都还是"纯文本世界"**。Step 4 留过一个 TODO 在 `user_service.go`:`// MVP: accept any string up to 500 chars; tighten in Step 9 when OSS lands`。Step 4 的 `dto/post.go` 也注释明指:`structured steps move out when post_steps subtable lands (Step 9)`。Step 9 要兑现这两个旧账。

更关键的是,Step 10 审核模块即将进入开发——而审核必须拿到帖子里所有图片 URL 才能送阿里云内容安全。如果 Step 9 不把图片字段落地,Step 10 根本无米下锅。

但图片上传不是简单"加个接口"这么轻。商业级项目对图片有三层约束:**Go 服务器不过图片流量**(成本+性能)、**严格白名单**(防恶意第三方域名)、**可追溯**(audit_log 要拿得到每张图)。三者叠在一起,意味着我必须设计一套 presign → 客户端直传 OSS → callback 的解耦链路,而不是粗暴地 multipart/form-data 怼到 Go 进程里。

### 2. 设计思考

进入假设清单前,有三个决策反复横跳了一下。

**决策一:STS 临时凭证 vs PresignedURL。** PRD-Phase3 §8 原本写的是 STS——服务端调阿里云 STS API 申请临时 AK/SK,客户端拿来用 OSS SDK 上传。这是企业级最规范的做法,权限可以精细到 bucket 前缀。但 STS 有两个 MVP 阶段不必背的代价:一是**每次签发都要调一次阿里云 STS API**(增加延迟和费用);二是**客户端必须集成 OSS SDK**(前端打包体积、移动端 SDK 适配都要操心)。我建议改用 **PresignedURL**:服务端用永久 AK/SK 给一个 PUT URL 签名,TTL 15min,客户端纯 HTTP PUT 就能上传。权限粒度从 bucket 前缀降到单个 object_key,但配合短 TTL 和服务端持有的 nonce,安全性足够。这构成对 PRD 的偏离,要在 Step 12 PRD 收口时显式写入 v3.0。

**决策二:CallbackReq 接受什么。** 一开始我设计的 callback DTO 含 nonce + public_url + size 三个字段——和很多教程里写的一样,客户端把"我上传完了,文件在哪"告诉服务端。但仔细想攻击面:如果客户端能传 URL,任何拿到 nonce 的人都可以传一个第三方域名 URL 进来,绕过整个白名单。**最终决定 CallbackReq 只含 nonce**,服务端从 presign 时就存进 Redis 的 NonceRecord 里读 public_url。这把"客户端是不是被劫持"的攻击面砍到只剩"猜中别人的 128-bit 随机 nonce"——理论上不可能。这个收紧不在 PRD 里,但属于安全侧的正向偏离。

**决策三:`image_urls` JSON 字段的 Go 端形态。** post_steps 子表的 image_urls 是 JSON 列。一开始我建议 MVP 用 `string` + service 层手动 `json.Marshal/Unmarshal`,30 行代码搞定。小离一句"哪个更好用用哪个"让我重新考虑——更优雅的做法是自定义 `StringArray` 类型实现 `sql.Scanner` 和 `driver.Valuer`,把 marshal 推到 DB 边界,业务代码看到的就是 `[]string`,append/range/len/JSON 输出全部原生。代价是多 30 行边界代码,但 service 全员清爽,后续 Step 10 AuditConsumer 拉 image URLs 也不用解 JSON。这是个一次性投入永久收益的决策。

三个决策定下来,假设清单成型,接下来就是批量生产。

### 3. 实现节奏

按既定的"四组批量化生成"节奏走:

**第 1 组(数据层)**: migration 000005 建 post_steps 表 + 自定义 StringArray Scanner/Valuer + PostRepository 追加 CreateWithSteps/LoadSteps。这一组最朴素,但暗藏一个事务点:`CreateWithSteps` 必须在 GORM 事务里先 INSERT post 拿到 LAST_INSERT_ID,再回填 step.PostID,再批量 INSERT steps——半提交状态(post 落了 steps 没落)对详情页是灾难性的。GORM 的 `Transaction(func(tx *gorm.DB) error{...})` 模板让这件事只有 8 行代码。

**第 2 组(OSS 抽象层 + 配置)**: pkg/oss 五个文件 + OSSConfig 接到 Config + yaml 双份 + errcode 追加。这一组最大的工程亮点是 **MockClient 内嵌 `net/http` listener**。dev 模式下,MockClient 在 NewClient 时 goroutine 启动一个 listener 监听 127.0.0.1:18080,接受 `PUT /<object_key>` 请求,落到 in-memory map。这让 verify_step9.sh 能跑**完整的 presign → 真实 PUT → callback 链路**,完全不依赖阿里云,也不需要 mock HTTP 库。Close 时通过 `http.Server.Close()` 优雅关闭,`sync.Once` 保证幂等。

**第 3 组(业务层)**: upload_cache + upload_service + user_service/post_service 加白名单校验。重点是 `ConsumeNonce` 用 Redis 6.2+ 的 GETDEL 命令——atomic read-and-delete。之前 GET-then-DEL 两步会留一个竞态:两个并发 callback 各自 GET 到记录后再 DEL,两个都成功。GETDEL 关掉这个口子。go-redis v9 直接 surface 为 `rdb.GetDel(ctx, key)`。

**第 4 组(接入层 + 收尾)**: upload handler + main.go §7.9 装配 + verify_step9.sh + Makefile。装配点位很讲究:`oss.NewClient` 失败 fail-fast `log.Fatal`,shutdown 链路在 HTTP 之后、EventBus 之前关 OSS Client。这样保证不会有新上传请求时再关 mock listener,也不会有 in-flight upload 被半路截断。

### 4. 踩坑记录

**踩坑一:assert_json 期望值带引号(本步唯一一次返工)。** 我写 verify_step9.sh §9 时,对 `.data.steps[0].text` 和 `.data.steps[1].image_urls[0]` 的断言期望值都包了 `"..."` JSON 引号。小离实跑发现报错——因为项目里 `assert_json` 用的是 `jq -r`(raw output),返回的是不带引号的原始字符串。期望值多包一层引号就永远断言失败。这是我对自家工具链没看完整造成的——之前 fetch verify_common.sh 时扫了一眼就过了,没记牢 `-r` 这个细节。小离手动改了两处。教训:**断言工具的 raw vs JSON 模式必须先确认**,不要凭印象写期望值。

**踩坑二:url_prefix 大小写一致性。** IsAllowedURL 做的是 `strings.ToLower` 双侧再 `HasPrefix`。如果配置里写了 `Http://127.0.0.1:18080/` 大写 H,白名单会全失败。一开始 config.validate 没强制小写,后来加了 `if cfg.OSS.URLPrefix != strings.ToLower(...)` 强校验。这样 prod 部署时审计 yaml 文件能看到"配的就是用的",无需运行时心证。

**踩坑三:mock listener 端口必须与 url_prefix 一致。** MockClient 的 UploadURL 是 `http://<listener addr>/<object_key>`,PublicURL 是 `cfg.OSS.URLPrefix + object_key`。两者**主机部分必须一致**才能让 verify 脚本里"PUT 到 upload_url"再"读 public_url"成立。dev yaml 把两者都写成 127.0.0.1:18080,加了注释强调"MUST equal"。

### 5. 收获

完成 Step 9 后,几个工程认知层面的提升:

- **PresignedURL 不是简单的"省一次 API 调用"**,而是把客户端从 SDK 集成里解放出来。在移动端、小程序、Web 多端场景下,纯 HTTP PUT 是真正的 universal interface。
- **MockClient 内嵌 listener** 这种"用 Go 标准库做高保真 fake"模式可以推广到很多场景:mock SMTP server、mock RabbitMQ、mock S3。比起手动写 HTTP roundtripper mock 表达力强得多。
- **Server is the source of truth for resource metadata**——CallbackReq 只含 nonce 这一个设计,实际上把整个上传链路的可信边界划得很清晰:nonce 之前服务端说了算,nonce 之后客户端只是"通知一声"。这个分层会在 Step 10 审核里继续受益(送审的图片 URL 都来自 server-known 的 post_steps,不接受任何 client-supplied)。
- **伏笔到 Step 10**: post + post_steps 双表布局已就绪;AuditConsumer 订阅 `event.post` 后,只要 `SELECT cover_url FROM posts WHERE id=?` + `SELECT image_urls FROM post_steps WHERE post_id=?`,一次扫描拿到全部待审图片。Step 9 给 Step 10 喂好了数据。

- Gin 路由组与中间件嵌套顺序
- Viper 配置加载与 env 覆盖
- UUID v4 vs crypto/rand 16-byte hex

### 设计模式与架构思想
- Provider 工厂模式 + Mock/Real 三件套
- 直传架构(Direct Upload Pattern)
- Token Bucket vs Sliding Window 限流
- Nonce 防重放 + Server-side Source of Truth
- 状态机驱动审核(铺垫 Step 10)
- Defense in Depth(三层白名单:DTO binding + service 白名单 + OSS server)

### Go 语言特性与并发模型
- `sql.Scanner` / `driver.Valuer` 接口
- `sync.Once` 在 Close 中保证幂等
- `net/http.Server.Close()` 与 in-flight 请求处理
- `crypto/rand.Read` vs `math/rand`
- `errors.Is` 与 sentinel error 模式
- `context.Context` 透传与超时传递
- GORM 闭包事务的 panic 自动回滚

### 工程实践(测试/部署/可观测性)
- 内嵌 HTTP server 做高保真 mock
- 端到端验证脚本的 raw mode jq 陷阱
- bash assert helper 库的设计
- migration 编号管理与回滚脚本对偶
- env-injected AccessKey 与 yaml 占位
- `.gitignore` 敏感文件硬化
- grep 自检敏感信息的工作流

### 数据安全与合规
- AccessKey 永远不入仓库
- URL 白名单防 SSRF / 跨域劫持
- nonce 一次性 + TTL 防重放

- 短 TTL 预签名 URL 替代长期 token
- crypto/rand 而非 math/rand 生成安全 nonce
- 服务端拥有 metadata 的真值来源

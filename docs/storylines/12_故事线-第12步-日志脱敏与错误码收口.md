# 故事线 · 第 12 步 · 日志脱敏 + 错误码体系最终收口

---

## Part A · 本步故事线

### 1. 起点

进入 Step 12 时，第二轮（补齐功能）的 11 个步骤全部落地，项目已具备完整的用户系统、内容发布、Feed、审核、搜索、关注、图片上传、手机号加密。但有一类问题在代码中静静躺着：**日志打了什么，从没人系统检查过。** Step 7 的注释甚至留了一句"cosmetic debt，Step 12 收口"——这是一张被动预约的欠账单。

另一面是错误码。从 Step 3 开始积累到 Step 11，errcode.go 先后追加了 8 个模块段位，但 PRD 错误码全表（v2.2 §7.3）只更新到 412xxx，440/450/460/470/480 五个后续段位从来没有进入过文档体系，头部注释也有一处逻辑矛盾（470xxx/480xxx 注释说"5xx"但 code 前缀是 4）。

### 2. 设计思考

**决策一：脱敏在哪一层做？**

最直觉的方案是在每个 `zap.L().Info(...)` 调用处手工 mask——但这等于依赖人工纪律，永远有人会写漏。另一个方向是在 logger 初始化时注入一个 "field redactor"，对所有 zap field 做正则扫描——但这个方案的性能代价高，且对 JSON path 里的嵌套结构无效。

最终选择**两层并行**：
- **结构层**（pkg/crypto/mask.go + pkg/logger/fields.go）：把 `MaskPhone`/`MaskedPhone(zap.Field)` 做成公开 API，让调用方"刻意选择"脱敏的 field 类型，而不是事后正则兜底。这借鉴了 Go 标准库"explicit is better than implicit"的设计气质。
- **防御层**（middleware/logger.go sanitizeQuery）：即使有人把 phone 放进 query string，中间件在记录前会替换掉 value，不需要调用方记得处理。

这两层之间有明确的边界：结构层管"主动打的日志字段"，防御层管"被动进来的 HTTP 元数据"。

**决策二：470xxx/480xxx 要不要改前缀？**

这是本步唯一的争议点。XYYZZZ 规范里 X=4 对应客户端错误，X=5 对应服务端错误——但项目里所有模块（包括明确是 HTTP 500 的 audit 和 encryption）都用 4 前缀。改成 570xxx/580xxx 确实"更对"，但：

1. 这两段 code 从未对外暴露（never returned to HTTP callers），只用于日志 tag
2. 改动影响 audit_consumer.go 里的硬编码字符串 + 调用链，引入回归风险
3. "统一"本身就是一种价值——全部用 4 前缀比"部分用 4、部分用 5"更容易记

用户确认：保持不变，用包注释解释清楚即可。这个决定本身就是一个很好的面试素材：**一致性有时比"正确"更重要**。

**决策三：安全响应头要不要加 dev/prod 分支？**

Step 10 的短信 sender 有 mock/aliyun 两套实现，按 provider 切换。安全响应头要不要类似处理？结论是不需要——X-Content-Type-Options、X-Frame-Options 这类头对浏览器行为的约束在 dev 环境同样适用，且在 dev 环境"关掉"意味着测试路径和生产路径有差异，反而是隐患。单一实现更干净。

### 3. 实现节奏

第一批（工具层）先写，理由是 middleware 和 service 层对它们有依赖，工具层先落地能让后面的修改一次到位。`pkg/crypto/mask.go` 很薄——MaskPhone 8 行，MaskToken 8 行——但把 user_service.go 里私有的 `maskPhone` 提升为公开 API 这个动作，意义在于**把能力边界从 service 包扩展到 pkg 层**，未来 audit_consumer、follow_service 等任何地方都能用。

`pkg/logger/fields.go` 的 `MaskedPhone(phone) zap.Field` 是一个轻量的"安全原语"——调用方只要看到 `logger.MaskedPhone` 就知道这里已经脱敏，不需要去读实现。

第二批（middleware）里 `sanitizeQuery` 是最有意思的细节。`url.ParseQuery` + 遍历 map 替换，总共 20 行，但处理了两个边界：① URL decode 后再比较（`Phone=...` 和 `phone=...` 都能命中）；② `[unparseable]` 兜底——恶意构造的 query string 不会导致明文落日志。

### 4. 踩坑记录

**坑一：测试期望值写错**。`MaskPhone("1234567")` 的期望我写成了 `"123****567"`，实际逻辑是 `phone[:3] + "****" + phone[len-4:]` = `"123****4567"`。len=7 时 `phone[3:]="4567"`，不是 `"567"`。单测跑了才发现，改掉即可，没有影响实现逻辑。`MaskToken("12345678")` 同理——我以为 8 位 token 会走 "前 8 位 + ..." 的分支，但代码写的是 `<= 8` 时全遮，正确行为。

**坑二：Security 中间件注册后 server 没有响应新 header**。第一次重启时 OSS mock 的 18080 端口还被旧进程占着，新 server 启动失败，但旧 server 仍在响应请求（没有 security headers）。现象是"重启了，但 header 没变"，排查了两分钟才意识到有两个进程。`lsof -ti:8080 | xargs kill -9` + `lsof -ti:18080 | xargs kill -9` 彻底清除后才正常。

**坑三：verify 脚本的 XSS header 大小写**。脚本检查 `X-XSS-Protection: 1; mode=block`，而 Go net/http 的 canonical header form 是 `X-Xss-Protection`（不是全大写的 XSS）。curl 输出原始大小写，导致 assert_contains 失败。解决方案：只检查 `mode=block` 这个 value 部分，不检查 header name 大小写——这也是更健壮的写法。

### 5. 收获

这步是第二轮的收口，实际代码量不大，但它迫使我把**整个项目的日志面**扫了一遍。扫完之后发现：除了 middleware query 这个隐患，其余所有日志调用都已经用 `phone_hash` 而非明文 phone，Step 11 的加密改造顺手把这个问题也解决了。

进入第三轮（生产基础设施）的下一步是 EventBus 切换 RabbitMQ——这是第一个真正的基础设施升级，涉及 connection management、channel pool、ACK 机制、死信队列。现有的 channel.go EventBus 实现会保留作为 dev fallback（provider: "channel"），RabbitMQ 实现通过 provider: "rabbitmq" 切换。

## Part B · 关联知识点清单

**基础概念（语言 / 框架层面）**
- HTTP 规范中响应头名称的 canonical form（net/http 的大小写归一化）
- url.ParseQuery 与 query string 注入防御
- Go 包可见性：私有函数 → 公开 API 的提升时机判断

**设计模式与架构思想**
- 防御纵深（Defense in Depth）：结构层（显式 API）+ 防御层（中间件兜底）并行
- 一致性 vs 正确性的取舍（470xxx/480xxx 前缀决策）
- 最小惊讶原则：`MaskedPhone` 的命名让调用方一眼知道已脱敏

**Go 语言特性与并发模型**
- （无）

**工程实践（测试 / 部署 / 可观测性等）**
- 端口占用导致"旧进程响应，新进程假装重启"的排查方法
- 白盒单元测试 sanitizeQuery（测试包级私有函数）
- verify 脚本中 header 断言应 check value 而非 name 大小写

**数据安全与合规**
- PII（Personally Identifiable Information）的日志脱敏级别设计
- 手机号 mask 策略：前 3 + **** + 后 4（工信部推荐格式）
- Token 日志 mask 策略：短 token 全遮，长 token 前 8 + "..."

**可能被延伸追问的关联领域**
- OWASP 推荐的 HTTP 安全响应头完整列表（CSP、HSTS 未在本步加入的原因）
- 生产环境 CORS 白名单配置（当前 `*` 的 dev 策略何时收紧）

# 第 19 步 · 数据库迁移与部署联调（故事线）

## Part A · 本步故事线

### 1. 起点：一个看不见的幽灵和一台被清空的服务器

进入 Step 19 这扇门的时候，我背着两份遗产。一份是上窗口留下的「Viper 嵌套 env 覆盖」死结——`APP_DATABASE_DSN` 怎么注入都覆盖不掉 `config.prod.yaml` 里的占位符，app1/app2 在 `Restarting (1)` 里循环不止。另一份是上窗口几次混乱排查留下的"零输出 exit 1"幽灵——某些尝试下 app 启动后 stdout 一字不出、直接 `exit 1`，无从下手。前者已在 commit `c4c3d2c` 落地修复：`pkg/config/config.go` 引入了 `v.BindEnv("database.dsn", "APP_DATABASE_DSN")` + Unmarshal 后用 `v.GetString("database.dsn")` 强制回填 `cfg.Database.DSN` 的双保险模式，绕开了 Viper mapstructure 对嵌套键的合并不可靠行为。后者依然挂在那里，没人知道它是什么。

外加一台被清空重做的香港 ECS：没有容器、没有镜像、没有数据卷、没有代码，只剩 Docker daemon 和已开的安全组 22/80/443。任务很直白：从这块白板把后端跑起来，让 Mac 上的 curl 能打到 `47.238.29.251/health/ready` 拿 200。

### 2. 设计思考：两次抗诱惑

开工契约要求在写任何 runbook 之前先去 GitHub fetch 真实代码——这条纪律本身就是上窗口"凭印象写出错代码"踩坑后定的。我先拉了 11 份关键文件：config.go、config.prod.yaml、docker-compose.prod.yml、main.go、Dockerfile、.env.prod.example、init-master.sql、init-slave.sh、Makefile、migrations 目录、nginx.conf。fetch 过程中真正改变后续路线的，是两个决策点。

**Risk 1：单主还是主从？** Step 19 原计划是"先单主，读写分离留给 Step 14 后补"——这看起来很务实。但我盯着 docker-compose.prod.yml 看了一会，发现 compose 本来就强制 `mysql-slave` healthy 才让 app 启动，`mysql-slave-init` 整套 GTID 复制脚本已经写好。如果选 A（单主），意味着我们要把这套基础设施"假装存在但忽略"，并在 `.env.prod` 留下永远不被覆盖的 yaml 占位符 `slaves_dsn`，造成"app 起得来、第一个读请求被 DBResolver 路由到 slave 然后炸"的隐雷。选 B（保主从、注入真实 `APP_DATABASE_SLAVES_DSN`）反而更接近最终架构，且能让 Step 14 设计在真实公网环境直接实证。我的判断是软件最难补的就是"先简单后补"——每一次"以后再说"都是一笔技术债的活期存款。选 B。

**Risk 2：要不要预防性硬化 config.go？** Viper 嵌套 env 的不可靠行为，理论上不只影响 `database.dsn`——`redis.password`、`mq.url`、`oss.access_key_id`、`jwt.secret`、`encryption.phone_key` 全是嵌套键。但反向证据让我冷静下来：上窗口的故障文档清楚记录了 app 失败时已经走到 `Access denied for user '<DB_USER>'` 阶段——这意味着 `validate()` 已经过了，而 release 模式下 validate 强制要求 `jwt.secret≥32`、`encryption.phone_key=64hex`、`cors 无 *`。这些值原本都是 yaml 里的占位符，能过 validate 说明它们经 `AutomaticEnv` 覆盖成功了。`database.dsn` 是唯一病态，根因可能是 mapstructure 在某些键值组合下的特殊路径，与一般嵌套键不同。预防性扩展 BindEnv 到全部嵌套 secret 是"治未病"，但违反故障文档自己定下的"一次一变量、对照定位"纪律，且引入了额外代码改动需要 push 后才能继续 fetch。我选 A：runbook 里加一道"配置探针闸口"，在受控一次性容器里观测，让证据自己说话。

这两次都是抗"防御性大改"的诱惑。回头看，这两个决策都对了——主从在本步顺利打通、Risk 2 的探针给出了干净的"全过"结论。

### 3. 实现节奏：五组阶段化推进

Runbook 拆成五组，每组一个验证闸口，没过不进下一组。G1 是基础工作：服务器端独立生成 Deploy Key（私钥不跨机器搬运是铁律）、用 openssl 一次性生成 8 个密钥写进 `.env.prod`。这里有几个看似细节但都是踩坑沉淀：`REPL_PASSWORD=repl123` 必须对齐 `init-master.sql` 硬编码值，否则复制建立失败；`APP_MQ_URL` 必须写完全展开的字面值并用 hex 密码，因为 docker-compose env-file 对 `${}` 内嵌引用的插值行为脆弱，hex 密码也规避了 `/@:` 在 amqp URL 里的转义坑；`SMS/AUDIT` 的 AK/SK 给 `mock-unused` 占位是因 compose `:?` 守卫对空值同样拒启，即使 provider=mock 时 app 根本不读。G2 起基础设施 4 容器，前台跑 `mysql-slave-init` 观察 GTID 复制建立的四步——dump master 3.8MB、restore 到 slave、`CHANGE REPLICATION SOURCE` 配合 `SOURCE_AUTO_POSITION=1`、`START REPLICA` 后 `SET GLOBAL super_read_only=ON`。日志展示 `Replica_IO_Running: Yes` + `Replica_SQL_Running: Yes` + `Seconds_Behind_Source: 0`，B 决策的第一个证据落地。

G3 在 master 上建 `cooking@%`、给最小权限（库级 SELECT/INSERT/UPDATE/DELETE/CREATE/DROP/REFERENCES/INDEX/ALTER，全局仅 USAGE），然后立刻在 slave 上 `SELECT user FROM mysql.user WHERE user='cooking'`——确认 GTID 把 `CREATE USER` 透传过去了。这是 MySQL 8 一个很优雅的设计：`super_read_only=ON` 拦截所有外部写，但复制 SQL 线程作为"系统线程"被例外放行。然后用一次性 `migrate/migrate:v4.17.1` 容器挂进 prod 网络、以 cooking 用户跑 6 迁移——这里有意用 cooking 用户而非 root，顺带实证最小权限集足以支撑全部 DDL，这是个面试可讲的细节。迁移完 master 和 slave 都是 7 张表 + `schema_migrations version=6 dirty=0`。

### 4. 踩坑记录：揭穿幽灵

G4 是本步真正的高潮，也是上窗口幽灵被揭穿的现场。我设计的探针是一段 `compose run --entrypoint sh ... -c '/app/server > /tmp/probe.log 2>&1 & ...等30秒抓 "http server listening" ... cat /tmp/probe.log'`。第一次跑出来 `EXIT=0` + `===PROBELOG===` 之后空白。我本能地以为是"上窗口的零输出幽灵又出现了"，准备转 strace 级排查。然后用户的一句话点醒了我："命令本身有问题"。

我回头看了 `config.prod.yaml` 的 log 段——`console: false`，`filename: /var/log/cooking/app.log`。zap 在 prod 模式下根本不往 stdout 写，全部写文件。`/app/server > /tmp/probe.log 2>&1` 重定向的是 stdout，但里面本来就没东西——所有日志静静躺在 `/var/log/cooking/app.log` 里。**上窗口的整个"零输出 exit 1"幽灵从来就不是幽灵——是观测路径的盲区**。当时的 app 大概率早就跑到 `log.Fatal` 把错误写进了文件，stdout 当然空白，进程当然 `exit 1`。所有人都在盯错地方。

把探针的 `cat` 改指到 `/var/log/cooking/app.log` 后，真相一秒浮现：`cooking-platform starting mode:release ... mq_provider:rabbitmq sms_provider:mock oss_provider:aliyun` → `mysql connected` → `redis connected` → `event bus initialized provider:rabbitmq` → 4 个 Consumer 启动 → `http server listening :8080` → 三次 `/health/ready 200` → SIGTERM → 三阶段 graceful shutdown → `server exited cleanly`。Risk 2 的全部嵌套 env 覆盖（redis.password / jwt.secret / phone_key / oss.AK） 一并裁决通过。

然后是这一步最让我意外的事——用户在看到日志干净通过后，没有逐条跑我精心拆的 Group 5，而是直接 `docker compose up -d` 全栈起，12 容器全 healthy，localhost curl 200，Mac 公网 curl 也 200。我留下来的 5 组细化 runbook 在那一刻被"懂了原理、跳过仪式"的实战节奏短路了——这本身就是良好工程的表现，仪式只在原理不通时才有价值。

### 5. 收获：可观测性设计要包括"调试时怎么看日志"

这一步最值钱的认知不是部署本身，而是"零输出"幽灵的解构：**prod 安静哲学（不污染 stdout、所有日志归一到文件）和 debug 习惯（盯 docker logs / stdout）天然冲突**。可观测性设计长期被理解为"运行时怎么把日志/指标汇集到哪个系统"，但它同时也必须解决"调试时人怎么找到日志"。当这两者错位时，会出现"app 一切正常但调试者看不见任何信号"的灾难现场——这正是上窗口的真实样貌。这个认知会反向写进 Step 12（日志脱敏 + 错误码收口）的设计输入，也会推动 Step 20 压测前考虑把 `/var/log/cooking/*.log` 挂出宿主机或接 Loki sidecar，让日志不再"沉到容器底"。

另一个伏笔留给 Step 20：本步主从复制建立了，但还没在真实负载下验证 DBResolver 的 Random Policy 是否真把读请求路由到 slave。本步的 `Seconds_Behind_Source=0` 是"无负载下的复制健康"，不是"真实负载下的读延迟"。压测时要造一个读密集场景，配合 Prometheus 看 slave 实际接到的 QPS，那才是 Step 14 主从设计的最终交付。

## Part B · 关联知识点清单

### 基础概念（语言/框架层面）
- Viper 配置库的优先级模型（override > flag > env > config > default）
- Viper `AutomaticEnv` vs `BindEnv` 在嵌套键 + Unmarshal 路径下的差异
- Viper `v.Get` 与 `v.Unmarshal` 的内部机制差异（AllKeys 收集来源）
- zap structured logging 双 sink（console + file）
- zap 在 release 模式下的默认编码器（JSON）与字段约束
- Docker Compose env-file 的 `${}` 变量插值规则与边界
- Docker Compose `${VAR:?error}` 必填守卫语法
- Docker Compose `depends_on: condition: service_healthy` 行为
- Go `database/sql` 驱动的 `multiStatements` 默认行为
- GORM `mysql.Open` 的惰性建连特性
- GORM DBResolver 的 Random Policy 与读写分流

### 设计模式与架构思想
- 三件套抽象（interface + Mock + Real，按 provider 字段切换）
- 配置层的"快速失败" validate gate
- 主从读写分离（CP-leaning 的最终一致性视图）
- 最小权限原则（least privilege）在数据库授权上的具体表达
- 单一职责的 init script 与 idempotent 设计
- 一次性容器模式（jobs / migrations / probes）vs daemon 模式
- 受控观测 vs 暴力日志的调试方法论
- 仪式可跳过：理解原理后绕过流程化 runbook

### Go 语言特性与并发模型
- `log.Fatal` / `os.Exit` 的副作用（不跑 defer）
- 优雅停机：`signal.Notify` + ctx cancel 传播
- 三阶段 shutdown：phase1 http server stop → phase2 event bus close → phase3 资源释放
- errgroup + 父 ctx 取消的 fan-out 协议
- Consumer drained 协议（batch flush + close ack）
- channel 与 ack 模式在 RabbitMQ Consumer 中的协同

### 工程实践（测试/部署/可观测性等）
- MySQL GTID 复制（`mysqldump --set-gtid-purged=ON` + `SOURCE_AUTO_POSITION=1`）
- `super_read_only` 与 replication SQL 线程的例外机制
- Idempotent init script 设计（`SHOW REPLICA STATUS` probe）
- 健康检查链条（compose healthcheck → app `/health/ready` → nginx upstream → 公网 curl）
- 一次性 migrate 容器挂网络模式（避免宿主机装 CLI）
- `compose run --rm --no-deps --entrypoint sh` 跑受控探针
- prod 日志归集策略与 debug 工具链的耦合设计
- Deploy Key 单边只读 + 服务器端独立生成密钥（私钥不跨机器搬运）
- 12-factor app 的 III「Config in the environment」具体落地（含跨 yaml 占位符 + env 注入混合模式）

### 数据安全与合规
- AES-GCM 32-byte key（64 hex 字符）
- JWT secret 强度（≥32 字符）的 cryptographic 论证
- URL-safe 密码（hex / base32）规避 `/@:?#` 在 amqp/mysql DSN 的转义坑
- `.env.prod` gitignore + 600 文件权限
- 库级最小权限 SQL GRANT（无 GRANT/SUPER/FILE）
- `WITH GRANT OPTION` 漏放权限的实战教训
- `init-master.sql` 硬编码密码 vs env 注入的取舍（first-init 权威 vs 灵活性）
- bash_history 敏感命令清理（`history -c && > ~/.bash_history`）
- Repository 私有 + Deploy Key 只读的双重边界

### 可能被延伸追问的关联领域
- Viper 嵌套 env 覆盖问题的多种解决方案（BindEnv 显式绑定 / SetDefault 占位 / 配置中心拉取 / 直接 os.Getenv）
- zap 多 writer 自定义实现（io.MultiWriter / 自定义 zapcore.WriteSyncer）
- prod 日志 ship 到 Loki/ELK 的 sidecar/agent 方案与 stdout 直采差异
- Restart 循环 vs 一次性受控容器在调试中的本质差别（错误现场冻结）
- MySQL 8 `caching_sha2_password` vs `mysql_native_password` 兼容性与性能取舍
- golang-migrate 在 schema 漂移检测中的局限（无双向 diff，只信 `schema_migrations`）
- 配置探针在 Kubernetes 中的演化形态：`initContainer` + readiness probe + startup probe 三段式
- 主从复制延迟监控指标：`Seconds_Behind_Source` 的统计学含义与替代指标（heartbeat 表 + 时间戳 diff）
- DBResolver 在写后立即读场景的一致性问题（write-after-read consistency / sticky session 方案）
- Compose v2 与 v3 schema 在生产部署中的真实差异（虽然官方已停止区分）

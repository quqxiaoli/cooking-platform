# 第 20 步故事线 · 公网验证 + 压测

> Step 20 是 cooking-platform 项目的最后一步，也是整个项目第一次真正"暴露在公网"接受真实流量验证的一步。本文以工程师视角讲述本步从启动到收官的完整决策链路。

---

## Part A · 本步故事线

### 1. 起点：站在公网入口前

Step 19 末，服务器已经能通过 `curl http://47.238.29.251/health/ready` 返回 200——12 个容器全 healthy、主从复制 `Seconds_Behind_Source=0`、OSS 真实接入完成。但还差三件事才能算"真的上线"：HTTPS、端到端验证、压测。这就是 Step 20 要做的全部。

进入 Step 20 之前发生了一件没在 PRD 计划里的变更：域名从 `laidbackck.cn` 改为 `mellowck.com`。本来要等工信部域名审核 + ICP 备案（合计可能 30 天）的卡点，被一个**反直觉**的组合彻底绕过——**香港服务器 + .com 域名**就不需要 ICP 备案，因为阿里云香港 ECS 在中国大陆境外，工信部备案规则不覆盖。这是 Step 18 选择香港服务器埋下的伏笔，到 Step 20 才兑现红利。

启动时还带了一项 Step 19 末发现的债务作为输入：prod 配置 `log.console:false` 让容器 stdout 零输出，调试时找不到日志。这条最后决定挂为遗留债务（接 Loki 是个新子项目，不该塞进上线步骤），但它在本步的 verify 脚本里反咬了一口——这是后话。

### 2. 设计思考：三个关键取舍

**HTTPS 用 certbot webroot 还是 standalone？** standalone 模式让 certbot 自己起 80 端口监听，要求 nginx 暂停；webroot 模式让 nginx 在 80 端口开一个 `/.well-known/acme-challenge/` location，certbot 把挑战文件写到 webroot 目录、nginx 静态提供。选 webroot——续期不需要停 nginx，cron 自动续期更稳定。代价是 nginx 80 端口必须永久保留 ACME 路径（其他全 301 跳 HTTPS），结构稍复杂但运维换来零停机续期。

**SSL 配置选 Modern、Intermediate 还是 Old？** Modern 只支持 TLSv1.3，会拒掉 Android 5.x 以下设备；Old 兼容到 SSL 3.0，对应 IE6 时代。22-28 岁目标用户全部设备都在 TLSv1.2+ 覆盖范围内，选 **Intermediate**：TLSv1.2 + 1.3 + ECDHE-only 密码套件（免 dhparam 文件依赖）。SSL Labs 评级冲 A+，HSTS max-age 设 1 年，留出后续升级到 preload 名单的空间。

**压测怎么测？** 这是本步最大的认知转折。最初的设想是"wrk -t4 -c50 -d30s 测极限 QPS"，但实际跑出来 99% 请求被 nginx 限流挡掉——单 IP 自压测物理上**测不出 backend 极限 QPS**，因为 nginx api_zone 100r/s + burst 20 是设计上的 per-IP 安全网，本身就是要拦截这种突发流量的。改换 wrk2 的 -R 限速模式，把 RPS 限到 80（限流上限 100 留 20% margin），测「合理负载下的延迟分布」。这是个能写进面试的工程取舍——**单机自压测的局限性必须先认知到，否则测出来的是 nginx 503 延迟而不是 backend 真实延迟**。

### 3. 实现节奏：四阶段流水

阶段 1 · HTTPS 落地分两波。第一波改 nginx.conf 加 ACME 挑战路径 + docker-compose 加 `/etc/letsencrypt` 卷挂载，让 nginx 能服务 certbot 的挑战请求；第二波在证书签好后启用 443 server + HTTP→HTTPS 301。中间有个测试用的 `curl /acme-challenge/test` 命令——这地方栽了第一个跟头：`root /var/www/certbot` 的语义是"把整个 URL 路径拼到 root 后面"，所以测试文件必须放在 `.well-known/acme-challenge/` 子目录下，不能扔在 webroot 根。这是 nginx `root` vs `alias` 指令的经典歧义。

阶段 2 + 阶段 3 合并跑：写一个 `verify_step20.sh` 11 个 section 端到端验证。HTTPS 探活、注册、登录、发帖、Feed、详情、点赞、关注、搜索全跑一遍；中间穿插主从 `Com_select` 计数器的基线和增量采样。第一次跑 17/15，看起来惨；定位到根因是脚本去 docker logs 抓 SMS mock 验证码，但 prod stdout 是空的（log.console:false 的债务在这里反咬一口），日志在容器内 `/var/log/cooking/app.log` zap JSON 格式。改用 `docker exec ... cat /var/log/cooking/app.log | jq` 后变 27/4。再修一波 DTO 字段（CreatePostReq 必须 title + scene_tag）+ 一波 412104 合法码识别（pending 帖子返回 412104 是审核状态机的正确行为，不是 bug），最终 31/0/0。**slave 占比 91%，这是项目里最难"事后证明"的设计决策——DBResolver Random Policy 在公网真实流量下完美生效**。

阶段 4 压测迭代了四轮才出真实数据：第一轮 wrk 没加 `--latency` 延迟全空、并发 50 触发限流；第二轮加 `--latency` 但还是被限流；第三轮上 `-R` 限速，发现 apt 装的 wrk 4.1.0 不支持 -R；第四轮装 wg/wrk 4.2.0 还是不支持——`-R` 其实是 wrk2（giltene fork）独有的。从源码装 wrk2 4.0.0 后才跑出真实数据：health P99 12.85ms，业务接口 P99 22-39ms，远低于 PRD SLO 的 100ms。

### 4. 踩坑记录

**bind mount 单文件的 inode 失效**：第二波 HTTPS 改完 nginx.conf 后 `git pull` 到服务器，`nginx -s reload` 但 443 不监听。诊断半天发现宿主机文件和容器内文件 sha 不一致——bind mount 在容器启动时按 inode 绑定，git pull 走的是"写临时文件→atomic rename"，新文件 inode 变了，容器还指向旧 inode。修复方式：`docker compose up -d --force-recreate nginx`。**教训：生产环境避免 bind mount 单文件，改挂目录**（遗留债务 #3）。

**HEAD vs GET 验证方法误判**：连续三轮诊断"www 子域名 HTTPS 404 但主域名 200"，反复怀疑 server_name 配置、Host 校验中间件、HTTP/2 :authority pseudo-header 怪行为，甚至 fetch 了 main.go 看路由——根本不存在 Host 校验。最后定位到根因：**我给的验证命令在主域名用了 `curl -i`（GET）、在 www 用了 `curl -sI`（HEAD），Gin 的 `r.GET("/health", ...)` 不响应 HEAD 方法**，HEAD 永远 404，GET 永远 200，跟域名无关。这件事让我反思"单变量调参"原则——验证命令必须复现生产真实流量的方法，不要同时变两个变量。

**SMS per-IP 限流自压测**：单 IP 反复跑 setup 触发应用层 SMS 限流（`sms:ip:<sha256>` 计数器累积），用户 2 永远抓不到验证码。修复：脚本 setup 用户间加 sleep 8s + 保留手动清 key 应急路径。**真实生产场景每个用户来自不同 IP 不会触发**，是单机自压测的固有限制。

**工具链推荐错误**：我连续推荐了不存在的 wrk flag（apt 4.1.0、wg/wrk 4.2.0 都不支持 -R），让你白装了一次。教训：每次 propose 工具用法时先 `--help` 验证，不要凭印象推荐 flag。这是这个窗口里我做得最差的一段，已经写进偏离点登记备查。

### 5. 收获与伏笔

收获最大的不是 HTTPS 也不是压测数据，而是**主从读写分离 91% 占比的实证**——Step 14 那个抽象的"DBResolver Random Policy"在公网真实流量下被一个 Com_select 计数器对比验证得清清楚楚。这种"事后证明设计正确"的体验，是整个项目里最少见的瞬间，因为分布式系统的很多设计是"按概率发生"的，平时跑测试看不到、只能在统计意义上验证。

伏笔是几条遗留债务：prod 日志归集（Loki/Promtail）让"safe but blind"的生产日志变成"safe and observable"；nginx.conf 单文件 bind mount 改目录挂载；多机客户端压测方案。这些都是 cooking-platform v2 的输入。

Step 20 落地后，cooking-platform 20 步全部完结——一个从 0 到公网上线的商业级 Go 后端项目正式成立。

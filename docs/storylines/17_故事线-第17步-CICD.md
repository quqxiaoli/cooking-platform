# 故事线 · 第 17 步 · GitHub Actions CI/CD + PRD v3.0

## Part A · 本步故事线

### 1. 起点：第三轮生产基础设施的收口

进入 Step 17 时，第三轮（Step 13–16）的核心工作已全部完成：RabbitMQ 生产加固、MySQL 主从、Nginx 双实例、Prometheus+Grafana 监控。项目已经能跑起来一套完整的生产级基础设施。Step 17 是这一轮的收口：补上 CI/CD 流水线，然后写下整个项目的"最终答案"——PRD v3.0。

一个自然的问题是：为什么 CI/CD 要放到最后？原因是 Step 1–16 的代码虽然质量在持续提升，但基础设施的形态一直在变（单实例→双实例，Channel Bus→RabbitMQ，无监控→Prometheus），如果在前期就配了 CI/CD，pipeline 每隔几步就要改动，反而增加噪音。现在架构稳定了，pipeline 才值得投入。

### 2. 设计思考：CI 测试策略的选择

开工时面临一个关键决策：**CI 中的测试怎么跑？**

现有测试文件只有 4 个，全是纯单元测试（logger 脱敏函数、ChannelBus 逻辑、crypto 加解密）。两个方案：

- **方案 A**：`go test -short ./...`，跳过需要外部依赖的测试。简单，但"short"这个 flag 意味着我们主动承认 CI 是残缺的。
- **方案 B**：GitHub Actions `services:` 块起真实 MySQL + Redis，加一对集成测试，CI 环境跑全量。多 1–2 分钟，但拿到了"CI 跑真实数据库"的基础设施。

我本来倾向方案 A，因为当前 4 个测试没有外部依赖，两个方案结果一样。用户反问"方案 B 是不是更规范？"，于是我重新梳理了这个决策的面试价值：GitHub Actions `services:` block 是一个常考的 CI 知识点，能讲清楚"为什么选择在 CI 里起真实数据库而不是 mock"是一个好故事。最终确认方案 B，并同时加了两个集成测试（`TestMySQLPing` / `TestRedisPing`）让 services 块不是摆设。

**ADR 决策关联**：conventions §2 Step17 规范（`go vet + go test + staticcheck`），方案 B 是对该规范的升级，纳入 PRD v3.0 偏离点。

这也说明了一个工程原则：CI 不只是"能跑测试"，而是"测试环境尽量接近生产环境"。方案 B 的代价（多 1 分钟）远低于它带来的信心（数据库连接、schema 查询在 CI 里真实可见）。

### 3. 实现节奏

**第一批：三个 workflow 文件**

先写 `pr.yml`，这是最复杂的：services 块要声明 MySQL 和 Redis，并配置 healthcheck options（`--health-cmd`、`--health-interval`、`--health-retries`）。这里有个细节：GitHub Actions 的 services 容器 healthcheck 通过 Docker healthcheck API 实现，必须在 `options:` 字段里写 `--health-cmd`，而不是 services 下的 `healthcheck:` key（那个是针对 step 的 wait，语法不同）。

`build.yml` 很直接：push main 时 `go build ./...` + `docker build`（smoke test，验证 Dockerfile 没坏）。

`release.yml` 引入了 `docker/metadata-action@v5`，这个 action 的作用是从 git tag 自动提取版本号并生成镜像 tag 列表（`v1.0.0` → `1.0.0` + `latest`）。配合 `docker/build-push-action@v5` 可以一次推送多个 tag。

**第二批：集成测试**

在 `internal/integration/ping_test.go` 里写了两个测试：
- `TestMySQLPing`：读 `TEST_MYSQL_DSN` env var，`sql.Open` + `db.Ping()`，连不上就 Fatal
- `TestRedisPing`：读 `TEST_REDIS_ADDR` env var，`redis.NewClient` + `PING` 命令

关键设计：env var 不存在时 `t.Skip()`，不 Fatal。这样本地 `go test ./...` 能通过（4 个单元测试 + 2 个 SKIP），CI 里设了 env var 才真正跑。

**第三批：verify 脚本**

`verify_step17.sh` 8 段验证：workflow 文件存在→YAML 合法（用 ruby 的 yaml 库，避免 pyyaml 未安装问题）→触发器正确→services 块完整→go build/vet→集成测试文件存在→单元测试通过→集成测试本地 SKIP。

第一次跑时 §2 YAML 验证失败，原因是 `python3 -c "import yaml"` 报 `ModuleNotFoundError`（pyyaml 未安装）。改用 `ruby -e "require 'yaml'"` 即可，macOS 预装 Ruby + stdlib yaml。

**PRD v3.0 生成**

PRD v3.0 整合了 13 条偏离点（Step 9–16）、5 项架构体检修复结论、错误码全表（8 段位）、配置项全表（16 个子结构）、Redis Key 全表（14 条）、Consumer 设计模式、监控 SLO、部署 Runbook、安全清单、已知技术债务（8 条）。

这个过程像是为整个项目做一次"考古"——翻出 Step 9 的 PresignedURL 偏离、Step 13 的 DLX 简化、Step 14 的 mysqldump 全量起步……每一条偏离都有当时的决策背景，现在汇总成一份文档，就是整个项目演进的完整记录。

### 4. 踩坑记录

**坑 1：pyyaml 未安装**  
写 verify 脚本时首先想到用 `python3 -c "import yaml"` 验证 YAML。但 macOS 的 Python3 默认不含 pyyaml，跑出 `ModuleNotFoundError`。换成 `ruby -e "require 'yaml'"` 秒解，ruby stdlib 包含完整的 YAML 解析器。

**坑 2（预防）：GitHub Actions services healthcheck 语法**  
`services:` 块里的 healthcheck 必须写在 `options:` 字段（以 `--health-*` flag 形式），不能写 `healthcheck:` 子 key（那是 docker-compose 语法，Actions 不支持）。PR 合并后在 Actions 里才能看到真实效果，本地只能检查 YAML 结构。

### 5. 收获

Step 17 的主要收获是**"把一个项目打包成一个故事"**。

CI/CD 本身是固定动作，最有意思的是 PRD v3.0 的生成过程：把 17 步里每个不得不偏离原始 PRD 的决策整理出来，发现它们都是有据可查的工程判断，不是随意偏离。PresignedURL vs STS、DLX 简化、mysqldump 全量起步……每一条都是"当时最优解"。

面试中能讲清楚这条"决策演进链"比单独解释某个技术点更有说服力。

**PRD v3.0 §N 需更新的标记**：本文档即 PRD v3.0 的生成步骤，PRD 已生成，无遗留更新需求。

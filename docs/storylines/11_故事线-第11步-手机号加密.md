# 故事线 · 第 11 步 · 手机号加密（AES-GCM + SHA256 pepper 迁移）

---

## Part A · 本步故事线

### 1. 起点：一个预谋已久的技术债

进入第 11 步时，项目已经完成了从搜索到内容审核共 10 个步骤。用户表 `users` 里有一个字段叫 `phone_encrypted`，名字叫"加密"，实际上从 Step 3 开始存的就是明文手机号，比如 `"13800138000"`。这是 ADR-022 的刻意决策：Step 3 优先跑通用户模块，加密这件事留给 Step 11 专门处理。

进入本步时遗留的状态很清晰：`phone_encrypted` 存明文，`phone_hash` 存的是 `SHA256(phone)` 无 pepper，两个字段的 schema 在 Step 3 时就已经按 PRD v2.2 §5 预留好了——`phone_encrypted VARCHAR(200)` 足够容纳 AES-GCM base64 密文，`phone_hash VARCHAR(64)` 是 hex string。本步要做的是把这两个字段的**值**从"暂时凑合"变成"生产合规"。

本步核心目标：
- `phone_encrypted`：明文 → AES-256-GCM 密文
- `phone_hash`：`SHA256(phone)` → `SHA256(phone + pepper)` 防彩虹表
- 向后兼容：dev 环境 key/pepper 为空时，行为与改动前完全一致

### 2. 设计思考：三个关键技术决策

**决策一：dev 空 key 不加密，生产才加密**

最直接的方案是"统一加密，dev 也用"，但这会让 dev 环境必须管理密钥，增加上手成本，CI 运行也需要密钥注入。我选择了"渐进式激活"：当 `phone_key` 为空字符串时，`EncryptPhone` 直接返回明文，`DecryptPhone` 直接返回原值。这样 dev 的行为与 Step 3-10 完全一致，零迁移成本；生产注入真实 key 后自动启用加密。

关联 PRD ADR-022：分阶段实施的初衷就是"避免单步引入密钥管理成为负担"，空 key 降级延续了这个精神。

**决策二：用 GCM auth-tag 失败作为"是否已加密"的探针**

迁移脚本需要区分"还是明文"和"已经是密文"。最笨的方法是加一列 `is_migrated TINYINT`，但这要改 schema、加 migration，破坏了本步"无 schema 变更"的预设。

观察到一个天然的区分：中国手机号是纯数字（11 位），E.164 格式也不超过 15 个字符。这类字符串 base64 解码后最多 11 字节，而 AES-GCM nonce 就要 12 字节——`DecryptPhone` 会在 `len(data) < nonceSize` 处报错。即便凑巧通过了 base64 解码，GCM 的认证标签（auth tag）会因为密文内容不合法而返回 `message authentication failed`。**AES-GCM 的认证属性天然成为了幂等检测机制**，不需要任何额外状态。

这是本步最有面试价值的设计点：把 AEAD 密码学属性（Authenticated Encryption with Associated Data）的认证保证，复用为数据层的幂等判断依据。

**决策三：GetMyProfile 解密失败时降级而非 500**

如果 key 轮换后忘记重跑迁移脚本，数据库里的密文用新 key 解不开，`DecryptPhone` 会返回错误。此时有两个选择：返回 500（强一致），或返回空 `phone_masked`（降级可用）。我选择了后者：记 warn 日志（附 errcode=480102）、`plain=""`、`maskPhone("")` 返回空串，用户个人页照常打开，只是脱敏号码显示为空。

理由是：用户最怕的不是号码显示不出来，而是整个页面报错。降级保留了可用性，同时 errcode 会被 Prometheus 捡到触发告警，运维有机会修复而不是先处理用户投诉。这是"可观测性驱动的降级策略"，面试中可以和强一致性方案做对比。

### 3. 实现节奏

开工后按三层推进：

**第一层——加密工具包 `pkg/crypto/phone.go`**。先写工具函数，不依赖业务代码，容易单独测试。`EncryptPhone` 用标准库 `crypto/aes` + `cipher.NewGCM`，nonce 用 `io.ReadFull(rand.Reader, ...)` 生成，每次加密 nonce 不同（IND-CPA 安全性）。输出格式选 `StdBase64(nonce || ciphertext || tag)`，base64 让密文是可打印字符串，不用改 MySQL 字符集。

**第二层——业务层改造**。`user_service.go` 三处改动联动：① `hashPhone` 包级函数变方法，委托 `crypto.HashPhone`；② `Login` 注册路径调 `EncryptPhone`；③ `GetMyProfile` 调 `DecryptPhone` 再掩码。顺手清理了 `pkg/crypto` 已有的 `doc.go` 旁边的空目录。

**第三层——迁移工具 `cmd/migrate-phone`**。独立 binary，复用 `config.Load()` 读取配置，不引入新依赖。游标分页用 `WHERE id > lastID ORDER BY id LIMIT 200`，主键游标在并发写入下稳定，避免 OFFSET 大表扫描。每行独立 UPDATE，无全局事务，崩溃可重跑。

### 4. 踩坑记录

**坑一：import 块多了一个 `fmt`**。给 `user_service.go` 加 `crypto` import 时，手动在 import 块里追加了 `"fmt"`，但文件本来就有 `"fmt"`。编译报 `duplicate import`，用 Edit 工具定向删除了重复行。

**坑二：`hashPhone` 从包级函数变为方法后，调用点忘了加 `s.` 前缀**。`SendCode` 和 `Login` 里原来都是 `hashPhone(phone)`，改完方法定义后忘了更新调用处，编译报 `undefined: hashPhone`。用 `replace_all` 批量替换了两处。两个"小坑"都在编译阶段发现，没有到运行时。

**坑三：验证脚本 SHA256 计算的跨平台写法**。macOS 没有 `sha256sum`，用的是 `openssl dgst -sha256`，输出格式是 `SHA2-256(stdin)= <hex>` 或 `(stdin)= <hex>`，取最后一列用 `awk '{print $NF}'`。Linux 上 openssl 也有类似输出格式，`$NF` 写法兼容。

### 5. 收获与伏笔

**理解提升**：本步让"密码学属性服务工程需求"变得具体——AES-GCM 的认证不只是安全保证，还是可以被工程代码利用的断言。`DecryptPhone` 失败等价于"数据非法"，这个等价关系在迁移脚本里被直接复用。

**向 Step 12 的伏笔**：`GetMyProfile` 现在调了 `DecryptPhone`，如果解密失败走 warn 日志路径，日志里会出现明文 `phone_hash` 辅助定位。Step 12 日志脱敏要关注这里——phone_hash 本身不是 PII（不可逆），但日志里不应出现任何可能辅助推断手机号的信息，比如用户 ID + phone_hash 的组合。

**向 Step 18 的伏笔**：生产部署时需要在 docker-compose 里注入 `APP_ENCRYPTION_PHONE_KEY` 和 `APP_ENCRYPTION_PHONE_PEPPER`，并在 `make deploy` 前跑 `make migrate-phone`。顺序很重要：先迁移数据，再替换运行中服务的配置，否则新代码用新 key 解不开旧明文，`GetMyProfile` 全量降级。

## Part B · 关联知识点清单

### 基础概念（语言 / 框架层面）
- AES-256-GCM（AEAD 加密方案）
- Nonce（Number Used Once）及其随机生成要求
- Base64 标准编码 vs URL 编码
- SHA-256 哈希与彩虹表攻击
- Pepper vs Salt 的区别（Pepper 全局唯一且保密，Salt 行级存储）
- GORM `Updates(map)` 避免零值覆盖

### 设计模式与架构思想
- 渐进式激活（Progressive Activation）：feature flag 的轻量变种
- 降级策略（Graceful Degradation）vs 强一致性的取舍
- 幂等设计（Idempotency）在数据迁移脚本中的实现
- 游标分页 vs OFFSET 分页的性能差异

### Go 语言特性与并发模型
- `crypto/cipher.AEAD` 接口（`Seal` / `Open`）
- `io.ReadFull` 的使用场景（保证读满 n 字节）
- 包级函数 vs 方法接收者的选择依据
- `replace_all` 精确替换多处调用点

### 工程实践（测试 / 部署 / 可观测性等）
- 单元测试覆盖密码学行为（IND-CPA 测试：两次加密不同）
- `--dry-run` 模式在迁移脚本中的实现
- 迁移脚本的幂等性设计（run-to-completion）
- errcode 驱动的可观测性降级（480102 触发 Prometheus 告警）

### 数据安全与合规
- PII（Personally Identifiable Information）字段保护
- 字段级加密（Field-Level Encryption）vs 全盘加密的取舍
- 密钥管理：环境变量注入 vs KMS（本步用环境变量，KMS 留 Step 18）
- 分阶段迁移（Phased Migration）的安全窗口设计

### 可能被延伸追问的关联领域
- AES-GCM 的认证标签（Authentication Tag）如何防篡改
- 密钥轮换（Key Rotation）如何在不停机的情况下完成
- 为什么不用 bcrypt 存储手机号（bcrypt 不可逆，无法展示脱敏号码）
- `phone_hash` 为什么不用 HMAC-SHA256 替代 SHA256+pepper（等价安全，HMAC 实现更规范但 Go 标准库 SHA256 concat pepper 足够）

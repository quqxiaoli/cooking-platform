## Part A · 本步故事线（约 1200 字）

### 起点：两条悬而未决的线

进入 Step 10 时，项目有两条明显的"欠账"：

其一，`pkg/sms/sender.go` 的 aliyun case 直接 `return error("not implemented yet")`——Step 3 写注册登录时预留的坑，靠 mock 撑过了 7 步。其二，`post_service.go` 里有一行注释：`// MVP: directly visible ... Step 10 flips this`，帖子一创建就 `is_visible=1`，审核状态机形同虚设。

Step 10 的任务就是把这两条线都收紧：接入真实的阿里云 dysmsapi，并让内容审核流水线真正跑起来——发帖变 invisible，AuditConsumer 审完再翻回 visible。

### 第一个拦路虎：架构争论

开工假设清单阶段，我按 PRD §1 的事件流 ASCII 表（`event.post → 提交审核 API`）设计了"两跳"架构：AuditConsumer 先订阅 TopicPost 调 Aliyun API，拿到结果后再发 AuditEvent 到 TopicAudit，第二个 goroutine 订阅 TopicAudit 写 DB。

用户喊停，指出这是读错了文档。PRD v2.2 §9.2 带有 `[v2.2 from Step 4]` 标记，明确写的是："PostService.Create 改为发布 `event.audit`，AuditConsumer 订阅 TopicAudit 内部自己调 API + 更新字段"——**一跳，producer 是 PostService 而不是 Consumer，天然无环路**。

§1 的那张表是 v2.1 旧框架的遗留，被 §9.2 的修订补丁覆盖。CLAUDE.md §4 说"带 v2.2 标记的是权威修订"，是我没认真对照。收敛之后反而更简洁：PostService.Create 额外 publish 一条 AuditEvent(AuditStatus=0)，AuditConsumer 只有一个订阅 goroutine，链路清晰。

### 实现节奏：从数据层到 Consumer

**数据层**先走：audit_logs 表的设计核心是"只追加不更新"。每次状态流转写一行，不覆盖历史——这是内容安全合规的基本要求，事后核查靠的就是完整的变更链。我把 `UpdatedAt` 字段从 AuditLog model 里去掉，用注释显式说明"行不可变"，防止以后有人下意识加 `Save()`。

PostRepository 接口追加 `UpdateAuditStatus`，用 `map[string]interface{}` 而非 struct 传更新字段——原因是 GORM 会跳过零值字段，`is_visible=0`（拒绝时）正好是零值，如果用 struct 会静默失败，帖子永远翻不回 invisible。这个细节踩坑了一下，靠在 interface 注释里写明才没再犯。

**外部依赖层**的小插曲是 Aliyun SDK 方法名。我写 `aliyun.go` 时用了 `CreateImageScanRequest()` 和 `client.ImageScan()`，逻辑上合理但编译直接报 undefined。去 SDK 源码里 grep 才发现图片同步扫描的方法名是 `ImageSyncScan`，对应 `CreateImageSyncScanRequest()`——老 SDK 命名不统一，async 和 sync 各走不同 API，没有在文档显著位置说明。这种错误只有编译才暴露，文档不会告诉你。

**AuditConsumer** 的降级策略是这步最核心的业务决策：Green API 失败时，帖子保持 `is_visible=0 / audit_status=0`（pending），**不**强行翻 visible。fail-closed。日志里打 `errcode=470101` 方便 Grafana 告警。写顺序是：先写 audit_log（合规日志不能丢），再 UpdateAuditStatus（即使前一步失败也继续，帖子状态比日志更重要）。

### AuditStatus=0 的语义重用

AuditEvent struct 里 AuditStatus 的注释写的是 `1=机审通过 2=疑似...`，没提 0。PostService 发出的提交信号用了 `AuditStatus=0`（model.AuditStatusPending），AuditConsumer 靠 `if e.AuditStatus != 0 { return }` 来区分"提交请求"和"结果通知"。

这是一个"复用现有枚举值"的设计，好处是不用新增 event 类型；潜在问题是将来如果有外部系统发来 AuditStatus=0 的噪声事件，Consumer 会误触发 API 调用。在系统边界清晰（TopicAudit 只有 PostService 发布）的 MVP 阶段这个风险可控，写进偏离点留给 Step 12 收口时评估。

### 第二个 Migration 踩坑：TEXT 列不能有 DEFAULT

audit_logs 表建好后，第一次 `make migrate-up` 直接报 `Error 1101: BLOB, TEXT, GEOMETRY or JSON column 'raw_response' can't have a default value`。

原因是 `raw_response TEXT NOT NULL DEFAULT ''`——MySQL 8.0 严格模式（sql_mode 含 `STRICT_TRANS_TABLES`）禁止给 TEXT/BLOB 类列设置 DEFAULT 值。这条规则在 MySQL 5.x 宽松模式下会静默忽略，迁到 8.0 才暴露，文档里不显眼。

解决方式：SQL 里删掉 `DEFAULT ''`，GORM model tag 同步去掉 `default:''`。但此时 migration 已处于 dirty state（`schema_migrations` 表里 version=6, dirty=1），直接 `migrate up` 会报 "Dirty database version 6. Fix and force version."。修复流程：

```
migrate force 5   # 清掉 dirty flag，退回上一个干净版本
migrate up        # 重新应用 000006
```

之后 `make verify-step10` 9 段全绿。

### is_visible=0 的连锁效应

改完 PostService 之后，GetDetail 的可见性检查（`is_visible != PostVisible → return 404`）变成了真实门卫。verify_step10.sh §2 就测这个：发帖后立刻查，应该 412104。

但这里有个时序微妙之处——MockAuditor 是毫秒级的，ChannelBus 是进程内同步投递，AuditConsumer goroutine 几乎立刻就能处理。两次 curl 之间有网络往返 + bash 解析，大概 30-100ms，Consumer 通常已经处理完了。所以 §2 是个略带"时序假设"的断言：它成立是因为 bash 够慢，MockAuditor 够快，但在极端快的机器上可能失效。生产 Aliyun 模式下这个问题不存在（API 需要 200ms+，帖子有几百毫秒的不可见窗口）。这个注意事项写进故事线备忘。

### 收获

- **PRD 多版本文档冲突的处理方式**：带明确版本标记（`[v2.2 from Step X]`）的修订段优先级高于旧框架图，下次看到类似注记立刻对照。
- **GORM zero-value 陷阱**：用 map 传更新字段是防御性编程，凡是可能包含零值的更新都不能用 struct。
- **SDK 要先编译**：Aliyun 老 SDK 的命名不可直接从文档猜测，写完必须编译确认。
- **fail-closed vs fail-open 的选择依据**：内容安全场景默认 fail-closed（保守），电商促销场景可以 fail-open（宽松）。选哪个取决于"误放"和"误拦"哪个代价更高。
- **MySQL 8.0 TEXT 列不能有 DEFAULT**：严格模式下 `TEXT NOT NULL DEFAULT ''` 直接报 Error 1101；宽松模式（5.x）会静默忽略，升到 8.0 才暴露。Migration 变成 dirty state 后需先 `migrate force <prev_version>` 再重跑，不能直接修 SQL 后 `migrate up`。
- **验证脚本的时序假设**：§2「发帖后立刻 404」依赖"bash 两次 curl 间隔 > MockAuditor 处理时间"，在 mock 环境下必然 flaky，直接移除；§1 CREATE 响应已断言 is_visible=0，语义充分覆盖。

## Part B · 关联知识点清单

**基础概念（语言 / 框架层面）**
- GORM `Updates(map)` vs `Save(struct)` 的 zero-value 差异
- alibaba-cloud-sdk-go 的 `SetContent([]byte)` / `GetHttpContentString()` 非标准 RPC 风格
- Gin 中间件链 context 与 goroutine 隔离

**设计模式与架构思想**
- fail-closed vs fail-open 降级策略选择
- Append-only 审计日志（不可变行）
- Event-driven 状态机：PostService 发"请求事件"，Consumer 驱动"状态转移"
- 三件套模式（interface + mock + real）在外部依赖解耦中的应用

**Go 语言特性与并发模型**
- ChannelBus fan-out：多个 goroutine 订阅同一 topic，消息广播而非竞争消费
- goroutine 生命周期：Consumer goroutine 受 ctx 管理，Subscribe 阻塞直到 ctx 取消

**工程实践（测试 / 部署 / 可观测性等）**
- 端到端验证脚本中的时序假设（§2 "立即 404" 的脆弱性）
- errcode 只用于结构化日志 tag 而非 HTTP 响应的分层设计
- 多版本 PRD 文档冲突解决策略

**数据安全与合规**
- 内容审核合规性：不可变审计日志 + 原始响应存档
- fail-closed 在内容安全场景的必要性
- RAM 子账户最小权限原则（Green 专用 AccessKey 独立于 OSS/SMS）

**可能被延伸追问的关联领域**
- Aliyun Green 新版 API（2022，imageaudit + textmoderation）与老 API 的区别
- 内容审核异步回调模式（webhook callback）vs 同步轮询
- 人工审核队列设计（audit_status=2 疑似帖子的 admin 工作流）
- GORM 事务：UpdateAuditStatus 和 audit_log.Create 是否应放一个事务

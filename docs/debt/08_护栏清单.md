# 护栏清单 · 刻意做对，重构时不许当债优化掉

> **优先级**：不低于债务清单。任何后续重构若动到下列设计，必须先在本表对应条目下登记决策理由。
> **状态**：v0（Phase 0 锁定 15 条候选，各 Batch 收口时可追加，Final 定稿）

## 条目固定字段

| 字段 | 说明 |
|------|------|
| **编号** | `GUARD-NN` |

## v0 候选条目（Phase 0 锁定）

### GUARD-01 · EventBus 接口分离（Publisher / Subscriber / EventBus）

- **锚点**：`internal/event/bus.go` / PRD §3.1
- **关联 ADR**：PRD §3.1
- **为什么是 deliberate 决策**：service 只依赖 `EventPublisher`，consumer 只依赖 `EventSubscriber`；接口窄到极致后，dev（Channel）/ prod（RabbitMQ）双栈可零业务代码修改切换。
- **误删后果**：service 若拿到完整 `EventBus`，会出现"业务代码里偷偷 Subscribe"的反模式，dev/prod 切换被锁死。

### GUARD-02 · ChannelBus / RabbitMQBus 双实现共存

- **锚点**：`internal/event/channel.go` + `internal/event/rabbitmq.go` / PRD §3.2
- **关联 ADR**：PRD §3.2
- **为什么是 deliberate 决策**：dev 不依赖外部 broker，单元测试与本地开发零基础设施门槛；prod 切 RabbitMQ 只改 `mq.provider` 配置。
- **误删后果**：本地开发被迫起 RabbitMQ；新人 onboarding 成本飙升；单元测试无法在 CI 里直接跑（必须 services 块兜底）。

### GUARD-03 · Feed 游标分页 + 版本号缓存

- **锚点**：`internal/cache/feed_cache.go`、Redis Key `feed:ver` + `feed:s{scene}:v{ver}:c{cursorMs}` / PRD §6.3
- **关联 ADR**：PRD §6.3
- **为什么是 deliberate 决策**：Offset 分页在动态时间线下会重复/漏数据；版本号 INCR 是 O(1) 全量失效的标准手法。
- **误删后果**：用户翻页时遇重复内容；新帖发布后缓存陈旧无法被定向清理。

### GUARD-04 · 手机号 AES-GCM 密文 + SHA256(phone+pepper) 索引双字段

- **锚点**：`pkg/crypto/phone.go`、`internal/model/user.go`（PhoneEncrypted / PhoneHash 双字段） / PRD §11.1
- **关联 ADR**：PRD §11.1 / progress/11 偏离点
- **为什么是 deliberate 决策**：明文不入库（合规），仍能用 hash 做唯一索引；pepper 让 hash 离线穷举失效。
- **误删后果**：合规事故 + 数据泄露风险；hash 无 pepper 时彩虹表可秒破。

### GUARD-05 · 审核状态机 + `is_visible` 冗余字段（fail-closed）

- **锚点**：`internal/model/post.go`（is_visible 默认 0 + audit_status）/ PRD §1
- **关联 ADR**：PRD §1 / progress/10 偏离点（AuditStatus=0 复用为"提交请求"信号）
- **为什么是 deliberate 决策**：默认不可见，审核通过才可见；任何代码遗漏审核流都不会泄漏内容。
- **误删后果**：未审内容直接进 Feed，合规事故 + 公关事故。

### GUARD-06 · `DrainChan` + `context.Background()` 优雅停机

- **锚点**：`internal/consumer/helper.go`、`internal/consumer/like_consumer.go:240` / PRD §4.1 / conventions §1.2
- **关联 ADR**：PRD §4.3 / 体检 M4
- **为什么是 deliberate 决策**：父 ctx 已取消，仍需把 in-flight 事件刷进 MySQL；用独立 background ctx 是"drain 必须完成"的硬保证。
- **误删后果**：滚动重启时丢事件；like_count、view_count 持续漂移。

### GUARD-07 · Lua 原子（`addLikeScript` / `decrClampScript`）

- **锚点**：`internal/cache/like_cache.go` / PRD §6.2 / conventions §1.3
- **关联 ADR**：体检 M3 修复决策
- **为什么是 deliberate 决策**：SADD + INCR、GET + DECR 两条命令之间的竞态会产生用户可见错误；Pipeline 不保证原子。
- **误删后果**：高峰期 like_count 出现负数 / 重复 INCR；用户看到的"已点赞"状态与计数不符。

### GUARD-08 · Config-First（所有可调参数进 `pkg/config/config.go`）

- **锚点**：`pkg/config/config.go`（含 ConsumerConfig / CacheConfig / MQConfig）/ conventions §1.1
- **关联 ADR**：体检 M1 / M2 修复决策
- **为什么是 deliberate 决策**：调参 = 改 yaml + 重启，而不是改代码 + 重新构建镜像；生产压测后调参成本几乎为 0。
- **误删后果**：硬编码倒退后，每次调批量大小都要走完整 CI 流水线。

### GUARD-09 · 三件套接口 + Mock + Real（sms / oss / audit）

- **锚点**：`pkg/sms/*`、`pkg/oss/*`、`pkg/audit/*`
- **关联 ADR**：CLAUDE.md §七
- **为什么是 deliberate 决策**：dev 零外部依赖；prod 经配置切换；集成测试可在两端切换验证。
- **误删后果**：dev 依赖阿里云账户；测试无法离线运行；新人配齐 AK 后才能跑起来。

### GUARD-10 · Upload nonce GETDEL + server 唯一元信息来源

- **锚点**：`internal/service/upload_service.go`、`internal/cache/upload_cache.go`、`internal/handler/upload.go` 的 CallbackReq（只接 nonce）/ PRD §11.2 / progress/09 偏离点
- **关联 ADR**：progress/09 偏离点
- **为什么是 deliberate 决策**：callback 只接 nonce，server 自行查所有元信息；GETDEL 一次性原子消费杜绝重放。
- **误删后果**：白名单绕过 + 跨账户挂载（攻击者伪造 public_url 让其它人的资源关联到自己的帖子）。

### GUARD-11 · OSS URL 二次白名单（avatar_url 在 UpdateProfile 里再校验）

- **锚点**：`internal/service/user_service.go:UpdateProfile` 调 `oss.IsAllowedURL` / PRD §1
- **关联 ADR**：CLAUDE.md §九（防御深度）
- **为什么是 deliberate 决策**：即使 upload 流被绕过，UpdateProfile 仍能拦下恶意头像。
- **误删后果**：恶意客户端可绕过 upload 流直接 PATCH 带恶意 URL 的头像。

### GUARD-12 · INSERT IGNORE + `uk_user_post` 处理重复点赞

- **锚点**：`internal/consumer/like_consumer.go:flushWithCtx` + `migrations/000003_create_likes_table.up.sql` uk_user_post
- **关联 ADR**：PRD §4 / like_consumer 文件头注释
- **为什么是 deliberate 决策**：依赖 DB 唯一约束 + RowsAffected 兜底 at-least-once 重复，应用层无需复杂去重。
- **误删后果**：切 RabbitMQ at-least-once 后 like_count 虚高；用户看到点赞数翻倍。

### GUARD-13 · MySQL DBResolver 读写分离（写 master / 读 slave）

- **锚点**：PRD §5.2 / `cmd/server/main.go` 中 GORM 装配 / progress/14 偏离点
- **关联 ADR**：PRD §5.2
- **为什么是 deliberate 决策**：所有 SELECT 自动路由到 slave，master 只承接写；上层零感知。
- **误删后果**：master 单点承压；高峰期写延迟拖累整站。

### GUARD-14 · 限流三维（per-phone window / per-phone daily / per-IP daily）

- **锚点**：`internal/service/user_service.go:SendCode` 的三段 IncrementAndCheck / `internal/cache/user_cache.go`
- **关联 ADR**：PRD §11 / CLAUDE.md §七
- **为什么是 deliberate 决策**：三维独立计数，任何一维超限即拒；单维度漏洞被其他维度兜底。
- **误删后果**：短信轰炸通道（攻击者用大量号码 + 单 IP，或单号码 + 大量代理 IP 绕过）。

### GUARD-15 · JWT 黑名单 fail-open

- **锚点**：`internal/service/user_service.go:VerifyAccessToken`（Redis 错时只 Warn 不阻断）
- **关联 ADR**：`internal/cache/user_cache.go:IsJWTBlacklisted` 注释
- **为什么是 deliberate 决策**：Redis 抖动不连带 401；黑名单 TTL 与 access token TTL 对齐，风险窗口有限。
- **误删后果**：fail-closed 改造后 Redis 故障 → 全站登录失败；P0 事故。

## Batch 收口追加（v1.0 定稿）

### GUARD-16 · OSS URLPrefix 必须以 `/` 结尾，`IsAllowedURL` 防御深度依赖此约束

- **锚点**：`pkg/oss/aliyun.go:IsAllowedURL` + `configs/config.{yaml,docker.yaml,prod.yaml}` `oss.url_prefix` 均以 `/` 结尾
- **关联 ADR**：Batch 4 / TD-UPLOAD-01
- **为什么是 deliberate 决策**：`strings.HasPrefix(url, cfg.OSS.URLPrefix)` 不带 trailing slash 时，攻击者可伪造 `https://cooking-prod.oss-cn-beijing.aliyuncs.com.attacker.com/...` 绕过白名单；trailing slash 是"边界字符"兜底。
- **误删后果**：白名单可被同前缀 + 任意后缀绕过，恶意头像 / 帖图 URL 直接通过 `UpdateProfile` / `Callback` 入库。

### GUARD-17 · Shutdown LIFO 顺序固定为 HTTP → ConsumerManager → OSS → EventBus → Redis → MySQL

- **锚点**：`cmd/server/main.go:264-310`（含头注释 line 21 写明 LIFO 顺序）
- **关联 ADR**：Batch 6 + Final
- **为什么是 deliberate 决策**：HTTP 先关，避免新事件 publish；Consumer 接着 drain，把 in-flight 事件刷进 MySQL；OSS 关 MockClient 嵌入式 HTTP listener；EventBus 关让残留 Publish 报错而非阻塞；最后才动 Redis / MySQL（被全员依赖）。任一顺序倒置都会引入丢事件 / 死锁 / panic。
- **误删后果**：滚动重启时事件丢失；或 Consumer 写 MySQL 时 MySQL 已关 → panic stack。**注意 TD-SYS-04 是这套设计的薄弱面（外层无 budget），但顺序本身不许动。**

### GUARD-18 · prod `phone_key` / `phone_pepper` 必须 env 注入，`cfg.Validate()` 在 release 模式 fail-closed

- **锚点**：`pkg/config/config.go` Encryption 段 + `configs/config.prod.yaml:100-102` 用 `<ENCRYPTION_PHONE_KEY>` / `<ENCRYPTION_PHONE_PEPPER>` 占位 + `APP_ENCRYPTION_PHONE_*` 环境变量
- **关联 ADR**：Batch 5 / TD-CRYPTO-01 / PRD §11.1
- **为什么是 deliberate 决策**：明文手机号不入库（合规）；pepper 让 hash 离线穷举失效。dev 允许空 key + 明文 fallback 仅为本地便利，**prod 模式（server.mode=release）下空 key 必须 fatal**，否则 TD-CRYPTO-01 会让明文落库。
- **误删后果**：合规事故 P0；hash 失去 pepper 后彩虹表可秒破。

### GUARD-19 · Nginx `proxy_next_upstream error timeout http_503` 三件套不许动

- **锚点**：`deploy/nginx/nginx.conf:41-43`
- **关联 ADR**：Batch 6 / TD-INFRA-05 / progress/15 偏离点
- **为什么是 deliberate 决策**：开源 Nginx 没有 active health check，**唯一的 failover 兜底**就是这套 passive 策略：连接错误 / 超时 / 503 时自动切到对端 upstream。删掉等于双实例变单实例 + 单点故障。
- **误删后果**：单 app 实例假死时，所有用户请求挂在它身上，无人切流；P0 可用性事故。

### GUARD-20 · `mysql-slave-init` 必须保持 `restart: "no"`

- **锚点**：`docker-compose.yml:80` `restart: "no"`
- **关联 ADR**：Batch 6 / TD-INFRA-04
- **为什么是 deliberate 决策**：slave-init 是一次性容器，跑 `CHANGE REPLICATION SOURCE TO` + mysqldump；改成 `unless-stopped` 后每次容器重启都会重新 dump 已经在 replicating 的 slave，**破坏 slave 数据 / 中断业务**。
- **误删后果**：dev 复位 / restart 时 slave 进入异常状态，IO_Running=No；下次 `make docker-up` 起不来。

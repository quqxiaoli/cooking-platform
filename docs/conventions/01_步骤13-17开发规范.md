## 一、从体检修复沉淀的新规范

### 1.1 Config-First：所有可调参数必须进配置

**规则**：任何在不同环境（dev / staging / prod）或随负载需要调整的数值，  
**必须**定义在 `pkg/config/config.go` 对应结构体中，**不许**写成包级常量或魔法数字。

**判断标准**（满足任意一条 → 必须进 config）：

- 超时 / TTL / 间隔 / 窗口大小
- 批量大小 / 队列深度 / 缓冲倍数
- 速率限制阈值
- 外部服务地址 / 端口 / 路径前缀

**执行流程**：

1. 在 `Config` 根结构体追加字段
2. 在对应子结构体定义具体字段（`mapstructure:"snake_case"`）
3. `registerDefaults()` 中追加默认值（保证零配置可运行）
4. `validate()` 中追加正值 / 范围校验
5. `configs/config.yaml` 追加对应 YAML 段（附注释说明调参意图）

**反例**（禁止）：

```go
const pvBatchSize = 100            // ❌ 硬编码
const likeFlushInterval = 3 * time.Second  // ❌ 硬编码
```

**正例**：

```go
// pkg/config/config.go
type PVConsumerConfig struct {
    BatchSize     int           `mapstructure:"batch_size"`
    FlushInterval time.Duration `mapstructure:"flush_interval"`
}

// registerDefaults
v.SetDefault("consumer.pv.batch_size", 100)
v.SetDefault("consumer.pv.flush_interval", "5s")
```

### 1.2 Consumer 模板更新（含 DrainChan）

**完整模板**（`internal/consumer/like_consumer.go` 为参照）：

```
struct 字段：bus / db(或repo) / batchSize / flushInterval
NewXxxConsumer(bus, dep, cfg config.XxxConsumerConfig) *XxxConsumer
Start(ctx) → 创建 eventCh(batchSize*4) → subWg.Add(N) → goroutines → flushLoop
flushLoop → ticker(flushInterval) → 三路 select: eventCh / ticker / ctx.Done
ctx.Done 路：subWg.Wait() → DrainChan(eventCh, apply) → flush(context.Background())
```

**强制规则**：

- drain 阶段必须用 `consumer.DrainChan[T]`，禁止再写 `drainLoop:` labeled-for 块
- flush 后的 drain 必须用 `context.Background()`，不可用已取消的 ctx
- 关闭日志格式：`zap.L().Info("xxx consumer drained", zap.Int("total_processed", n))`
- 构造函数第三个参数固定为 `cfg config.XxxConsumerConfig`

### 1.3 Redis 原子变更：能 Lua 就 Lua

**规则**：凡是"读后写"或"条件写"（SADD→INCR、SISMEMBER→SET、GET→DECR）  
都必须用 Lua 脚本保证原子性。Pipeline 只用于纯写且幂等的批量操作。

**已有参照脚本**：

| 场景 | 位置 | 脚本说明 |
|------|------|---------|
| SADD + 条件 INCR | `like_cache.go: addLikeScript` | SADD 返回 0 时跳过 INCR |
| DECR + 钳位 0 | `like_cache.go: decrClampScript` | GET 为空或 ≤0 时 SET 0 |

**判断清单（任意一条 → 必须 Lua）**：

- [ ] 操作结果依赖上一条命令的返回值
- [ ] 需要"仅当 key 不存在时才写"（用 SET NX EX，不是 GET+SET）
- [ ] 需要"仅当计数器 > 0 时才递减"
- [ ] 两条命令之间的竞态会产生用户可见的错误状态

### 1.4 RabbitMQ EventBus 使用规范

**实现文件**：`internal/event/rabbitmq.go`（Step 13 前体检期间完成）

**启用方式**（仅改配置，业务代码零修改）：

```yaml
# configs/config.yaml
mq:
  provider: rabbitmq
  url: "amqp://cooking:cooking123@127.0.0.1:5672/"
  timeout: 10s
```

**当前语义（MVP）**：`autoAck=true`，`DeliveryMode=Transient`（at-most-once）

**Step 13 生产加固清单**（在 rabbitmq.go 内完成）：

- [ ] `autoAck=false` + `delivery.Ack(false)` after successful handler
- [ ] `delivery.Nack(false, true)` on handler error（requeue）
- [ ] 死信队列（DLX）声明：`countBatchSize` 次 Nack 后进 DLX
- [ ] `DeliveryMode = amqp.Persistent`（消息持久化）
- [ ] 连接断开自动重连（指数退避，最多 5 次）
- [ ] `QueueDeclare` 改为 durable=true 命名队列（供多实例共享消费）

**Exchange 拓扑约定**（不可更改）：

```
Exchange: cooking.events   type: topic   durable: true
Routing key: == event.Topic 字符串（如 "event.like"）
Queue naming（生产）: cooking.<topic>.<consumer_group>（如 cooking.event.like.like-consumer）
```

## 二、Step 13–17 各步规范

### Step 13：EventBus 切 RabbitMQ

**必须完成**：
- Docker Compose 添加 RabbitMQ 服务（management plugin 启用，方便调试）
- `rabbitmq.go` 生产加固（ACK / DLX / 持久化 / 重连）
- `config.prod.yaml` 中 `mq.provider=rabbitmq`
- `verify_step13.sh` 验证：发布一条事件 → consumer 消费 → MySQL 确认写入

**禁止**：

- 修改任何 service / handler / repository 文件（接口已稳定）
- 改变 Topic 字符串常量（routing key 对齐）
- 直接在生产配置硬写 RabbitMQ 密码（走 `APP_MQ_URL` env var）

**偏离登记触发条件**：若 at-least-once 语义导致 consumer 去重逻辑需要改动，登记偏离。

---

### Step 14：MySQL 主从复制（GTID）

**必须完成**：
- Docker Compose 添加 MySQL slave 节点（GTID 模式）
- GORM DBResolver 配置读写分离（写 → master，读 → slave）
- 验证：SHOW SLAVE STATUS 确认复制正常；读操作命中 slave

**禁止**：

- 在 slave 上执行 DDL（migration 只在 master 跑）
- 把只读服务（搜索、Feed 读）的 SELECT 留在 master

**新增 config 字段**（必须进 config，不许硬编码）：

```go
type DatabaseConfig struct {
    // 现有字段...
    SlavesDSN []string `mapstructure:"slaves_dsn"` // 主从切换
}
```

---

### Step 15：Nginx 反向代理 + 双实例负载均衡

**必须完成**：
- Docker Compose 中两个 Go 实例（`app1`、`app2`）+ Nginx upstream
- Nginx 配置 upstream keepalive，health check 路由 `/health/ready`
- 验证：连续 100 次请求均匀分布到两实例（nginx log 确认）

**此步骤暴露的潜在问题**：

- Upload 模块的 nonce Redis key 必须跨实例共享（已是 Redis 存储，✅ 无问题）
- Session / JWT 无状态（✅ 无问题）
- 本地文件路径依赖（日志文件、临时文件）→ 各实例独立，需确认日志聚合方案

### Step 16：Prometheus + Grafana 监控

**必须接入的指标**（优先级排序）：

| 优先级 | 指标 | 采集位置 |
|--------|------|---------|
| P0 | HTTP 请求数 / 延迟（P50/P95/P99）| Gin middleware |
| P0 | Consumer 处理速率 / 队列积压 | Consumer flushLoop |
| P1 | Redis 命令延迟 | go-redis hook |
| P1 | MySQL 连接池使用率 | GORM callback |
| P2 | Go runtime（GC pause / goroutine 数）| expvar / pprof |

**标签规范**：

```
handler=<route>   method=<GET|POST|...>   status=<2xx|4xx|5xx>
consumer=<name>   topic=<event.topic>
```

**禁止**：把 Grafana dashboard JSON 硬编进代码仓库（用 `docs/grafana/` 存 JSON 导出）

## 四、文档留存规范

Step 13–17 每步结束时，除标准收尾交付物外，还需额外维护：

### 4.1 生产配置文档

新建 `configs/config.prod.yaml`（Step 13 开始维护）：

- 每个生产环境特有的配置段必须有注释说明**为什么和 dev 不同**
- 敏感值用占位符 `<env:APP_XXX>` 标记，**不放真实值**
- 格式：

  ```yaml
  mq:
    provider: rabbitmq           # dev=channel; prod 用 RabbitMQ 实现持久化
    url: "<env:APP_MQ_URL>"      # amqp://user:pass@host:5672/
    timeout: 10s                 # 生产网络延迟高于 dev，拨号超时放宽
  ```

### 4.2 架构决策记录（ADR）

Step 13–17 每个关键决策（选型、取舍、不做什么）必须在对应故事线 Part A 中明确：

- **决策背景**：为什么要做这个选择
- **备选方案**：考虑过什么其他方案
- **选择理由**：为什么选这个而不是别的
- **后续影响**：这个决策对后续步骤有什么约束
